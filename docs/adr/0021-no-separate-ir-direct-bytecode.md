# ADR-0021: No separate IR; the backend emits bytecode directly from the AST

Status: Accepted

## Context

A compiler often carries independent intermediate representations — a domain IR, a decoder IR, an encoder IR — between the front end and code generation. Souther has to decide whether the MVP needs them.

## Decision

Souther keeps no independent Domain IR / Decoder IR / Encoder IR. Type checking runs on the AST, and the ClassFile backend emits `.class` directly from the AST via `java.lang.classfile`, not through javac. Derived decoders/encoders are filled into the AST from the data shape and generated as Raoh combinator calls directly as bytecode.

## Consequences

The front end fills derived decoders/encoders into the AST from each data's shape, and the backend reads the behavior-side tracked information (input/output types, requirement set, construction permission) together with the `let` body to emit bytecode. souther-runtime stays Raoh-independent — the Raoh dependency lives only in generated bytecode and the app.

Room is left to insert an IR at this stage if a future backend needs it (for example, a human-readable Java-source backend). The MVP does not carry one, so there is less machinery to keep in step with the language.
