# ADR-0004: Derive decoders/encoders from data shape; delegate external representation to Raoh

Status: Accepted

## Context

The boundary between the outside world and the domain has to parse external representations into internal `data` and serialize them back. If the mapping to keys, discriminators, and normalization were written into the domain definitions, the domain would carry representation concerns it should not own. But the boundary still needs decoders and encoders for every type.

## Decision

Decoder and Encoder never appear in Souther syntax; they are derived from a data's field names and shape. External representation, parsing, error accumulation, and discriminator routing are owned by the neutral boundary library Raoh. The ClassFile backend emits Raoh's `Decoder`/`Encoder` combinators directly as bytecode, per input source (raw value / Map = raoh-core, `JsonNode` = raoh-json, jOOQ `Record` = raoh-jooq).

## Consequences

A `data` definition carries only `data` / `invariant` / `behavior`; the correspondence to keys, normalization, and discriminators is kept out of it. The derivation covers the common cases (JSON key = field name, discriminator `"type"` with the case name as tag, `T?` → present/absent, list per element). When it is not enough — a different key name, normalization, defaulting, a business-specific discriminator, a persisted-representation version split — the boundary (Java / Raoh) writes a custom codec that assembles the external representation and calls the data's invariant-checking construction. Representation change is confined to the boundary and never enters the domain definition. For a business-specific sum discriminator this is cheap on the decode side: each case decoder is emitted as a public `Case.decoder()` and the sum decoder is only a `discriminate("type", …)` over them, so the boundary reuses the per-case decoders and rewrites just that one layer with its own key and tags. The encoder is less symmetric — the derived sum encoder bundles the tag with each case's encoding — so a custom sum encoder re-tags each case's Map itself.

souther-runtime does not depend on Raoh: only generated bytecode and the apps that use it do. Primitive parsing, applicative error accumulation, and discriminator routing are all Raoh's combinators; Souther only maps a data's shape (keys, types, discriminators, case tags) onto Raoh's decoder construction. The decoder skeleton is common across sources; only the leaf field access differs per source, so each source variant is generated as the shape allows (raw value / Map always; JSON for shapes without temporals; jOOQ `Record` for flat shapes). Souther owns the domain (types, invariants, behaviors); the boundary owns representation and its parsing.
