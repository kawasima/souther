# ADR-0046: `example` — compile-time-checked tests written in the model

Status: Accepted

## Context

Souther's domain logic is pure and total: a `behavior` is an `input data -> output data` function whose
outside-world dependencies are pushed to the boundary (an injected behavior; ADR-0029). Yet the only way
to test that logic was to write JUnit on the Java side against the compiled `.sou`: decode a raw map into
the input type, reflectively `apply` the behavior, encode the result, and assert. The domain is written
in the model's own ubiquitous language, but its tests are Java glue in another language — the part that
should be easiest to test is the most foreign to write.

The specification's own division of labor (§Purpose) already says value constraints are guarded by
"tests and the types of implementations". `smdd-book` chapter 8 ("a testable specification model") makes
the same point at length: structure is guarded by types, value rules by tests over `data`→`behavior`, and
the recommended flow is to keep the model as the single source of truth and *generate* an extensional view
(Gherkin) from it rather than hand-maintain one (§8.6.4). The persistent failure mode of executable-
specification tooling (Cucumber, FitNesse, Concordion) is drift: the executable spec is a separate artifact
wired to the implementation, maintained by hand, and it rots.

Running an example at compile time and failing the build on a mismatch is not new in itself — Lean's
`example`, Agda/Idris, and (in general-purpose languages) Pyret's `where:`, Roc's `expect`, Rust's
doctests, and D's `unittest` all do it. What is unusual is the venue and its precondition: doing it for a
*business domain model that is itself the deliverable*, which is only tractable because Souther's model is
pure and total. Ordinary domain code touches a database, time, and null, so it cannot be evaluated by the
compiler — you need a running system and mocks. Souther's purity is what lets the compiler evaluate the
model directly.

## Decision

Add an `example` construct: compile-time-checked example rows attached to a behavior, evaluated as part of
compilation. A mismatch is a compile error (`E1905`), exactly as a false constant invariant is (ADR-0032).

Surface. An `example` names a target behavior and lists `|`-led rows; each row is an optional business
description, an input argument tuple, `->`, and an expected result:

```
example 提出する
    | "上限内なら提出できる" : (申請準備中 { 予定費用 = 金額(50000) }, "…") -> 提出済み
    | "上限超過は却下"       : (申請準備中 { 予定費用 = 金額(100001) }, "…") -> 却下 { 理由 = "high_cost" }
```

The expected side asserts at two granularities: a bare type name asserts only the result arm (which case
of the output sum), while a construction, newtype application, or literal asserts the whole value by
structural equality. `example`, `examples`, and `for` are contextual soft-keywords (like `intrinsic`), not
reserved words, so they never collide with the `example.*` module namespace.

Placement. The base form is inline, next to the behavior, following Lean/Pyret/Roc. To keep fixtures out of
the model file, an example may instead live in an attached `examples for <module>` file — an example-only
file that names its target module; the compiler merges it into that module. This is not a general
multi-file module: the attached file carries only examples (`E1906` otherwise), needs no `import` (it is
part of the target module and sees its names directly), and does not open module fragmentation or the
question of which file owns `exposing`. General multi-file modules were considered and rejected for this
change; a module's identity is header-derived, so the merge stays local to `compileModules`.

Evaluation. Only a behavior with a `let` body and no `requires` is evaluable — the same runnable set the
CLI `run` uses; an injected, `>->`, or dependency-taking target is refused with a reason (`E1902`). The
evaluator reuses the existing boundary machinery rather than emitting new code: it turns each fixture
expression into its neutral form, decodes it into the parameter type through the derived decoder, applies
the behavior reflectively on the just-generated bytecode, and compares. Every failing row is reported (not
fail-fast), so a build surfaces the whole set. A JUnit-generating alternative was rejected: the compiler
has no test-source sink, and a compile-time check is stronger and immediate.

Lifecycle. Examples are a compile obligation, not a test-runner phase. They run wherever the module
compiles: in the editor (the LSP compiles a self-contained module on save, so a failing example shows as a
diagnostic), on `souther compile` / the annotation-processor build (a mismatch fails the build), and thus
before any `mvn test` phase. This makes example-driven development a compile-driven red/green loop:
totality is enforced by exhaustive `match` in the type system, while an example drives the value rule for a
representative input — the division `smdd-book` §8.7 describes.

## Consequences

- The model file (or an attached file next to it) becomes the single source of truth for both the domain
  and its checked examples. There is no separate, drift-prone executable-spec artifact for the pure core;
  a future renderer can emit the Gherkin tables of `smdd-book` §8.6.4 from these checked examples.
- Testing the pure domain no longer requires Java glue. The boundary (an injected behavior, its failure
  mapping) is still tested with JUnit — examples cover only the compile-time-evaluable core (`smdd-book`
  §8.5.5: keep external dependencies as inputs and the core stays example-able).
- A constraint change fails the right thing: adding a case to a behavior's input sum breaks exhaustive
  `match` at the type level (`E1201`); tightening a value range makes an example whose fixture now violates
  the invariant fail. The `smdd-book` §8.7 change taxonomy maps onto "fails at the type" vs "fails at an
  example".
- Scope not covered: examples on `>->` pipelines and on pure helpers, property-based generation, and the
  Gherkin renderer are left for later. An attached example file gets syntax and parse diagnostics in the
  editor but not on-save evaluation (it needs its target module, absent from a single file).
