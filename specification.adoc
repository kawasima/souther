[#purpose]
== 目的

Souther は、仕様モデル駆動開発（SMDD）の*仕様DSL*を実装モデルへ素直に書き写すためのJVM向け言語である。

仕様DSLは、業務のルールを `+data+`（AND / OR / List / `+?+`）と `+behavior+`（`+->+` / `+>->+`）だけで表す。値そのものにかかる制約（「金額は0以上」「承認者は申請者の上長」など）は、そこには書かない。生成経路の閉じ込めも、外界への依存も書かない。この三つは、仕様DSLでは意図的にコメントへ預けられている。

Souther の役割は、この*コメントに預けられた三つを実行可能にすること*にある。

[cols="1,1",options="header"]
|===
| 仕様DSLがコメントで表すもの | Souther で実行可能にする手段

| 値の制約（`+// 0以上+`, `+// 承認者は上長+`） | `+invariant+`

| 「この値は検証済みである」という不変条件 | 生成経路の閉じ込め（`+decoder+` / `+constructs+`）

| `+// 依存: 製品カタログ+` / `+// 副作用: メール送信+` | 葉なら実装（`+let+`）を書かない `+behavior+`、使う側なら `+requires+`（Java実装を注入）

|===

仕様DSLの構造（区別・必須・複数性・状態遷移・全域性）はそのまま Souther の型に写り、値制約と外界依存が Souther で型と注入に格上げされる。「構造は型で、値制約はテストと実装の型で、外界依存は注入で守る」という SMDD の分業を、実装モデル側で引き受ける言語である。

[#mapping-principle]
=== 対応の原則

仕様DSLの各構成要素は、次のように Souther へ写る。

[cols="1,1",options="header"]
|===
| 仕様DSL | Souther

| `+data 金額 = 整数 // 0以上+` | `+data 金額 = Int  invariant value >= 0+`（newtype。<<newtype>>）

| `+data X = A OR B+`（A・Bは名前付きdata） | `+data X = A \| B+`（直和、ケースは既存の名前付きdata）

| `+data 立替 = 単位型+` | `+data 立替+`（フィールドを持たないdata）

| `+data X = A AND b+`（Aはレコード、bは追加項目） | `+data X = { ...A, b: B }+`（フィールド合成）

| `+List<T>+` / `+T?+` | `+List<T>+` / `+T?+`

| `+behavior f = In -> Out+`（Outが複数の成功状態） | `+behavior f : (...) -> Out1 \| Out2+`（素の直和）

| `+behavior f = In -> Out OR 失敗+` | `+behavior f : (...) -> Out \| 失敗+`（失敗もただの data。本線離脱は合成で決まる）

| `+f >-> g+` | `+f >-> g+`

| `+// 依存+` / `+// 副作用+` | 実装を書かない `+behavior+`

| `+// 事前条件+` | `+require ... else+`

|===

[#boundary]
=== 外界との境界

Souther は外界そのものを扱わない。境界はプリミティブ値・コレクションと Decoder / Encoder で表す。

[,text]
----
外界
  ↓
プリミティブ値・コレクション
  ↓ Decoder
内部data
  ↓ behavior
内部data
  ↓ Encoder
プリミティブ値・コレクション
  ↓
外界
----

JVMは、生成された型と振る舞いをJavaプログラムから利用可能にする配布基盤として使う。

[#design-principles]
== 設計原則

[#closed-construction]
=== dataの生成経路を閉じる

ある `+data T+` の値を生成できるのは、次のいずれかに限る。

* `+T+` の Decoder
* `+constructs T+` を宣言した behavior の実装（Souther の `+let+`、または注入された Java 実装）
* コンパイラ生成コード

戻り値型に `+T+` を書いただけの behavior は、`+T+` を生成できない。生成の権限を持つのは behavior であって、その実装ではない。だから `+constructs+` は behavior に書く（<<constructs>>）。`+let+` と Java 実装は、同じ宣言から権限を受け取る。

どの behavior にも対応しないヘルパー `+let+`（<<fn-declaration>>）も、data を構築できる。そのときは、構築する data が、その `+let+` を呼ぶ behavior の `+constructs+` に挙がっていなければならない。

[#invariant-on-construct]
=== 不変条件は生成時に必ず検査する

`+data+` に宣言した不変条件は、すべての生成経路で検査する。不変条件に違反する値は返さない。

違反したときの行き先は、境界の内と外で違う。Decoder では Raoh の `+Result+` の失敗になり、ドメインの中では中断する（<<violation-destination>>）。

[#separate-representation]
=== 外部表現と内部dataを分離する

Decoder は外部表現から内部dataを生成し、Encoder は内部dataを外部表現へ変換する。外部表現を内部dataへ直接キャストしてはならない。

[#behavior-composition-unit]
=== behaviorを合成の単位とする

内部計算、外界依存、検証、変換はすべて behavior として表現し、`+>->+` で逐次合成する。behavior は所属（クラス）を持たないトップレベルの入出力関係である。

behavior が型として言うのは*仕様だけ*である。単純な behavior は実装を持たない。実装は `+let+` として書く（<<fn>>）。書かなければ、Java から注入される（<<injected-behavior>>）。例外は `+>->+` 合成で、これは書けばそれ自体が実装になる。

仕様DSLに載る振る舞いだけが behavior で、実装のヘルパーは `+let+` に留まる。こうしておくと、behavior の一覧が仕様DSLの一覧と食い違わない。

[#no-impl-for-outside]
=== 外界依存には実装を書かない

DB、HTTP、ファイル、時刻取得、ID生成などの外界依存は言語内で実装しない。仕様DSLで `+// 依存:+` や `+// 副作用:+` として注記したものが、これにあたる。言語内では型だけを宣言する。

[,text]
----
behavior findMember : (id: MemberId) -> Member | FindMemberError    // let が無い
----

実装はJava側から注入する。実装（`+let+`）を書かないことが、外から与えるという意味である（<<injected-behavior>>）。これを使う側は `+requires+` にその名前を書く（<<requires>>）。依存（外部を読むだけ）と副作用（外部を変える）の区別は、用途を示すドキュメントである。値の合成規則には影響しない。

[#unmarked-sum]
=== 業務結果を直和で表現する

業務上の結果はすべてドメインdataの直和として返す。*業務上の*「失敗」は言語の一級概念ではない。behavior の出力は印の無い直和である。`+>->+` で下流が消費しなかったケースが、本線から外れる（<<composition>>）。メモリ不足やネットワーク断のような*予期しない失敗*は、型に並べない。

invariant 違反はここに並ばない。それは業務結果ではなくモデルのバグで、行き先は中断である（<<algebraic-types>>）。

複数の独立した検証エラーは、Decoder 内部で集積する（<<case-propagation>>）。

[#asymmetric-interop]
=== Java相互運用は非対称とする

Javaから生成物を利用できる。Souther から任意のJava APIを直接呼び出すことはできない。

[,text]
----
Java   -> Souther の生成物     許可
Souther -> 任意のJava API       禁止（実装を書かない behavior 経由でのみ外界に触れる）
----

[#out-of-scope]
== 対象外

Souther は次を持たない。設計判断であって、後で足すためのものではない。

* 任意のJava API呼び出し
* 可変状態
* 例外（後述）
* null
* スレッド
* 非同期処理
* 継承
* structural intersection types（`+A & B+`。理由は<<union-intersection>>）
* リフレクション
* マクロ
* 型クラス
* 高階種
* 依存型
* SMTによる証明
* 停止性証明
* 独自VM
* パッケージマネージャ
* REPL

外界依存の実装はJava側へ委譲するため、Souther自身はDB、HTTP、ファイルなどのAPIを持たない。

*例外について。* 禁じるのは、*業務フローの制御に例外を使うこと*である。業務上の結果は直和のケースで返し（<<unmarked-sum>>）、throw も catch も書けない。

一方、モデルのバグの報告先までは塞がない。invariant 違反は*中断*する（<<algebraic-types>>）。中断は Souther の式にならず、捕まえる構文も無いので、業務フローの制御には使えない。

境界の Java Binding（実装を書かない behavior の Java 実装。<<injected-behavior>>）は別である。*プラットフォーム障害*（DB ダウン、ネットワーク断、ホスト応答なしなど、ドメインの帰結でないもの）に対しては例外を投げてよく、Souther はそれをケースに畳まず素通しして境界へ伝播させる。言語（`+.sou+`）に例外は無いが、外界に触れる Java 側は投げうる。区別は「*ドメインの帰結か、プラットフォーム障害か*」── 見つからない・在庫が足りないといった業務として意味のある結末はケース、インフラが使えない・応答しないは例外である（<<java-impl-rules>>・ADR-0029）。

[#modules]
== ソースファイルとモジュール

拡張子は `+.sou+` とする。1ファイルにつき1モジュールを定義する。

[,text]
----
module example.businesstrip
----

モジュール宣言などのキーワードと標準ライブラリは英語だが、ユーザー定義の識別子（型・フィールド・behavior）はUnicodeを許容する。仕様DSLが日本語で書かれているため、業務語彙をそのまま識別子にできる。識別子は束縛・比較の前に *NFC 正規化*する。日本語では合成済み文字と結合文字列が同じ見た目になりうるので、正規化しないと同じに見える名前が束縛で食い違う。ソースは UTF-8 とする。

[,text]
----
data 出張申請 = 申請準備中 | 提出済み | 事前承認待ち | 事前承認済み
----

公開要素は `+exposing+` で列挙する。

[,text]
----
module example.businesstrip exposing (
    出張申請,
    出張申請を提出する,
    事前承認する
)
----

`+exposing+` は型の粒度で公開する。公開した data の `+decoder()+` / `+encoder()+` は常に公開 API なので（<<jvm-codec>>）、`+出張申請.decoder+` のように member を絞る書き方はしない——絞れるものが無いからである。

公開する behavior が `+>->+` 合成のときは、その出力を `+名前 : ケース+` の形で exposing に書く（<<declared-composition-output>>）。合成の出力は段から推論されるが（<<composition-with-requirements>>）、公開境界では推論に任せず署名を要求する。別モジュールや Java から使う契約が上流の変更で黙って変わらないようにするためである。書かなければコンパイルエラー（E1605）とする。

[,text]
----
module example.order exposing (
    注文を確定する : 確定注文 | 在庫不足
)
----

exposing はモジュールのインターフェイス記述にあたる。実装は推論に任せ、境界だけ明示する。OCaml の `+.mli+`、F# の `+.fsi+`、TypeScript の `+.d.ts+` と同じ立場である。

[#imports]
import は明示的に列挙する。ワイルドカードimportは禁止する。循環importはコンパイルエラーとする。

[,text]
----
import example.employee (
    従業員,
    従業員ID
)
----

標準ライブラリも同じ形で取り込む。修飾子 `+List+` / `+String+` / `+Map+` / `+Bool+` / `+Int+` / `+Decimal+` を名前に、使う関数を列挙する。取り込めば、その先は修飾なしで書ける（<<stdlib>>）。

[,text]
----
import List ( map, filter, fold )
import String ( trim, split )
----

これは Elm の `+import List exposing (map)+` にあたる。修飾アクセス（`+List.map+`）は取り込まなくても常に効くので、`+import+` は修飾子を省くための宣言にすぎない。同じ裸の名前を二つのモジュールから取り込むと曖昧になる——`+import String ( length )+` と `+import List ( length )+` を一つのモジュールに両方書くと、`+length+` がどちらか決まらない。そのときは片方を取り込み、もう片方は呼び出しを修飾する（`+String.length(...)+`）。ワイルドカード禁止と同じ理由で、暗黙に降ってくる名前を作らない。

[#reserved-words]
== 予約語

[,text]
----
module   import    exposing
data     invariant
behavior constructs requires
match    with      as
let      if        then     else
require
true     false
----

演算子・記号: `+=+` `+|+` `+>->+` `+|>+` `+?+` `+->+` `+.+` `+...+` `+[+` `+]+`

比較・論理・算術: `+==+` `+/=+` `+<+` `+<=+` `+>+` `+>=+` `+&&+` `+||+` `{plus}` `+-+` `+*+` `+/+`（`+/+` は除算。`+/=+` は不等号。否定は前置演算子でなく `+souther.bool+` の `+not+`。<<stdlib-bool>>）

リスト: `+[+` `+]+`（リテラルとガード内包）、`{plus}{plus}`（連結）。<<stdlib-list>> に規則がある。

`+>->+` は結合的である（<<type-routing>>）。`+|>+` は値パイプで、値を関数へ順に通す——`+e |> f(a)+` は `+f(a, e)+`（<<pipe>>）。

`+->+` は矢印で、型どうしを結ぶところ——behavior の宣言（<<behavior>>）と `+let+` の引数に書く関数型（<<fn-declaration>>）——にも、無名の本体を導くところ——`+match+` のケース（<<match>>）とラムダ（<<blocks>>）——にも、同じ `+->+` を使う。F#・Elm と同じで、矢印は一種類である。`+=+` は名前を右辺に束縛する（右辺が型か項かは <<behavior>> の規則で決まる）。

`+decoder+` / `+encoder+` / `+discriminate+` は予約語ではない。外部表現との対応づけは構文ではなく、data の形状から導出する（<<decoder>>・<<encoder>>）。`+error+` / `+success+` / `+failure+` も予約語ではない。*業務上の*失敗は特別な型でも構文でもなく、ただのドメインdataであり、本線から外れるかは合成で決まる（<<unmarked-sum>>・<<behavior>>・<<composition>>）。

invariant 違反を運ぶ型名も無い。ドメインの中で違反すれば中断するので、運ぶ値が要らない（<<algebraic-types>>・<<violation-destination>>）。

[#delimiters]
=== 区切りと曖昧性解消

完全な形式文法（BNF）は後回しにする（<<future-work>>）。ただし読み書きで迷う区切りと優先順位は先に決めておく。

* `+data+` の宣言フィールドも、レコードリテラル `+型名 { ... }+` のフィールドも `+,+` 区切りである（<<product-data>>・<<record-literal>>）。宣言は型を与えるので `+名前: 型+`、リテラルは値を束縛するので `+名前 = 値+` と書く（`+:+`＝型を持つ、`+=+`＝こう定義される。ADR-0026）。`+...+` スプレッドは型を合成する要素だが、フィールドと同じカンマ列に並ぶ（<<field-spread>>・ADR-0012・ADR-0030）。`+invariant+` は型本体（`+{ ... }+` や `+= Y+`）の後ろに置く節で、フィールド列には加わらない（<<invariant>>・ADR-0031）。
* 引数リストと `+constructs+` / `+requires+` の列挙は `+,+` 区切り（<<behavior-io>>・<<constructs>>・<<requires>>）。
* `+|+` は直和のケース区切りで、`+->+` より緩く結びつく。`+(x: T) -> A | B+` は `+(x: T) -> (A | B)+` と読む（出力全体が直和）。入力に無名の `+|+` は書けない（<<union-intersection>>）。
* `+>->+` はパイプラインの段区切りで、二項演算子ではなく n 項の平らな列である。結合的（<<type-routing>>）。段の位置に書けるのは behavior 名だけで、`+(f >-> g)+` のような括弧は書けない。
* `+|>+` は値パイプで、最も結合が緩く左結合である。`+a |> f |> g+` は `+g(f(a))+` と読む。左辺が右辺呼び出しの最後の引数に入り、パース時に呼び出しへ展開される（<<pipe>>）。behavior を合成する `+>->+` とは別で、こちらは式の値を通す。
* `+?+` は型の後置（`+従業員ID?+`）で、`+Option<従業員ID>+` へ脱糖する（<<optional>>）。
* `+.+` はフィールド参照、`+...+` は spread（<<record-literal>>）。
* 型の位置に書けるのは、名前付き型・stdlib のジェネリック型（`+List<T>+` など）・無名の出力直和（`+A | B+`）・後置 `+?+` である。関数型 `+(A) -> B+` は `+let+` の引数位置だけ（<<fn-declaration>>）。

[#external-representation]
== 外部表現

Decoder と Encoder が扱う外部表現は、Souther 自身の型ではなく、境界ライブラリ Raoh の中立値モデル（scalar / List / Map からなる木）である。Souther はドメインの型・invariant・behavior を所有し、外部表現とその解析・生成は縁のライブラリ Raoh に委ねる（<<separate-representation>>、<<decoder>>）。

Souther は data の形状から、decode は境界で使う入力源ごと（素の値/Map＝raoh-core、JSON の `+JsonNode+`＝raoh-json、jOOQ の `+Record+`＝raoh-jooq）に、encode は中立な Map（raoh-core）へ導出する。decode の骨組み（フィールド・キー・判別子）は源に依らず共通で、末端のフィールド取得だけが源ごとに変わる。JSON 出力は Map を境界でシリアライズする（`+objectMapper.valueToTree+` 等。<<codec-generation>>）。

外部表現には null が現れうるが、内部dataではnullを禁止する。null は境界（Raoh の入力）にのみ存在し、Decoder を通ってドメインに入ることはない。

[#builtin-types]
== 組み込み型

[#primitives]
=== プリミティブ型

[,text]
----
Bool
Int          // 符号付き64ビット整数
Decimal      // java.math.BigDecimal
String
Date         // 年月日。ISO8601。java.time.LocalDate
DateTime     // 年月日時分秒。ISO8601。java.time.LocalDateTime
----

`+Date+` / `+DateTime+` は仕様DSLの `+年月日+` / `+年月日時分秒+` に対応する。値の生成は Decoder（ISO8601文字列から）を経る。現在時刻の取得は言語内では行わず、実装を書かない behavior として宣言する（<<injected-behavior>>）。

==== 数値リテラル

整数リテラルは `+500+` のように書き `+Int+` である。`+Decimal+` のリテラルは `+m+` を付ける——`+500m+`・`+1.5m+`・`+0.5m+`（F# の `+m+` サフィックスに倣う）。`+m+` の無い小数点付きリテラル `+1.5+` はエラーとする。Souther には二進浮動小数点型が無く、小数の既定を `+Decimal+`（正確な十進）にしているので、`+Int+` と `+Decimal+` をリテラルの時点で明示させる（ADR-0033）。

==== 等価

`+==+` と `+/=+` は*値の等価*である。参照の同一性ではない。Souther の data は不変の値なので、同じものかどうかはフィールドが同じかどうかで決まる。

[,text]
----
require 承認者ID == 申請.申請者.上長ID      // 別々に作られた 従業員ID でも、value が同じなら等しい
----

* プリミティブは値どうしを比べる
* data は*フィールドを再帰的に*比べる。`+...+` スプレッドで取り込んだフィールドも含む
* 直和は、同じケースでその中身が等しいときに等しい
* `+List<T>+` は要素を順に、`+Map<String,T>+` はキーと値を比べる
* `+T?+` は、どちらも `+None+` か、どちらも `+Some+` で中身が等しいときに等しい

比べられるのは*同じ型どうし*だけである。違う型の比較はコンパイルエラーとする。

順序（`+<+` `+<=+` `+>+` `+>=+`）は `+Int+`（数値順）・`+String+`（辞書順）・`+Decimal+`（値。スケールは見ない）・`+Date+` / `+DateTime+`（時間順）に対して定める。data や直和には順序を与えない。Elm は順序を `+Int+` / `+Float+` / `+Char+` / `+String+` にしか与えず、日時は不透明型として `+Int+`（ミリ秒）に直してから比べさせる——JavaScript の上に立っているからである。Souther は JVM の上に立ち、`+BigDecimal+` / `+LocalDate+` / `+LocalDateTime+` がいずれも `+Comparable+` なので、これらにも順序を与える。両辺は同じ型でなければならない。

`+Decimal+` はスケールを見ない。`+1.0+` と `+1.00+` は等しい。

[,text]
----
data 金額 = Decimal
// DBの列から読めば 1.00、JSONから読めば 1.0。同じ金額として扱う
----

スケールは値から消えるわけではない。Encoder は読み込んだスケールをそのまま出すので、往復（<<round-trip>>）は壊れない。消えるのは*同一性の判定から*だけである。丸め桁数がドメインの関心事なら、`+Decimal+` のスケールではなく invariant か独立したフィールドとして書く。

`+Decimal+` を含む data の `+hashCode+` は、スケールを落として（`+stripTrailingZeros+`）から計算する。等価がスケールを無視するなら、ハッシュも無視しなければ `+Map+` のキーにしたときに壊れるからである。Java 側から見ても同じ値等価で、生成された data クラスは `+equals+` / `+hashCode+` を全フィールドから実装するので `+Map+` のキーにそのまま使える（<<jvm-product>>）。Java の `+==+`（参照比較）とも `+BigDecimal.equals+`（スケールを見る）とも別物である。

[#collections]
=== コレクション型

[,text]
----
List<T>
Map<String, T>
----

コレクションは不変である。

コレクションは要素について*共変*である。ケース `+A+` は直和 `+S+` として使えるので（<<sum-data>>）、`+List<A>+` は `+List<S>+` が期待される所へ、`+Map<String, A>+` は `+Map<String, S>+` へ、`+A?+` は `+S?+` へ渡せる。これが健全なのは、コレクションが不変で、後から兄弟ケースを書き込む余地が無いからである。可変配列の共変（Java）が不健全なのは書き込みが穴になるからで、そこが違う。Scala の不変 `+List+`・Kotlin の読み取り専用 `+List+` が共変なのと同じ立場である。共変はケース→直和の向きにだけ効き、兄弟ケースどうし（`+A+` と `+B+`）は関係しない。

[#algebraic-types]
=== 組み込み代数型と補助型

[,text]
----
Option<T>    = Some(T) | None
Never                      // 値を持たない型。起こり得ないケース（空の直和）
Unit                       // 値が1つだけの型
----

Souther に `+Result+` / `+Either+` 型は無い。behavior の出力は印の無いドメイン直和で、成功と失敗を分ける2枠の型を持たない（<<behavior>>・<<composition>>）。

`+Never+` と `+Unit+` は、表層で書ける型名ではない。読み方の概念である。`+Never+` は起こり得ないケースを表す。単一ケースの出力（`+-> A+`）は、失敗row が空だと読める。`+Unit+` は仕様DSLの `+単位型+` に対応する。ただしフィールドを書く手段としては提供せず、常に「フィールドを持たない data」（<<sum-data>>）で表す。

多相なのは、ここに挙げた stdlib の型（`+Option+` / `+List+` / `+Map+`）だけである。ユーザー定義のジェネリクスは無い。`+data+` も `+behavior+` も型引数を取らない。

==== invariant 違反は中断する

ドメインの中で invariant を持つ data の構築が違反したとき（<<invariant-mvp>>）、計算は*中断する*。値も返さず、ケースも返さない。

[,text]
----
behavior 値引きする : (d: 見積) -> 値引き済み
    constructs 値引き済み
----

`+値引き済み+` の invariant を破る計算を書いても、出力の型は変わらない。破れば中断する。中断は Souther の式にならず、捕まえる構文も無い。だから業務フローの制御には使えない（<<out-of-scope>>）。境界（<<java-example>>）では Java 側で捕まえて 500 などに写す。業務の失敗（400）とは別の扱いになる。

書き手が違反を返したいなら、それは業務の判断なので `+require ... else+` で明示的に書く（<<require>>）。invariant は「ここを通る値は必ずこうである」という宣言で、破れることを想定した分岐ではない。

decode 側の違反は中断しない。Raoh の `+Result+` の失敗が運ぶ（<<decoder-error>>）。同じ invariant を検査していても、境界の内と外で扱いが違う。

==== Option の現れ方

`+Option<T>+` も*型として書くことはできない*。次の二つの経路でだけ現れる。

* フィールドの `+T?+`（<<optional>>）が脱糖した先
* 標準ライブラリの戻り値（`+List.get+` / `+Map.get+`。<<stdlib-list>>・<<stdlib-map>>）

behavior の出力には書けない。「見つからないかもしれない」を業務の結果として言いたいなら、それはドメインの直和である（<<unmarked-sum>>）。

[,text]
----
behavior findMember : (id: 会員ID) -> 会員 | 会員なし     // こう書く
----

`+Some(T)+` は <<sum-data>> の規則の例外である（ケースは名前付きdataへの参照で、ペイロードを書けない）。`+Option+` は組み込みなので、この形をユーザーが真似ることはできない。中身は `+match+` で取り出す。`+| Some v+` の `+v+` は、`+Some+` ではなく包まれている値に位置束縛される（<<match>>）。

[#optional]
=== 任意型

任意性は `+?+` で表す。

[,text]
----
data 出張完了 = {
    事前承認者: 従業員ID?
}
----

これは次へ脱糖する。

[,text]
----
事前承認者: Option<従業員ID>
----

`+?+` フィールドに値を入れるときは、`+従業員ID+` の値をそのまま書けば `+Some+` に包まれ、`+None+` と書けば空になる。`+Some(...)+` をユーザーが書くことはない（<<algebraic-types>> のとおり、この形は組み込み専用）。

[,text]
----
出張完了 { 事前承認者 = 承認者ID }    // Some(承認者ID) に包まれる
出張完了 { 事前承認者 = None }        // 空
----

`+Some+` への包みも `+None+` も data 生成ではないので、`+constructs+` は要らない（<<constructs>>）。取り出しは `+match+` の `+| Some v+` / `+| None+` で行う（<<match>>）。

`+?+` はフィールドの任意性以外には使用しない。

Decoder の外部型は次である（Raoh の Decoder）。

[,text]
----
Decoder<T> = 外部表現 -> Result<T>          // 失敗は Raoh の Issues を運ぶ
----

[#data]
== data

`+data+` は内部世界の型付きデータを定義する。すべてのフィールドは不変である。

宣言は `+data 名前 = 右辺+` の形をとる。右辺が `+{ ... }+` なら直積、`+A | B+` なら直和、単一の型名 `+Y+` なら `+Y+` を包む newtype（<<newtype>>）、右辺を書かなければ単位である。仕様DSLの `+data 名前 = ...+` と同じ形にそろえてある。

[#product-data]
=== 直積data（AND）

仕様DSLの `+AND+` は直積に写る。フィールド名は業務上の役割で、型は別に指定できる。

[,text]
----
// 仕様DSL: data 従業員 = 従業員ID AND 役職 AND 上長ID
data 従業員 = {
    id:    従業員ID
    , 役職:   役職
    , 上長ID: 従業員ID       // 役割名（上長ID）と型（従業員ID）が異なる
}
----

役割と型は分けて書ける。仕様DSLでは、同じ概念が層ごとに別名を持つ。`+承認者ID+`（操作の入力）が `+事前承認者ID+`（状態に保存）になり、`+pre_approver_id+`（DB列）になる。`+事前承認者: 従業員ID+` のように、フィールド名に役割を、型に概念を書ける。

[#field-spread]
=== フィールド合成（スプレッド）

仕様DSLでは、共通項目を持つ状態が `+data 提出済み = 出張申請共通項目 AND 提出日時+` のように書かれる。`+出張申請共通項目+` の全フィールドを持ち、さらに `+提出日時+` を足す、という意味である。Souther はこれを `+...+` スプレッドで表す。

[,text]
----
data 出張申請共通項目 = {
    申請者:    従業員
    , 予定費用:   金額
    , 出張先:    String
    , 出張目的:   String
    , 出張開始日: Date
    , 出張終了日: Date
}

// 仕様DSL: data 提出済み = 出張申請共通項目 AND 提出日時
data 提出済み = {
    ...出張申請共通項目     // 共通項目のフィールドを平らに取り込む
    , 提出日時: DateTime
}
----

`+...+` スプレッドの意味を次のように定める。

* 取り込んだフィールドは平らに並ぶ。`+提出済み+` の値は `+x.申請者+` のように直接参照できる（`+x.共通.申請者+` のような入れ子にはならない）。
* 取り込んだ data の invariant も引き継ぐ。`+出張申請共通項目+` が持つ不変条件は `+提出済み+` の生成時にも検査される。スプレッドという記法だが、運ぶのはフィールドだけではなく invariant も来る。
* フィールド名が衝突したらコンパイルエラーとする。
* スプレッドは継承ではない。`+提出済み+` は `+出張申請共通項目+` の部分型ではなく、両者に代入互換はない。フィールドを共有するだけである。
* スプレッドはフィールドと同じカンマ列の要素である。`+{+` の中に `+,+` 区切りで並び、複数書ける。慣例として先頭に置く。ReScript のレコード型スプレッド（`+{ ...a, ...b, c: C }+`）に倣った記法で、値レベルのレコードリテラルのスプレッド（<<record-literal>>）と同じ `+...+` である。

共通項目を平らにせず、1つの値として持ち回りたい場合は、通常のフィールドとして入れ子にする（`+共通: 出張申請共通項目+`）。仕様DSLの `+AND+` はスプレッド、明示的な入れ子は通常のフィールド、と使い分ける。

[#sum-data]
=== 直和data（OR）

仕様DSLの `+OR+` は直和に写る。*ケースは既に定義済みの名前付きdataを参照できる*。出張申請モデルのように、状態それぞれが固有のフィールド・invariant・遷移behaviorを持ち、`+OR+` で束ねられる構造をそのまま表す。

[,text]
----
// 仕様DSL: data 出張申請 = 申請準備中 OR 提出済み OR 事前承認待ち OR 事前承認済み
data 出張申請 =
    申請準備中
  | 提出済み
  | 事前承認待ち
  | 事前承認済み
----

入れ子の直和もそのまま書ける。

[,text]
----
data 費用負担区分 = 自社負担 | 先方負担
data 自社負担     = 立替 | 仮払い | 会社カード
----

ケースは常に別途宣言された名前付きdataへの参照である。ケースの位置にインラインのレコードを書くことはできない。仕様DSLの `+OR+` も同じで、ケースは必ず別に宣言される（`+data 役職 = 管理職 OR 一般社員+` に対し `+data 管理職 = レベル+` / `+data 一般社員 = 単位型+`）。

*ケースの値は、その直和の値である。* 関数型言語と同じく、ケース型（例: `+提出済み+`）の値は、その直和型（`+出張申請+`）が期待されるところ——フィールド代入・引数・戻り値のいずれ——でも透過的に使える。上方向（ケース→直和）だけを許し、下方向（直和→特定ケース）は `+match+` を要する。ネストした直和（`+data 自社負担 = 立替 | 仮払い+` が `+費用負担区分+` のケース）も同様に、葉のケースまで畳んで判定する。

[#unit-data]
=== 単位data

フィールドを持たないdataは、本文を書かずに宣言する。仕様DSLの `+= 単位型+` に対応する。

[,text]
----
// 仕様DSL: data 立替 = 単位型
data 立替
data 仮払い
data 会社カード
data 先方負担
data 一般社員
----

単位dataは直和のケースとして頻出する（`+data 役職 = 管理職 | 一般社員+` の `+一般社員+` など）。

単位dataの値は、*名前をそのまま書いて構築する*（関数型言語で0引数コンストラクタが値になるのと同じ）。`+T {}+` のような空レコードは使わない。

[,text]
----
// 実装（let）で単位を構築する。behavior の constructs を要する（2.1・12.3）
[先方費用負担 | 費用負担が先方]     // 先方費用負担 は単位。名前だけで構築
----

これはガード内包で（<<stdlib-list>>）、条件が成り立てば `+先方費用負担+` 1件、成り立たなければ空のリストになる。

式の中の裸の識別子は、束縛済みのローカルがあれば変数、無ければ組み込みの値（`+None+` など。<<algebraic-types>>）、無ければ同名の単位dataの構築、どれでもなければ未定義、と解決する。Souther は識別子の大小で構文的に区別しない（日本語の業務語彙をそのまま識別子にするため）ので、この解決はシンボルテーブルで行う。

ローカルが組み込みの値を隠すのは常に誤りなので、*組み込みの値の名前*（`+None+`・丸めモード `+HALF_UP+` 等。<<stdlib-decimal>>）を束縛の名前——`+let+`・`+match+` の束縛・ラムダの引数——に使うのはコンパイルエラーとする。隠れた組み込みが黙って届かなくなるのを防ぐ。単位dataの名前はドメイン語彙なので束縛に使えるが、その場合ローカルが優先され、その名前の単位data構築は同スコープでは書けない。

[#field-visibility]
=== フィールドの可視性

フィールドの*参照*は、その data の型が見えるところから可能である。同一モジュールなら、そのモジュールのどの data も読める。別モジュールの data は、`+exposing+` で公開されていれば——import できる時点で公開されている——そのフィールドを読める。公開した data は全フィールドが読める（フィールド単位の公開はしない）。behavior は自分の入力 data のフィールドを、それが別モジュール由来でも読める。

制約されるのは*参照ではなく生成*である。フィールドを読めることと、その data の値を新たに構築できることは別の権限で、越境参照を許しても検証を経ていない値が作れないという保証（<<asymmetric-interop>>）は無傷である。値を外へ渡すために getter だけの behavior を定義する必要はない（<<behavior-composition-unit>>）。

JVM では、公開 data のフィールドに公開の読み取りアクセサを生成する（<<jvm-product>>）。モジュール＝パッケージなので、越境で読むにはこれが要る。Java からも読めるが、それは encoder が既に全フィールドを JSON として出しているのと同じ露出で、守っているのは生成である。コンストラクタは非公開のままで、Java が値を作るには invariant を検査する decoder を通すしかない（<<jvm-construction-privacy>>）。

[#union-intersection]
=== union types と intersection types

Souther は union types を持つ。次の三つの形で現れる。

* 名前付き直和 `+data X = A | B+`（sealed。網羅性検査の対象）
* behavior 出力の無名 union `+-> 提出済み | 事前承認待ち+`（<<unmarked-output>>）
* behavior 出力の失敗ケースを含む無名 union `+-> Out | E1 | E2+`（<<type-routing>>）

無名 union は behavior の*出力にだけ*現れる。入力（パラメータ）の型に無名 union は書けない。直和を渡したいときは名前付き data にする（<<unmarked-output>>）。名前付きにすると sealed になり、網羅性検査と JVM 上の型（<<jvm-sum>>）が付く。仕様DSL が入力に `+(A OR B)+` を書く箇所は、その直和を `+data+` として名付けてから参照する。出力の無名 union には JVM 上の型を生成する（<<jvm-anonymous-union>>）。

一方、structural intersection types（`+A & B+`。「A でも B でもある値」を表す型演算子）は*持たない*。仕様DSLの `+AND+` は「A のフィールドをすべて持ち、さらに足す」であって「A かつ B である」ではなく、<<field-spread>> の `+...+` スプレッド（フィールドを平らに取り込み、invariant も引き継ぎ、部分型にはしない）で過不足なく表せる。

[#newtype]
=== newtype（`+data X = Y+`）

右辺が単一の型名 `+Y+` のとき、`+X+` は `+Y+` を包む newtype である。`+Y+` はプリミティブ（`+String+` / `+Int+` / `+Date+` など）でも、名前付き data でも、直和でもよい。`+X+` は `+Y+` とは別の型（nominal に区別する）で、外部表現は `+Y+` と同じ（裸）である。仕様DSLの `+data X = Y+`（右辺が単一の名前）をそのまま書ける。

[,text]
----
// 仕様DSL: data 従業員ID = 文字列 // 空文字列ではない
data 従業員ID = String
    invariant String.length(value) > 0

// 仕様DSL: data 権限不足 = 役職
data 権限不足 = 役職

// 仕様DSL: data 費用明細 = 費目（費目は直和）
data 費用明細 = 費目
----

`+data X = Y+` は、`+value+` という単一フィールド `+value: Y+` を持つのと同じである。構築は型名に値を適用する `+X(v)+`（`+金額(500)+`・`+会員ID("m-01")+`）で、`+X { value = v }+` に等しい。参照は `+x.value+`、invariant は `+value+` で内側を指す。`+data X = { value: Y }+`（波括弧つき）との違いは*外部表現だけ*で、`+data X = Y+` は `+Y+` の表現をそのまま裸で読み書きし（`+従業員ID+` は `+"e-01"+`）、`+data X = { value: Y }+` は Object として `+{"value": "e-01"}+` になる。

構築子 `+X(v)+` は関数適用と同じ括弧付きで書く。juxtaposition の `+X v+` を採らないのは、Souther の適用が `+f(x)+` で揃っていることと、識別子に大文字規則が無いため `+金額 -5+` が構築か減算か決まらないためである（ADR-0032）。invariant 違反は他の data 構築と同じく中断する（<<violation-destination>>）。引数がコンパイル時定数で invariant を満たすと静的に確かめられる構築は中断しえないので、型本体のどこにでも置ける。`+金額(-5)+` のように定数が invariant を破ればコンパイルエラーになる。実行時の値を渡す構築は、他の invariant 付き構築と同じく behavior の結果位置に限られ、そこで違反すれば中断する。

newtype かどうかは*この構文で決まる*。単一フィールドの形からは推論しない。だから `+data 金額 = { value: Int }+` にフィールドを足しても表現が黙って変わることはない（`+{...}+` は単一フィールドでも常に Object）。裸表現がほしければ `+data 金額 = Int+` と書く。

`+|+` が無いので、`+data 権限不足 = 役職+` を1ケースの直和と読む余地は無い。直和は必ず `+|+` を持つ（<<sum-data>>）。`+Y+` が直和のときは、`+X+` の外部表現も `+Y+` の判別子つき表現になり、入れ子の直和のケースになれば葉まで畳んで振り分ける（<<sum-discrimination>>）。

[#invariant]
== invariant

[#invariant-declaration]
=== 宣言

仕様DSLでコメントとして書かれた値制約を、`+invariant+` として実行可能にする。

[,text]
----
// 仕様DSL: data 金額 = 整数 // 0以上、円単位
data 金額 = {
    value: Int
}
    invariant value >= 0

data DateRange = {
    start: Date
    , end:   Date
}
    invariant start <= end
----

[#invariant-mvp]
=== 生成時の検査

不変条件は実行時に検査する。コンパイラは、dataを生成するすべての経路に不変条件検査を挿入する。`+...+` スプレッドで取り込んだ不変条件も同じ経路で検査する。

[#invariant-expressions]
=== 不変条件で許可／禁止する式

許可: 比較、論理演算、算術演算、フィールド参照、組み込み関数、ブロック、そして下の3条件を満たす `+let+`。

[,text]
----
let 正の数 (v: Int) = v >= 0

data 金額 = Int  invariant 正の数(value)
data カゴ = { items: List<明細> } invariant List.all(items, x -> 正の数(x.数量))
----

呼べるのは behavior ではなく `+let+` である（<<fn>>）。invariant は data の生成経路で走るので、そこで外界に触れられては困る。behavior は外界に触れうる——実装を書かなければ Java 実装が注入される——ので、invariant からは呼べない。業務規則に名前を付けたいときは `+let+` にする。

`+let+` が invariant から呼べるのは次のすべてを満たすときで、いずれも静的に判定できる。

* 要求集合が空である（対応する behavior が `+requires+` を書いていない。<<requires>>）
* data を生成しない（呼ぶヘルパー `+let+` の構築集合も空。<<blocks>>）
* 再帰しない

一つ目は宣言を読むだけで済む。`+let 事前承認する+` は外界に直接触れないので <<purity>>の意味では純粋だが、`+behavior 事前承認する+` が `+requires 現在時刻+` と書いているので要求集合は空でなく、invariant からは呼べない。

ヘルパー `+let+` は常にこの条件を満たす。外界に触れたければ関数を引数で受けるしかなく（<<fn-declaration>>）、invariant の側にそれを渡す手段が無いからである。

禁止: Decoder、Encoder、dataの生成、再帰、非決定的処理。behavior の呼び出しを別立てで禁じはしない——invariant が直接呼ぶのは `+let+` であって behavior ではない（上記）し、`+let+` が外界に触れる behavior を経由すれば、その分だけ要求集合が空でなくなり条件1で弾かれる。純粋な（要求集合が空の）計算だけが invariant に入る。

[#violation-destination]
=== 違反の行き先は、境界の内と外で違う

同じ invariant を検査していても、どこで破れたかで扱いが変わる。

* *Decoder（境界）*: 違反は Raoh の `+Result+` の失敗（`+Issue+`）に入る（<<decoder-error>>）。外から来た値が不正だった、という*想定内の結果*である。どのフィールドが破ったかを `+path+` と `+code+` で言える。
* *behavior（ドメインの中）*: *中断する*（<<algebraic-types>>）。検証を通った値どうしの計算が規則を破った、という*モデルのバグ*である。

[,text]
----
data 値引き済み = { 額: Int } invariant 額 >= 0

behavior 値引きする : (d: 見積) -> 値引き済み          // 違反ケースは要らない
    constructs 値引き済み
----

*behavior の出力に違反の置き場所は要らない。* invariant を持つ data を構築するからといって、出力の型は変わらない。

業務として「額が負なら別の結果を返す」と決めているなら、それは invariant 違反ではなく業務の判断なので、`+require ... else+` で明示的に書く（<<require>>）。この二つは違うものである——invariant は「ここを通る値は必ずこうである」という宣言で、`+require+` は「こうでなければこう返す」という業務の分岐である。前者が破れるのはバグで、後者が成り立たないのは想定された結果である。

[#decoder]
== Decoder

[#decoder-role]
=== 役割

Decoder は外部入力から内部dataを構築する境界である。仕様DSLの parse-don't-validate（境界で生データを検証し、成功して初めてドメイン型を構築する）を実装する。Souther は data の形状から Raoh の Decoder を導出する。導出された Decoder は外部表現（<<external-representation>>）を読み、成功すれば T を構築し、失敗は Raoh の `+Result+`（`+Issues+` を運ぶ失敗側）が表す。

[,text]
----
会員ID.decoder() : Decoder<会員ID>     // 外部表現 -> Result<会員ID>（失敗は Raoh の Issues）
----

decode の失敗は Souther のドメイン型ではない。malformed input は業務結果ではなく、境界の関心事である（<<unmarked-sum>>）。だからドメインの直和ケースには並べず、Raoh の `+Result+` の失敗側に置く。decode は behavior ではなく境界の縁であり、`+>->+` の段にはならない（<<sequential-composition>>）。ドメインの世界の合成は、behavior と `+>->+` だけで行う。

Decoder は Souther の構文には現れない。data のフィールド名と型から既定の Decoder を導出する。dataの定義に書くのは `+data+` / `+invariant+` / `+behavior+` だけでよい。外部表現との対応づけ（キー名・正規化・判別子）をドメイン定義に混ぜないための分担である。既定の導出で足りないときは、境界（Java / Raoh）側でカスタム Decoder を書く（<<custom-codec>>）。

[#decoder-derivation]
=== 導出規則

既定の Decoder は次の規約で data の形状から導出する。

* JSONキーはフィールド名に一致する。Object の各フィールドを同名キーから読む。
* newtype（`+data X = Y+`。<<newtype>>）は、Object ではなく `+Y+` の表現から直接読む（`+Y+` がプリミティブなら裸のスカラ、名前付き data や直和なら `+Y+` の導出Decoder）。`+{ ... }+` は単一フィールドでも Object として読む。
* ネストした名前付きdataは、そのdataの導出Decoderで再帰的に読む。
* `+List<T>+` は要素ごとに `+T+` の導出Decoderで読む。
* `+T?+`（Option。<<optional>>）は、キーが無いか値が `+null+` なら `+None+`、値があれば `+T+` を読んで `+Some+` に包む。
* 構築時に invariant を検査し、違反は Raoh の `+Result+` の失敗（`+Issue+`）になる。decode は必ず成功か失敗のどちらかなので、違反の行き先は常にある（<<violation-destination>>）。
* 独立したフィールドのエラーは集積する（Applicative、<<case-propagation>>）。集積は Raoh の combinator が担う。

[,text]
----
// newtype は裸の整数から読む（data X = Y の形。8.7）
data 金額 = Int
    invariant value >= 0
// 導出Decoder: Int(n) -> 金額 { value: n }（invariant 検査つき）

// 複数フィールドの data は Object から、フィールド名＝キーで読む
data 従業員 = {
    id:    従業員ID
    , 役職:   役職
    , 上長ID: 従業員ID
}
// 導出Decoder: { "id": ..., "役職": ..., "上長ID": ... } を各フィールドの導出Decoderで読む
----

構築時の invariant 検査と spread（`+...+`）はレコードリテラルと同じ意味論に従う（<<field-spread>>、<<record-literal>>）。

[#sum-discrimination]
=== 直和dataの判別

直和dataの導出Decoderは、判別子フィールド `+"type"+` を見てケースの導出Decoderへ振り分ける。タグはケース名である。

[,text]
----
data 出張申請 =
    申請準備中 | 提出済み | 事前承認待ち | 事前承認済み
// 導出Decoder: "type" を見て "申請準備中" -> 申請準備中, "提出済み" -> 提出済み, ... へ振り分ける
----

入れ子の直和は、葉のケースまで畳んで振り分ける。

[,text]
----
data 自社負担     = 立替 | 仮払い | 会社カード
data 費用負担区分 = 自社負担 | 先方負担
// 導出Decoder: "type" は "立替" / "仮払い" / "会社カード" / "先方負担" のいずれか。
// "自社負担" というタグは外部表現に現れない。
----

畳むのは、`+"type"+` が一つしか無いからである。直接のケースにタグを付けると、往復（<<round-trip>>）が壊れる。`+費用負担区分+` の encode が `+{ "type": "自社負担" }+` を書いた時点で、どの葉だったかが失われる。続けて `+自社負担+` の decode が同じキーを読もうとして、自分のタグではないと拒む。葉で畳めば、どの階層の decoder に渡しても同じ外部表現が読める。ケース→直和の代入が葉まで畳んで判定する（<<sum-data>>）のと同じである。`+match+` の網羅性は各階層で見る（<<match>>）が、それは外部表現の話ではない。

ケースの中身は、そのケースの形で決まる。直積ケースは `+"type"+` と同じ Object にフィールドを平らに並べる（`+{"type":"提出済み", "申請者":…, …}+`）。単位ケースは `+"type"+` だけを書く（`+{"type":"一般社員"}+`）。newtype ケース（`+data X = Y+`。<<newtype>>）は裸のスカラに `+"type"+` を置けないので、中身を `+"value"+` キーの下に入れる。

[,text]
----
data 役職 = 管理職 | 一般社員
data 管理職 = レベル        // newtype
data 一般社員              // 単位
// 導出: 管理職 -> {"type":"管理職","value":3}, 一般社員 -> {"type":"一般社員"}
----

これは Raoh の `+discriminate("type", …)+` にそのまま乗る。判別子で `+"管理職"+` を選び、同じ Map をケースの decoder に渡すと、その decoder が `+"value"+` を読んで `+管理職+` を組む。直積ケースがフィールドを平らに読むのと同じ枠組みで、読む場所が `+"value"+` になるだけである。単独（直和のケースでない）の newtype は裸のまま（`+3+`）で、`+"value"+` 包みは直和の中でだけ現れる。

葉まで畳むのは入れ子の*直和*の話で、newtype ケースはそこで止まる。newtype は直和ではないので、`+data 権限不足 = 役職+`（役職 は直和）を `+事前承認理由+` のケースにすると、`+{"type":"権限不足","value":{"type":"管理職","value":3}}+` のように内側の 役職 は `+"value"+` の下で独立に判別される。

判別子キーやタグを業務固有のものにしたいとき（`+"status"+` を見て `+"draft"+` / `+"submitted"+` に振り分ける、永続表現の `+representation_version+` でバージョンを読み分けるなど）は、外部表現の都合であって data の形状からは導けない。境界側のカスタム Decoder で行う（<<custom-codec>>）。

[#custom-codec]
=== カスタム境界コーデック

既定の導出で足りないとき（別キー名、正規化、既定値補完、業務固有の判別子、永続表現のバージョン分岐）は、Java 側で外部表現を組み立て、data の（invariantを検査する）構築を呼ぶ Decoder を書く。<<java-example>>の `+JdbcFindMember+` が `+会員.decoder()+` を呼ぶのがこの形である。外部表現の変換は境界に閉じ、ドメイン定義には持ち込まない。

[#decoder-error]
=== Decoderエラー

decode の失敗は Raoh の `+Result+` の失敗側が運ぶ。エラーは Raoh の `+Issue+`（`+path+` / `+code+` / `+message+` を持つ）で表され、独立したフィールドの失敗は集積される（<<case-propagation>>）。Souther は*境界の*失敗語彙を所有しない。`+復号失敗+` や `+DecodeError+` というドメイン型は導入せず、境界での失敗の表現は縁のライブラリ Raoh に委ねる。

ドメインの中の違反（<<algebraic-types>>）とは扱いが違う。境界で破れたのは、外から来た値が不正だったということで、想定内の結果である。どのフィールドが破ったかを言えるべきで（`+path+` と `+code+`）、Raoh がそれを持っている。ドメインの中で破れたのは、検証を通った値どうしの計算が規則を破ったということで、モデルのバグである。だから中断する（<<violation-destination>>）。境界の語彙をドメインへ持ち込まず、ドメインの語彙を境界へ持ち出さない。

境界（HTTP 等）では `+Result+` の失敗を受け取り、適切な応答（例: 400）へ写す。<<java-example>>の `+MemberController+` が decode の `+Result+` を分岐するのがこの形である。

[#codec-generation]
=== 生成方式とRaoh

導出した Decoder は、独立した Decoder IR を経ず、ClassFile backend が Raoh の Decoder（`+net.unit8.raoh.decode.Decoder+` と combinator）を*直接バイトコードとして生成する*。生成された decoder は外部表現の中立木（<<external-representation>>。Raoh の `+Object+` / `+Map+` 入力）から読み、`+Result<T>+` を返す。

[,text]
----
Source -> AST -> (導出で decoder/encoder を AST に充填) -> 型検査 -> ClassFile backend -> .class（Raoh の Decoder を出力）
----

Souther のランタイム（souther-runtime）は Raoh に依存しない。Raoh に依存するのは生成されたバイトコードと、それを使うアプリだけである。プリミティブ解析・エラー集積（Applicative）・判別振り分け（<<sum-discrimination>>）は Raoh の combinator が担い、Souther は data の形状（キー・型・判別子・ケースタグ）を Raoh の decoder 構築へ写すだけである。

decoder は入力源ごとに生成する。素の値 / Map は raoh-core、JSON の `+JsonNode+` は raoh-json、jOOQ の `+Record+` は raoh-jooq の combinator を使う。骨組み（キー・型・判別子・ケースタグ）は源に依らず共通で、末端のフィールド取得だけが源ごとに差し替わる。各公開 data には、その形状が対応する源ぶんを生成する。素の値 / Map は常に生成する。JSON は temporal（`+Date+` / `+DateTime+`）を含まない形状に生成する。jOOQ `+Record+` は入れ子・List・Map を持たない平坦な形状（スカラ列と newtype 列のみ）に生成する。単位dataは入力を無視するので、全源に対応する。

[#encoder]
== Encoder

[#encoder-role]
=== 役割

Encoder は内部dataを外部表現（<<external-representation>>）へ変換する全域関数である。Souther は data の形状から Raoh の Encoder を導出する。

[,text]
----
出張申請.encoder() : Encoder<出張申請>     // 出張申請 -> 外部表現（Raoh の中立木）
----

Decoder と同じく、Encoder も Souther の構文には現れない。data のフィールド名と型から既定の Encoder を導出する（<<decoder-role>>）。encode も境界の縁であり、`+>->+` の段にはならない。

[#encoder-derivation]
=== 導出規則

既定の Encoder は Decoder と対称の規約で導出する。

* newtype（`+data X = Y+`。<<newtype>>）は `+Y+` の表現へ直接書く（`+Y+` がプリミティブなら裸のスカラ）。`+{ ... }+` は単一フィールドでも Object へ書く。
* 複数フィールドの data は Object へ書く。キーはフィールド名、値は各フィールドの導出Encoderで変換する。`+...+` スプレッドで取り込んだフィールドも平らに並ぶ。
* ネストした名前付きdataは、そのdataの導出Encoderで変換する。
* `+List<T>+` は要素ごとに `+T+` の導出Encoderで変換する。
* `+T?+`（Option。<<optional>>）は、`+Some(v)+` ならキーに `+v+` を書き、`+None+` ならそのキーを書かない（省略する）。`+None+` を書いて `+キー欠落 → None+` で読み戻すので、往復則（<<round-trip>>）が保たれる。
* 直和dataは判別子フィールド `+"type"+` にケース名のタグを付け、ケースの導出Encoderで変換する。直積ケースはフィールドを平らに、単位ケースはタグだけ、newtype ケースは中身を `+"value"+` に入れる（<<sum-discrimination>>）。入れ子の直和は葉のケースのタグを書く。

[,text]
----
data 金額 = Int  invariant value >= 0
// 導出Encoder: 金額 -> Int(self.value)（裸の整数）

data 提出済み = {
    ...出張申請共通項目
    , 提出日時: DateTime
}
// 導出Encoder: 提出済み -> { "申請者": ..., "予定費用": ..., ..., "提出日時": ... }
----

キー名や表現を外部仕様に合わせたいときは、境界側でカスタム Encoder を書く（<<custom-codec>>）。

[#round-trip]
=== 往復則

次を推奨する（要求はしない）。

[,text]
----
decode(encode(value)) == value
----

`+encode(decode(raw)) == raw+` は要求しない。Decoder は正規化・既定値補完・余剰項目削除を行いうるためである。

[#behavior]
== behavior

behavior は内部dataに対する振る舞いの*仕様*を定義する。所属を持たないトップレベルの入出力関係である。

宣言は `+behavior 名前 = 右辺+` の形をとる。右辺は二つあり、書いたものと、未定として残るものが逆になっている。

*引数リストと出力*（<<behavior-io>>）を書いたなら、書いたのは型である。実装は未定で、`+let+` を書けばそれが実装になり（<<fn>>）、書かなければ Java から注入される（<<injected-behavior>>）。この形の behavior は本体を持たない。仕様DSLが `+// 依存:+` とコメントで注記したものは、ここに来る。

`+>->+` による合成（<<composition>>）を書いたなら、書いたのは段をこの順につなぐという組み立ての手順で、これがそのまま実装である。だから `+let+` は要らず、注入もされない。型のほうが未定で、<<type-routing>> の規則で計算する。

どちらも仕様DSLの `+behavior+` と同じ形である。DSL も両方を `+=+` で書く。

[,text]
----
// 仕様DSL
behavior 事前承認する  = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認権限なし   // 右辺は型
behavior 注文を確定する = 注文を検証する >-> 価格を記載する >-> 在庫を引き当てる      // 右辺は合成
----

`+=+` の右に型が来ることも項が来ることもあるのは、DSL がそう書くからである。Souther が持ち込んだ区別ではない。同じ2行を Souther へ写すとこうなる。

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    constructs 事前承認済み, 承認権限なし          // 型。実装は let か Java 注入

behavior 注文を確定する = 注文を検証する >-> 価格を記載する >-> 在庫を引き当てる
                                                 // 合成。これ自体が実装

// 仕様DSL: // 依存: 時刻取得
behavior 現在時刻 : () -> DateTime                // 型を書いて let を書かない → Java から注入
----

仕様と実装を分けるのは、`+data+` が既にそうしているからではなく、*仕様DSLの1行がそのまま残るようにするため*である（<<non-functional>>）。`+data+` に `+invariant+` が乗るのと同じく、`+behavior+` には `+constructs+` が乗る。どちらも、仕様DSLがコメントに預けたものを Souther が実行可能にした分である（<<mapping-principle>>）。

[#behavior-io]
=== 入力と出力

仕様DSLの入力 `+AND+` は引数リストに写る。パラメータは常に `+名前: 型+` で書く。

[,text]
----
// 仕様DSL: behavior 事前承認要否を判定する = 予定費用 AND 申請者 AND 費用負担区分 -> 事前承認理由リスト
behavior 事前承認要否を判定する : (
    予定費用:    金額,
    申請者:      従業員,
    費用負担区分: 費用負担区分
) -> List<事前承認理由>
----

パラメータ名は読み手のためのもので、型の同一性には関わらない。実装（`+let+`）は自分で名前を付け、位置で対応させる。名前を省いて型だけ書くことはできない。`+(String)+` と書けたなら、それが「String という型」なのか「String という名前」なのか読めなくなる。

引数ゼロは `+()+` と書く（`+現在時刻+` のような時刻取得）。「引数が無い」ことと「`+Unit+` を一つ取る」ことは別で、`+Unit+` は表層に出さない（<<algebraic-types>>）。

[#unmarked-output]
=== 出力は印の無い直和

仕様DSLの出力 `+OR+` は、そのまま印の無い直和に写す。成功／失敗の区別も、`+error+` のような印も付けない。behavior は「入力から、あり得る業務結果のいずれか」への関数である。

[,text]
----
// 仕様DSL: behavior 出張申請を提出する = 申請準備中 -> 提出済み OR 事前承認待ち
behavior 出張申請を提出する : (申請: 申請準備中) -> 提出済み | 事前承認待ち
    constructs 提出済み, 事前承認待ち

// 仕様DSL: behavior 事前承認する = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認権限なし
// 承認権限なし はただの data。error でも Result でもない。
behavior 事前承認する : (
    申請:    事前承認待ち,
    承認者ID: 従業員ID
) -> 事前承認済み | 承認権限なし
    constructs 事前承認済み, 承認権限なし
----

どのケースが「本線から外れる失敗」かは、この behavior 単体では決まらない。`+>->+` で合成したとき、すぐ次の段が消費しないケースが本線を離れる。それが本線から外れるということである（<<composition>>）。`+承認権限なし+` を次の段が入力に取れば、そのケースは処理される。取らなければ、本線を離れてそのまま出力へ運ばれる。同じ値が、ある合成では外れ、別の合成では本線に乗る。

入力が直和のこともある（仕様DSL の `+(提出済み OR 事前承認済み) AND ...+`）。このとき*パラメータ型に無名の直和を直接書くことはできない*。まず直和を名前付き data として宣言し、パラメータにはその名前を書く。実装では `+match+` でケースを開く。

[,text]
----
// (提出済み OR 事前承認済み) を名前付きにする
data 完了可能申請 = 提出済み | 事前承認済み

// 仕様DSL: behavior 出張を完了する : (提出済み OR 事前承認済み) AND 実費用 AND 出張報告 -> 出張完了 OR 不正な実費用 OR 出張報告なし
behavior 出張を完了する : (
    申請:   完了可能申請,
    実費用:  金額,
    出張報告: String
) -> 出張完了 | 不正な実費用 | 出張報告なし
    constructs 出張完了, 不正な実費用, 出張報告なし
----

無名 union が現れるのは behavior の*出力だけ*である（<<union-intersection>>）。入力の型は常に単一の名前付き型（名前付き直和を含む）にする。名前付きにすると読む側は `+data 完了可能申請 = 提出済み | 事前承認済み+` を一度見れば入力の候補が確定し、JVM 側でも入力に名前が付いて Java 実装が単一の型として受け取れる（無名 union は共通スーパータイプを持たない。<<asymmetric-interop>>・<<jvm-anonymous-union>>）。

[#constructs]
=== data生成権限（constructs）

data を生成する behavior は、生成する型を `+constructs+` に書ける。これは内部の生成権限であって、依存（`+requires+`）と違い呼ぶ側からは見えない。だから let 実装のある behavior では*任意*で、省略すれば本体から推論する。書けば本体と*厳密一致*でなければならない（過小は E1002・過大は E1006）。注入 behavior は本体を持たず、宣言がファクトリ生成の入力になるので必ず書く（<<java-base-class>>）。

[,text]
----
behavior applyChange : (input: ChangeEmailRequest) -> Member
    constructs Member
----

複数dataを生成する場合は列挙する: `+constructs Member, AuditRecord+`。

次はdata生成とみなす: レコードリテラル（`+型名 { ... }+`）、直和dataのコンストラクタ、単位dataの構築、新しいdataを要素に含むコレクション生成。既存値をそのまま返す場合は生成とみなさない。

組み込みの `+Some+` / `+None+`（<<algebraic-types>>）は data 生成とみなさない。Option はドメインの data ではなく補助型で、invariant を持たないので <<invariant-on-construct>> に守るものが無い。だから `+?+` フィールドに値を入れる（<<optional>>）のに `+constructs+` は要らない——そもそも Option は型として書けないので `+constructs Some+` と宣言するすべも無い（<<algebraic-types>>）。

*`+constructs+` を書くなら、実装のどこで生成するものも漏らせない。* `+require ... else+` の離脱値（<<require>>）も同じで、`+else 承認権限なし+` は単位dataを一つ作るので、宣言するなら `+constructs 承認権限なし+` に含める（漏らせば E1002）。失敗を表すケースだからといって別扱いにはならない——<<closed-construction>> が閉じているのは生成であって、成功か失敗かは関係しない。

==== constructs は何のためにあるか

`+constructs+` は不変条件を守るためのものではない——それは `+__construct+` と Decoder が担い、どちらも生成時に必ず検査する（<<invariant-mvp>>）。`+constructs+` が担うのは*モデルの読み取り*で、宣言を見れば、その behavior が新しい値を*作る*のか、既にある値を*通すだけ*なのかが分かる。これは出力型からは分からない。

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    constructs 事前承認済み, 承認権限なし        // 新しい状態を作る

behavior 上限を検査する : (a: 金額) -> 金額 | 上限超過
    constructs 上限超過                        // 金額 は入力をそのまま返す。作らない
----

どちらも出力に `+金額+` / `+事前承認済み+` を持つが、片方は新しく作り、片方は素通しする。この区別は出力型に出ない。書けば宣言に残り、読みと取り違えの検査になる。省略すれば推論する——読みと検査が欲しければ書く（opt-in）。

出力に現れない型を新しく作ることも、ここに出る。`+事前承認要否を判定する = (...) -> List<事前承認理由>+` の `+constructs 高額出張, 権限不足, 先方費用負担+` は、出力型のどこにも無い情報である。

ヘルパー `+let+`（対応する behavior を持たないもの）も data を構築できる。ただしヘルパー `+let+` は仕様の言明ではないので、自分では `+constructs+` を宣言しない。その*構築集合は推移的に推論する*（呼ぶヘルパー `+let+` のぶんも畳む）。呼ぶ behavior が `+constructs+` を書くなら、その集合を含まねばならない（含まなければ E1002）。省略すれば、その集合も推論に畳み込まれる。ブロックも同じで、囲む behavior の権限の下で構築する（<<blocks>>）。

依存（`+requires+`）は違い、*必ず書く*（<<requires>>）。`+requires+` は呼ぶ側・注入する側への要求＝外向きの契約で、推論に任せると契約が黙って変わる。`+constructs+` は呼ぶ側から見えない内部権限なので、そこが分かれる。だから `+requires+` は宣言、`+constructs+`（let 実装のある behavior）は任意、と扱いを変える。`+>->+` 合成は出力も要求も推論する（<<composition-with-requirements>>）。

[#record-literal]
=== レコードリテラル・spread

構築は、構築する型名を書いたレコードリテラルで行う。

[,text]
----
事前承認済み {
    申請者 =      申請.申請者,
    予定費用 =     申請.予定費用,
    提出日時 =     申請.提出日時,
    事前承認日時 =  現在時刻(),
    事前承認者 =   承認者ID
    // ... 共通項目の残りも列挙
}
----

同名フィールドをまとめて埋めたいときは spread `+...src+` を使う。`+src+` の同名フィールドをコピーし、残りを明示する。`+src+` の余分なフィールドは無視し、`+型名+` に足りないフィールドがあればコンパイルエラーとする。

[,text]
----
// 申請（事前承認待ち）の共通項目・提出日時をコピーし、2つを足す
事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
----

同じ型のまま一部フィールドだけ差し替えるときも、型名を書いて spread を使う。`+x+` の型が `+Member+` なら `+Member { ...x, 住所 = 新住所 }+` と書く。専用の更新構文は設けない。

フィールド値がフィールドと同名の束縛のときは値を省略できる。`+Member { 住所 }+` は `+Member { 住所 = 住所 }+` の略記で、スコープにある同名の束縛を埋める。

レコードリテラルも spread も生成権限（`+constructs+`）を要する。構築点に型名が出るので、権限検査もフィールド検査もその場で済む。

[#blocks]
=== ブロック

リストの各要素に処理を施すには、引数としてブロックを渡す。ブロックは `+引数 -> 式+` と書く。

[,text]
----
// 仕様DSL: data 未検証注文  = ... AND List<未検証注文明細>
//          data 検証済み注文 = ... AND List<検証済み注文明細>
// 明細を1件ずつ検証して詰め替える。ここでブロックが要る。
behavior 明細を検証する : (xs: List<未検証注文明細>) -> List<検証済み注文明細>

let 明細を検証する (xs) = List.map(x -> 1件を検証する(x), xs)
----

ブロックは無名の `+let+` である。名前を付けないだけで、`+let+`（<<fn>>）と同じものと考えてよい。名前を付けたものは名前で渡せる。

[,text]
----
let 妥当か (r: 事前承認理由) = ...

List.all(理由リスト, r -> 妥当か(r))   // 無名のブロックで包む
List.all(理由リスト, 妥当か)           // 名前で直接渡す。同じこと
----

ブロックと `+let+` は引数として渡せ、`+let+` は引数に取ることもできる（<<requirement-propagation>>）。*data のフィールドに置くことだけができない。* 関数は外部表現に写せないので、置けたとしても decoder / encoder の導出（<<decoder>>・<<encoder>>）の対象外という例外を作ることになる。これは副作用とは無関係の、表現の制約である。仕様DSL自身も、差し替えたい規則を関数ではなく data として持っている（`+data フィールド定義 = ... AND バリデーションルール?+`）。

ブロックは `+let+` で名前に束縛して適用してもよい（`+let f = (x) -> ...+` のあと `+f(a)+`）。名前を付けた `+let+`（<<fn-declaration>>）と同じく、逃げ出さない——適用されるだけで値のまま外へ出ない——かぎり、ブロックは渡し先・適用先へ展開され、閉包を作らない。逃げ出すとき——関数を*返す*とき、実行時に選ぶとき（`+let f = if ... then (x) -> ... else ...+`）——は、自由変数を捕捉した閉包を生成する。ランタイムの `+Fn+` を実装した合成クラスで、捕捉した変数をフィールドに持つ。展開できるところは展開し、必要なところだけ閉包にする。上の *data のフィールドに置けない* 制約は、閉包を作れるようになっても変わらない。表現の制約であって、閉包の有無とは別だからである。

data を構築する点でもブロックとヘルパー `+let+` は同じである。ブロックは囲む behavior の `+constructs+` の下で構築し、括り出したヘルパー `+let+` はそれを呼ぶ behavior の `+constructs+` の下で構築する（構築集合を推移的に推論して突き合わせる。<<constructs>>）。どちらも構築の権限は behavior にあり、名前を付けても消えない。だから構築する `+[高額出張 { 基準金額 = 基準 } | …]+`（<<stdlib-list>>）を `+let 高額出張を作る (基準) = 高額出張 { 基準金額 = 基準 }+` へ括り出しても、呼ぶ behavior が `+constructs 高額出張+` を宣言していれば通る。

[#requires]
=== 依存（requires）

外界に触れる behavior は、何を要するかを `+requires+` に書く。

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    requires 現在時刻
    constructs 事前承認済み, 承認権限なし
----

書くのは、実装を持たない behavior の名前である（<<injected-behavior>>）。`+let+` の中でその名前を使うなら書く。ヘルパー `+let+` に渡して間接的に使う場合も、渡しているのは自分なので自分が書く。

`+requires+` を持てるのは、単純な behavior で `+let+` を持つものだけである。実装を持たない behavior 自身は書かない。それが依存の葉であり、名前が挙がること自体が依存の表明だからである。`+>->+` 合成も書かない。段の和で決まるので、段を見れば読める（<<composition-with-requirements>>）。

==== 仕様DSLのどこに対応するか

DSL は `+// 依存:+` / `+// 副作用:+` を*葉*に付ける。

[,text]
----
// 責務2: 出張申請をデータベースに保存する
// 依存: 出張申請データベース
// 副作用: データベースへの保存
behavior 出張申請を保存する = 未検証出張申請 AND 事前承認要否 -> 保存済み出張申請
----

これは Souther では実装を持たない behavior になるので、`+requires+` は要らない。DSL が合成に注記を付けないのも同じで、段を見れば読める。

DSL がこの二つしか持たないのは、副作用をワークフローの端に置くからである。「純粋な計算の途中で外界を一度だけ使う」という形は、仕様モデルには現れない。Souther はそれを許すので、DSL に写し元の無い位置——`+let+` を持つ behavior——が生まれる。`+requires+` は、DSL の注記を端でない behavior にも書けるようにしたものである。位置は増えるが、書く内容は DSL と同じ「何に依存するか」で、行が増えるわけではない。

==== requires は let の引数になる

`+requires+` に書いたものは、`+let+` の引数として現れる。業務上の入力の後ろに、宣言した順で並ぶ。

[,text]
----
let 事前承認する (申請, 承認者ID, 現在時刻) = {
    require 承認者ID == 申請.申請者.上長ID
        else 承認権限なし

    事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
}
----

behavior が2引数で `+let+` が3引数なのは、*behavior の型が部分適用後の型だから*である。

[,text]
----
let:      (申請, 承認者ID, 現在時刻) -> 事前承認済み | 承認権限なし
bind:     現在時刻 を固定する                          f_a(x) = f(x, a)
behavior: (申請, 承認者ID)          -> 事前承認済み | 承認権限なし
----

`+Handle.bind(new JdbcClock())+`（<<jvm-behavior>>）がこの部分適用である。束ねた後に残るのが behavior の宣言した型で、Java 側は `+apply(申請, 承認者ID)+` と2引数で呼ぶ。`+requires+` は呼び出し側の引数には現れない。

これは <<blocks>> 末尾の「要求はコンストラクタ引数である」を表層に出しただけである。`+let+` の中では `+現在時刻+` はただの引数で、効果でも特別な型でもない。だから `+let+` 側に `+requires+` を書く場所は要らない。

ヘルパー `+let+`（対応する behavior が無いもの）も同じ形に収まる。外界に触れたければ、関数を引数で受ける——ヘルパーは型を自分で書くので（<<fn-declaration>>）、そこに書けばよい。

[,text]
----
let 記録する (x: Int, 現在時刻: () -> DateTime) = ...       // ヘルパー。引数で受ける
----

`+let+` の中に現れる `+現在時刻+` は、どちらの場合も引数である。実装を持たない behavior をトップレベルの名前として直接呼ぶ経路は無い。だからヘルパー `+let+` に `+requires+` を書く場所が無いことは、制限にならない。渡す側が自分の `+requires+` に書き、引数として渡す。

==== 検査

`+let+` が使う実装なし behavior の集合と、`+requires+` に書いた集合は一致しなければならない。足りなければ E1602、余れば E1603 とする（<<e1602>>・<<e1603>>）。安全側に倒した過大な宣言も通さない。

[#tuple]
== タプル

`+(a, b, ...)+` は2つ以上の値を1つにまとめたタプルである（ADR-0036）。式レベルの値で——ヘルパー `+let+` が多値を返し、それを分解して受ける、といった計算の途中でだけ使う——data フィールドや behavior の入出力には書けない。関数（<<blocks>>）と同じく境界を跨がないので、コーデックは導出されない。ドメインの語彙は名前付きの data のまま保つ。

[,text]
----
let 統計 (xs: List<Int>) = (List.sum(xs), List.length(xs))   // 多値を返す

let 平均 (xs) = {
    let (合計, 個数) = 統計(xs)      // 位置で分解束縛
    ...
}
----

`+let (x, y) = t+` はタプルを位置で分解して束縛する。名前は左から順に要素へ対応する。1要素の `+(e)+` はただの括弧付き式で、タプルにはならない。タプルはそのまま `+let+` で保持したり、ヘルパーに渡したり、`+List<(a, b)>+` の要素にもなれる（一級の値）。要素の取り出しは分解だけで、`+fst+` / `+snd+` のような添字関数は持たない。

ラムダの引数 `+(a, b) -> ...+` は多引数なので（<<blocks>>）、タプルとは別である。ラムダの中でタプルを分解したいときは、1引数で受けて本体で `+let (a, b) = p+` と開く。

[#fn]
== let — 実装

`+let+` は behavior の実装である。behavior が仕様（<<behavior>>）で、`+let+` がそれを満たすもの、という分業になる。

[#fn-declaration]
=== 宣言と束縛

`+let+` は `+let 名前 (引数) = 本体+` と書く。同名の behavior があれば、それを実装する。

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    requires 現在時刻
    constructs 事前承認済み, 承認権限なし

let 事前承認する (申請, 承認者ID, 現在時刻) = {
    require 承認者ID == 申請.申請者.上長ID
        else 承認権限なし

    事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
}
----

型は behavior から取る。`+let+` 側は引数に名前を付けるだけで、位置で対応する。behavior 側の名前は読み手のためのもので、束縛するのは `+let+` 側の名前である。`+constructs+` は behavior にあるので、`+let+` には書かない（<<constructs>>）。

引数は、behavior の引数リストの後ろに、`+requires+` が宣言順で続く。`+let+` は注入前の関数で、behavior の型は注入後の型である。だから `+let+` のほうが、`+requires+` の数だけ引数が多い（<<requires>>）。`+requires+` が空なら、両者の引数は同じである。

対応する behavior が無い `+let+` は*ヘルパー*である。ドメインの振る舞いではないので behavior の一覧を汚さず、`+>->+` の段にも置けず、`+exposing+` にも出ない。型は自分で書く。

[,text]
----
let 正の数 (v: Int) = v >= 0
let 適用する (xs: List<Int>, p: (Int) -> Bool) = List.all(p, xs)
----

戻り値の型は推論する。宣言したいならそれは仕様なので、behavior を書く。

本体は式である。`+{ ... }+` はブロック式（<<block-expression>>）で、`+let+` や `+require+` を置きたいときに書く。`+let 正の数 (v: Int) = v >= 0+` と `+let 正の数 (v: Int) = { v >= 0 }+` は同じものである。

`+=+` は名前付きの定義を導く。無名のブロック（<<blocks>>）と `+match+` のケースの本体は `+->+` で導く——型の矢印と同じ記号である（F#・Elm と同じ）。名前付きの定義にラムダを束縛する形（`+let f = (x) -> ...+`）は採らない。

==== 規則

* 同名の behavior がある `+let+` は、引数の型を書かない。型は behavior から取る。数が合わなければコンパイルエラーとする。
* 同名の behavior が無い `+let+`（ヘルパー）は、引数の型を必ず書く。
* 一つの behavior に `+let+` を二つ書くことはできない。
* `+let+` が behavior を実装できるのは、両者が*同一モジュール*にあるときだけである。別モジュールの behavior は import して呼べるが、その実装を別モジュールの `+let+` で与えることはできない。名前で束ねる規則が分割コンパイルと衝突しないための線引きである。
* ヘルパー `+let+` は data を構築できるが、その構築集合は推移的に推論され、呼ぶ behavior の `+constructs+` に含まれねばならない（<<constructs>>・<<blocks>>）。`+>->+` の段にも置けず、`+exposing+` にも書けない。仕様DSLに載らないものだからである。
* 関数型（`+(A) -> B+`）を書けるのは `+let+` の引数の位置だけである。data のフィールド（<<blocks>>）にも behavior の出力にも書けない。behavior の出力は業務上の結果の直和であって、関数は業務上の結果ではない。
* `+let+` は関数を返せる（`+let adder (n: Int) = (x) -> x + n+`）。逃げ出す関数——返り値・実行時選択——は自由変数を捕捉した閉包になる（<<blocks>>）。渡すだけ・その場で適用するだけなら展開でき、閉包は要らない。戻り型は推論するので、関数を返すときも型は書かない。ラムダの引数型も書かない——適用のされ方から推論する（<<future-work>> の高度な型推論とは別の、局所的な規則である）。

[#injected-behavior]
=== 実装を書かない behavior — 外界依存

*型だけを書いた behavior に実装が無ければ*、実装を Java から注入する。仕様DSLの `+// 依存:+` と `+// 副作用:+` に対応する。

実装とは、同名の `+let+`（<<fn-declaration>>）か、右辺の `+>->+` 合成（<<composition>>）のどちらかである。`+>->+` 合成の behavior は `+let+` を持たないが実装は持っているので、注入の対象にならない。注入されるのは、型を書いて `+let+` を書かなかったものだけである。

[,text]
----
// 仕様DSL: // 依存: 時刻取得
behavior 現在時刻 : () -> DateTime

// 仕様DSL: // 依存: 会員リポジトリ
behavior findMember : (id: 会員ID) -> 会員 | 会員なし | 保存データ不正
    constructs 会員なし, 保存データ不正
----

出力ケースはドメインの帰結だけである。`+会員なし+`（見つからない）と `+保存データ不正+`（保存値が
ドメイン不変条件を破る）は業務として意味のある結末なのでケースにする。DB ダウンのような
*プラットフォーム障害*はここに置かない ── Java Binding が例外を投げ、Souther が素通しする
（<<java-impl-rules>>・ADR-0029）。

キーワードは要らない。実装が無いことが、外から与えるという意味である。仕様DSLにも `+required+` に当たる語は無く、依存かどうかはコメントで注記されるだけである。この形なら DSL の行がそのまま残る。

`+constructs+` は実装を書かない behavior にも要る（<<constructs>>）。宣言の読み方は、Souther で実装した場合と同じである。`+findMember+` は失敗2つを新しく作り、`+会員+` は作らない。`+会員+` は外から読んだものを decoder に通して得る、と言っている。<<java-base-class>> のファクトリはここから配る。

典型用途は DBクエリ、HTTP呼び出し、ファイル読み書き、時刻取得、ID生成、メッセージ送信。これらの具体的なAPIは言語コアへ持ち込まない。依存（外部を読むだけ）と副作用（外部を変える）の区別は用途を示すドキュメントであり、値の合成規則には影響しない。

実装を書き忘れると、エラーにならずその behavior が注入対象になる。ただしそれを呼ぶ側は `+requires+` にその名前を書かねばならず（<<requires>>）、書けば `+bind+` の引数に現れる（<<jvm-behavior>>）。呼ぶ側が居なければ `+.sou+` の中では捕まらない。

どの behavior に `+let+` を書き、どれを注入にするかは、DSL から機械的には決まらない。仕様DSLの `+// 依存:+` は義務ではなく注記なので、付いていないからといって内部実装だとは限らない。<<non-functional>> の1:1 は宣言の対応である。DSL の behavior 1行が、Souther の behavior 宣言1つへ写る。実装の形（`+let+` / 注入 / `+>->+` 合成）まで決めるものではない。実装の形は、DSL の `+// 依存:+`・`+// 副作用:+` を手がかりにモデル作成者が選ぶ。注記があれば注入、内部計算なら `+let+`、状態遷移の連なりなら `+>->+`、というのが目安である。

[#java-base-class]
=== Java基底クラス生成

注入対象の behavior（型を書いて `+let+` が無いもの。<<injected-behavior>>）は*抽象基底クラス*として生成する。`+Behavior+` を継承し、`+constructs+` に挙がった単位dataごとに、それを構築する `+protected+` ファクトリを持つ。

[,text]
----
behavior findMember : (id: 会員ID) -> 会員 | 会員なし | 保存データ不正
    constructs 会員なし, 保存データ不正
----

から概念的に次を生成する。

[,java]
----
public abstract class FindMember implements Behavior {
    protected FindMember() {}
    protected final 会員なし   会員なし()   { ... }   // 宣言した単位ケースだけ
    protected final 保存データ不正 保存データ不正() { ... }
    // apply(input) は Behavior から継承（未実装）。実装が override する。
    // DB ダウンのようなプラットフォーム障害はケースでなく例外。実装が投げ、Souther が素通しする。
}
----

Java側はこれを `+extends+` して `+apply+` を実装する（<<java-example>>）。成功値（`+会員+` など）は decoder で組み立て、失敗ケースは継承した `+protected+` ファクトリで作る。data のコンストラクタは非公開なので、実装は生成パッケージの外（どのパッケージ）にも置ける。単位でない／invariant を持つ出力ケースはファクトリを生成せず、decoder 由来で得る。

この抽象基底クラスは、`+exposing+` に挙がっているかに関わらず*常に公開*する。Java 実装がどのモジュールからでも `+extends+` できるためで、注入対象は定義上モジュールの外で実装されるからである。exposing はドメイン型の公開を制御するものだが、注入基底クラスは実装のための足場なので独立に公開する。これが無いと <<example>>の注入例が成立しない。

`+constructs+` に*単位でない* data が挙がっているとき、その data が `+exposing+` で公開されていなければコンパイルエラー（E1305）とする。ファクトリは単位dataにしか生成されず、非公開 data には decoder も無いので、Java 実装がその値を作る手段が無いからである。公開されていれば decoder（<<jvm-codec>>）経由で作れる。単位data／公開data のどちらかであることを要求する。

封じ込めているのは <<invariant-on-construct>> のほうである。invariant を持つ data は、`+__construct+` を通ろうと decoder を通ろうと必ず検査される。だから外から作れても、*検証を経ていない値は作れない*。単位dataには検証すべきものが無いので、外から作れても失うものが無い。

[#java-impl-rules]
=== Java実装の規則

Java実装は、*ドメインの帰結*を返す ── 成功値（`+会員+` など）と、宣言済みの失敗ケース（`+会員なし+` / `+保存データ不正+` など）である。

一方、*プラットフォーム障害*（DB ダウン、ネットワーク断、HTTP 失敗など、ドメインの帰結でないもの）はケースに畳まず、*例外として投げてよい*。Souther はその例外をケースにせず素通しして境界へ伝播させる（生成された `+>->+` パイプラインや `+apply+` は例外を握り潰さない）。言語（`+.sou+`）に例外は無いが（E1302・<<out-of-scope>>）、外界に触れる Java Binding は投げうる。境界（Java／フレームワーク側）がその例外を受けて扱う ── HTTP 503、リトライ、トランザクションのロールバックなど。VM障害や実装バグも同じく例外として抜ける（保証対象外）。

区別の指針。*ドメインの帰結*（見つからない、在庫が足りない、保存値がドメイン不変条件を破る、など業務として意味のある結末）はケースにし、ドメイン型に現れて全域性検査の対象になる。*プラットフォーム障害*（インフラが使えない・応答しない）は例外にし、ドメイン型を汚さず横断的関心事として境界で処理する。DB ダウンをケースに畳むと、DB に触れる behavior すべてのドメイン型がインフラのケースを抱えるうえ、実際に DB が落ちたとき返すべき正直なケース値が無い（「在庫不足」と偽るか、インフラのケースをでっち上げるしかない）。だから前者はケース、後者は例外に分ける（ADR-0029）。

例外の*型*は境界と実装が合意する。jOOQ + Spring なら境界は `+org.springframework.dao.DataAccessException+` を捕まえて 503 に写す。jOOQ 自身の `+org.jooq.exception.DataAccessException+` は Spring の型のサブクラスではないので、これは jOOQ に Spring の例外変換（Spring Boot の jOOQ auto-config が既定で入れる）を効かせている前提である。素の JDBC のように検査例外（`+SQLException+`）を投げる実装は、同じ `+DataAccessException+` 系の非検査例外（例: `+DataAccessResourceFailureException+`）に包んで送出する ── `+IllegalStateException+` や裸の `+RuntimeException+` にしない（`+DataAccessException+` 型のハンドラを素通りするうえ、インフラ障害をプログラムのバグに見せる）。ハンドラはこの障害型だけを狭く捕まえ、実装バグ（例外）はそのまま一般の 500 に落とす。

`+保存データ不正+`（保存値が decode 時にドメイン不変条件を破る）はドメインのケースに残す。インフラは動いてデータを返しており、そのデータがドメイン規則を破ることは*ドメイン*についての言明だからである。名前を付けるに値するかはモデル作成者の判断で、この規則はどちらにも強制しない。

[#set-invariant]
=== 集合不変条件は入力データで渡す

1件のオブジェクトでは判定できない不変条件（日程重複、一意性など）は、隠れた依存としてではなく、判定に必要な集合を入力データとして明示する。既存申請を読む副作用はワークフローの境界（実装を書かない behavior）に置く。

[,text]
----
// 仕様DSL: behavior 日程重複を確認する = 同一申請者の出張申請リスト AND 申請準備中 -> 申請準備中 OR 日程重複
data 日程重複

behavior 日程重複を確認する : (
    同一申請者の出張申請リスト: List<出張申請>,
    申請:                    申請準備中
) -> 申請準備中 | 日程重複
    constructs 日程重複        // 申請準備中 は入力をそのまま返すので新しく作らない

// 実装を書かない behavior（外界を読む）
behavior 同一申請者の申請を読む : (申請者: 従業員ID) -> List<出張申請>
----

`+日程重複を確認する+` は純粋な判断のまま保たれ、副作用は外側の、実装を書かない behavior に寄る。これは SMDD の DDDトリレンマの結論（純粋性はレイヤー分離ではなく、どの関数が副作用を持つかを明示すること）に対応する。

[#requirement-propagation]
=== requirement propagation

実装を書かない behavior を参照するものは、その要求を引き継ぐ。要求は、名前を書いた側へ外向きに積み上がる。`+let+` が実装を書かない behavior を呼べば、その `+let+` を持つ behavior の要求になる。さらに、その behavior を段に置いた合成へ積み上がる。

積み上がった先で、書くのか推論するのかが分かれる（<<requires>>）。

* 単純な behavior で `+let+` を持つもの: `+requires+` に*書く*。`+let+` の引数として受ける
* `+>->+` 合成: *推論する*。段の要求集合の和で決まる

[,text]
----
behavior loadAndConvert = findMember >-> toView      // requires は書かない
----

これは `+findMember+` の実装を要求する。複数を合成すれば和集合になる。合成に書かせないのは、段の宣言を見れば読めるからで、仕様DSLも合成には `+// 依存:+` を付けない。

どの段階でも呼び出しグラフは静的なので、要求集合は具体的な集合として決まる。変数が入らないことが、この設計を成り立たせている。推論と宣言を突き合わせられるのも、合成の和を計算できるのも、集合が具体的だからである（<<core-semantics>>）。

これは `+let+` を高階にしても変わらない。

[,text]
----
let 適用する (xs: List<会員ID>, p: (会員ID) -> 会員表示) = List.map(p, xs)
----

`+適用する+` の要求集合は空である。`+p+` が何を要求していても関係しない。`+p+` は閉じた状態で渡ってくるからである。要求はコンストラクタ引数として注入され（<<jvm-behavior>>）、`+new 記録する(this.現在時刻)+` を作った時点で中に入る。渡された側は、それが何を要していたか知る必要がない。`+bind+` はその引数を部分適用する操作にあたる。

[#composition]
== behavior合成

[#sequential-composition]
=== 逐次合成 `+>->+`

`+>->+` は behavior どうしをつなぐ。前段の出力ケースのうち後段の入力型が受け取れるものを後段へ流し、受け取らないケースはそのまま出力へ運ぶ（本線から外れる）。

[,text]
----
behavior handle = findMember >-> toResponse
----

`+>->+` の後ろに来る段は単一入力（1引数の behavior）である。`+>->+` は1つの値を次段へ渡すので、受け取る側が2つ以上取ることはできない。複数引数の behavior を後段に置きたいときは、段にせず実装の中でインラインに呼ぶか `+match+` で受ける。

*先頭の段は制限しない。* 先頭が受け取るのはパイプライン自身の引数なので、何個でもよい。パイプラインはそれをそのまま受け取る。

[,text]
----
// 仕様DSL: behavior 却下する = 事前承認待ち AND 却下者ID -> 事前承認却下済み OR 承認権限なし
//          behavior 差し戻す = 事前承認却下済み -> 申請準備中
//          behavior 却下して差し戻す = 却下する >-> 差し戻す
behavior 却下して差し戻す = 却下する >-> 差し戻す
// 却下して差し戻す : (事前承認待ち, 従業員ID) -> 申請準備中 | 承認権限なし
----

<<type-routing>> の規則が単一入力を要するのは右辺だけで、左辺が何個取ろうと routing には関わらない。

複数入力のパイプラインは、それ自体が複数入力の behavior なので、さらに別のパイプラインの後段には置けない。先頭にはできる。

段になるのは behavior であって `+let+` ではない。`+>->+` は仕様の側の合成で、仕様DSLもそう書く。ヘルパー `+let+` は仕様DSLに現れないので、段にも置けない。

decode / encode は behavior ではなく境界の縁であり、`+>->+` の段にはならない。境界では先に decode を済ませ、成功したドメイン値だけを behavior パイプラインへ渡し、出力を encode で外へ出す（<<decoder>>・<<encoder>>）。`+>->+` が合成するのはドメインの世界（behavior とその直和出力）だけである。

[#type-routing]
=== 型ルーティング合成規則

パイプラインは*本線*を運ぶ。段に渡すのは本線だけで、段が受け取らなかったケースは本線を離れ、そのパイプラインの残りの段には*二度と渡らない*。離れたケースは出力に積まれる。

前段の出力を直和 `+O_f+`、後段の入力型を `+I_g+` とする。`+O_f+` のうち `+I_g+` が受け取れるケースを後段へ流し、残りのケース `+R+` は本線を離れる。

[,text]
----
f : A -> B | R           g : B -> C           f >-> g : A -> C | R
f : A -> B | R           g : B -> C | S       f >-> g : A -> C | S | R
f : A -> B1 | B2         g : B1 | B2 -> C     f >-> g : A -> C
----

3段以上でも同じで、*本線だけが次の段に渡る*。

[,text]
----
f : A -> B | C     g : B -> D     h : C | D -> E
f >-> g >-> h : A -> E | C
----

`+h+` は `+C+` を受け取れる。ただし `+C+` は `+g+` の時点で本線を離れているので、`+h+` には渡らない。`+h+` に渡るのは `+g+` が出した `+D+` だけである。これが Railway である。「失敗」を特別扱いしているのではない。消費されなかったケースが本線を離れるだけで、どのケースが離れるかは合成の相手が決める（<<unmarked-sum>>）。

出力の直和は open error row として扱い、`+(C | R) | S+` を平坦化し、同名ケースは畳む。`+O_f+` のどのケースも `+I_g+` に合わなければ合成できない（<<e1701>>）。

本線を離れたことは、合成が覚えている。途中に名前を付けても忘れない。

[,text]
----
behavior fg = f >-> g          // 本線 D、離脱 C
behavior x  = fg >-> h         // h に渡るのは本線 D だけ。C は離脱のまま残る。x : A -> E | C
----

`+fg+` は本線が `+D+`、離脱が `+C+` だと覚えている。`+fg >-> h+` で `+h+` に渡るのは本線の `+D+` だけで、`+C+` は離脱したまま出力へ運ばれる。だから途中に名前を付けた `+fg >-> h+` は、一本に書いた `+f >-> g >-> h+` と同じ `+A -> E | C+` になる。

`+>->+` は結合的である。部分パイプラインを名前で括り出しても、括り方を変えても、結果は変わらない。

本線と離脱の分けは合成の配管であって、値に付く印ではない（<<unmarked-sum>>）。配管はパイプラインの両端でほどける。`+fg+` を単体で見たときの出力は、本線と離脱を畳んだ直和 `+D | C+` である。外部表現も従来どおり。同じ型のケースが本線と離脱の両方に出ても、配管がどちらのレールかを覚えているので取り違えない。

後段が一部のケースしか受け取らない、あるいはケースごとに違う処理へ分岐したいときは、`+>->+` ではなく `+match+` で枝を開く。

[,text]
----
match 出張申請を提出する(申請) with
    | 提出済み as s   -> ...
    | 事前承認待ち as p -> ...
----

[#composition-with-requirements]
=== 外界依存を含む合成

behaviorの内部表現を次として捉える。

[,text]
----
Behavior<Requirements, Input, Output>
----

`+Output+` はあり得る業務結果の直和で、失敗を分ける第4の枠は持たない。合成規則:

[,text]
----
f : Behavior<R1, A, O_f>
g : Behavior<R2, B, O_g>        // I_g = B が O_f のケースを受け取る
f >-> g : Behavior<R1 ∪ R2, A, (O_g) | (O_f から消費分を除いた和)>
----

`+R1+` と `+R2+` は段が宣言したものである（<<requires>>）。合成の `+R1 ∪ R2+` は合成が推論する。合成に `+requires+` は書かない。実装が内部か Java 側かは、値の合成規則に影響しない。`+Requirements+`（必要とする依存）と `+Output+`（生み出す結果の直和）は、独立に合成される。前者は集合和、後者は消費／伝播で決まる。

`+Output+` の消費／伝播は、後段へ渡る*本線*と、消費されず離脱したケースの2つに分かれる。この分けを合成が持ち回るので、`+R1 ∪ R2+` の集合和と同じく、`+>->+` は結合的になる（<<type-routing>>）。本線／離脱の分けは合成の配管に留まり、`+Output+` として外へ出るときは畳まれた直和に戻る。内部表現に失敗用の第4の枠は要らない。

[#single-case-composition]
=== 単一ケース出力の合成

出力が単一ケース（`+-> A+`）の behavior は、素通りするケースが無い（失敗row が空、`+Never+`）とみなせる。後段が `+A+` を受け取ればそのまま流れる。利用者が明示的に何かで包む必要はない。`+require ... else+` で早期に別ケースを返した場合、そのケースは <<type-routing>> の `+R+` として伝播する。

[#declared-composition-output]
=== 合成の出力を宣言して突き合わせる

`+>->+` 合成の出力は段から推論する（<<composition-with-requirements>>）。仕様DSL の合成行は出力を書かない（`+behavior 注文を確定する = 注文を検証する >-> ...+`）ので、既定はこの推論のままである。

推論した出力は、段が別モジュールの behavior だと*遠くの変更で黙って変わる*。上流の段がケースを増やせば合成の出力が黙って育ち、下流の段が入力を狭めれば、それまで処理されていたケースが離脱に回って（<<type-routing>>）やはり出力に増える。合成の定義側には、これを知らせるものが無い。

そこで、合成には出力を*任意で宣言できる*。書くと、推論した出力と*正確に一致*しなければコンパイルエラー（E1604）になる。過小（宣言に無いケースが出る）も過大（宣言したケースが出ない）も通さない。

[,text]
----
behavior 注文を確定する = 注文を検証する >-> 価格を記載する >-> 在庫を引き当てる
    -> 確定注文 | 在庫不足
----

`+-> 確定注文 | 在庫不足+` は合成の宣言であって、最後の段 `+在庫を引き当てる+` の出力ではない。段は behavior 参照（名前だけ）なので、パイプラインの後ろの `+->+` は合成のものと一意に読める。この行を書いておくと、上流がケースを増やしたときにここがエラーになり、下流まで黙って育たない。

入力は宣言しない。パイプラインは point-free で、入力は先頭の段が決めるからである（<<sequential-composition>>）。宣言できるのは出力だけである。依存（`+requires+`）も同じ遠隔変更の危険を持つが、合成では推論のままにする（<<requires>>）。

内部の合成ではこの宣言は任意である。だが `+exposing+` で公開する合成では*必須*とし、場所は定義行ではなく exposing の列に `+名前 : ケース+` と書く（<<modules>>）。公開境界を推論任せにすると、上流の段がケースを増やしたとき、別コンパイル単位（別モジュールや Java の非網羅 switch・JSON 直列化）にそれが黙って届く。定義側には知らせるものが無い。exposing に署名を置けば、増えた瞬間にその合成の定義がエラー（E1604）になり、書き手はケースを処理するか署名を更新するかを選ぶ。署名が無ければ E1605 とする。非合成の behavior は定義で型を書いているので、exposing に署名は書けない（書けば E1605）。この分担は ADR-0024 に対応する。

[#case-propagation]
== behaviorのケース伝播とDecoderのエラー集積

behavior の出力直和は open error row として扱い、`+>->+` で後段が消費しないケースは素通りで末尾へ伝播する（<<type-routing>>）。これがドメインの世界の合成様式である。

decode 側のエラー集積は縁のライブラリ Raoh が担う。導出された Object の Decoder は、各フィールドを同名キーから読み、独立したフィールドの失敗をすべて集める（Applicative）。

[,text]
----
{ "name": ..., "email": ..., "age": ... }
// name / email / age を各フィールドの導出Decoderで読み、独立した失敗をすべて集積する（Raoh）
----

導出Decoderのフィールド取得は互いに独立で、失敗をすべて集める。集積した失敗は Raoh の `+Result+` の失敗（`+Issues+`）にまとまる。カスタム境界コーデック（<<custom-codec>>）では、ある取得が先行の結果に依存することがある。その場合はその箇所から逐次（monadic）になり、先行が失敗すれば後続は実行しない。

集積（applicative）は decode 境界の中の話、ケース伝播は behavior 合成の話で、別の層に属する。decode は `+>->+` の段ではないので、両者が同じ直和上で混ざることはない（<<sequential-composition>>）。集積が成り立つのは、Decoder の各フィールド取得が互いに独立（前の値に依存しない）だからである。

[#control-flow]
== 制御構文

[#let]
=== let

[,text]
----
let normalized = String.trim(input)
----

再代入は禁止する。`+let+` はブロック式（<<block-expression>>）の中に置く。`+let x = v+` に続く残りがその束縛の届く範囲で、これは脱糖の時点で入れ子になる。

[#if]
=== if

[,text]
----
if condition then expression1 else expression2
----

`+if+` は式である。両分岐の型が一致すればその型に、異なる data なら両者を並べた直和になる（`+if c then 提出済み{...} else 事前承認待ち{...}+` は `+提出済み | 事前承認待ち+`）。プリミティブどうしが食い違う場合はコンパイルエラーとする。

[#match]
=== match

[,text]
----
match 出張申請 with
    | 申請準備中 as d   -> ...
    | 提出済み as s     -> ...
    | 事前承認待ち as p  -> ...
    | 事前承認済み as a  -> ...
----

スクルーティニの後ろに `+with+` を置き、ケースを `+|+` で1本ずつ並べる（F#と同じ形）。先頭ケースの `+|+` は任意である。波括弧は使わない。Souther はレイアウト非依存なので、Elm の `+case ... of+`（インデント依存）は採れず、`+|+` 区切りだけがレイアウト無しで成立する。

ケースの値は `+as+` で束縛する（`+| 提出済み as s+` の `+s+` は 提出済み で、`+s.提出日時+` と読む）。`+as+` はマッチした値全体に束縛する。

フィールドは、構築の逆向きの分解パターンで直接束縛できる。構築 `+会員 { id = x }+` と同じ `+{ }+`・`+,+`・`+=+` を使い、`+{ id }+` は同名変数への束縛のショートハンドである（<<record-literal>> と対）。使うフィールドだけ書けばよい。分解できるのは単一の名前付きケースだけで、or-pattern（下記）は直和型に束縛するのでフィールドを持たない。`+as+` と併用でき、`+| 会員 { id } as 全体+` はフィールドと全体の両方を束縛する。

[,text]
----
match c with
    | EmailContact { email = e } -> ...   // e は email フィールド
    | 会員 { id, メール }         -> ...   // { f } は f を同名変数に束縛
----

複数のケースを同じ処理に流したいときは、`+| A | B -> ...+` とケース内で `+|+` を並べる（or-pattern）。状態機械の直和で「これらの状態は同じ扱い」を1本で書ける。ケース区切りの `+|+` と or-pattern の `+|+` は同じ記号だが、`+->+` がケースのパターン部を終えるので、両者が衝突することはない。

[,text]
----
match 出張申請 with
    | 提出済み | 事前承認待ち as x -> ...   // x は 出張申請（直和型）
    | 申請準備中 as d -> ...
    | 事前承認済み as a -> ...
----

or-pattern の束縛は、単一ケースと扱いが違う。単一ケースの `+as+` はそのケースの型に束縛する（`+| 提出済み as s+` の `+s+` は `+提出済み+`）が、or-pattern の `+as+` は*スクルーティニの直和型*に束縛する。どれか1つのケース型では全ての選択肢に合わないからである。だから or-pattern の束縛からはケース固有のフィールドを読めず、直和のまま持ち回るか、もう一度 match で開く。網羅性はケースを1つずつ数える。あるケースを2つのケースが受けると重複エラーになる。この束縛規則は OCaml・Rust の or-pattern（全ての選択肢で同じ型に束縛する）と同じ立場である。

`+Option+`（<<algebraic-types>>）の `+Some+` は、包まれた値を*位置束縛*で取り出す。`+| Some v+` の `+v+` は `+Some+` ではなく中身に束縛される（F#・Elm の `+Some x+` / `+Just x+` と同じ）。`+Some+` は組み込みで唯一ペイロードを持つケースなので、`+as+`（他では値全体に束縛する）ではなく位置束縛で中身に届く。`+| Some as v+` とは書けない。`+?+` フィールドと `+List.get+` / `+Map.get+` の戻り値は、これで取り出す。

[,text]
----
data 出張完了 = { 事前承認者: 従業員ID? }

let 承認者の表示 (c: 出張完了) =
    match c.事前承認者 with
        | Some 承認者 -> 承認者.value      // 承認者 は 従業員ID（中身）
        | None        -> "なし"
----

直和dataに対するmatchは網羅的でなければならない。網羅性は直和が宣言した直接のケースについて判定する。入れ子の直和（`+data 費用負担区分 = 自社負担 | 先方負担+` の `+自社負担+` がさらに `+立替 | 仮払い | 会社カード+`）は、葉のケースを平らに並べない。`+自社負担+` を受けて、入れ子の `+match+` で分ける。各階層で独立に網羅性を見るので、ケースが重なることがない。ケース→直和の代入（<<sum-data>>）は葉まで畳んで判定する。match はその逆で、下方向に階層をたどる。

[#require]
=== require

[,text]
----
require amount <= balance
    else 残高不足
----

`+require+` は、出力に離脱ケースを持つ behavior の実装（`+let+`）の中で使う。条件が偽なら `+else+` の値を返してその場で終える。`+else+` の値は出力の直和のいずれかのケース（ただの data）である。実装は、返したいケースの値を最後の式に置くだけでよい。特別な包み込みは無い。

`+require+` は `+if+` への糖衣である。次の二つは同じ意味を持つ。

[,text]
----
require <条件> else <値>
<残りの本体>
----

[,text]
----
if <条件> then <残りの本体> else <値>
----

脱糖は構文解析の時点で行い、以降の段（型検査・生成権限検査・バックエンド）は `+if+` しか見ない。両分岐の型が違ってよいこと（<<if>>）が、この脱糖を成り立たせている。

[#block-expression]
=== ブロック式

`+{ ... }+` は式である。`+let+` や `+require+` を並べ、最後に結果の式を置く。

[,text]
----
let 事前承認する (申請, 承認者ID) = {
    require 承認者ID == 申請.申請者.上長ID
        else 承認権限なし

    事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
}
----

置くものが結果の式だけなら、ブロックは要らない。`+let 正の数 (v: Int) = v >= 0+` と `+let 正の数 (v: Int) = { v >= 0 }+` は同じものである。

ブロックは式なので、式が置ける場所ならどこにでも書ける（`+if c then { let x = f(a)  g(x) } else h+`）。中身は <<let>> と <<require>> の規則どおり脱糖され、AST には束縛と `+if+` として現れる。裸の `+{+` はブロック式で確定する。レコードリテラルは型名が前に付く（<<record-literal>>）ので、取り違えは起きない。

[#purity]
== 純粋性

Souther のコードは*外界に直接触れない*。次を禁止する。

可変変数、フィールド代入、static mutable field、乱数、現在時刻、環境変数、ファイル、ネットワーク、スレッド、例外、リフレクション。

外界依存は実装を書かない behavior として宣言し、実装は Java 側から注入する（<<injected-behavior>>）。`+let+` はそれを引数として受け取って呼べるので、`+let+` の結果が外界に左右されることはある。ただし触れ方は必ず注入されたものの呼び出しを経由し、その事実は `+requires+` に書かれる（<<requires>>）。これが SMDD の DDDトリレンマの結論——純粋性はレイヤーの分離ではなく、どの関数が副作用を持つかを明示すること——にあたる（<<set-invariant>>）。*Souther はこれを字義どおり「明示」にした。* 推論に任せれば、どの関数が副作用を持つかは宣言を読んでも分からない。

*「要求集合が空である」ことは、この意味の純粋性より強い。* この章が言うのは「直接は触れない」ことで、すべての `+let+` が満たす。要求集合が空とは、注入されたものを一つも受け取らないことで、一部の `+let+` だけが満たす。invariant から呼べるのは後者である（<<invariant-expressions>>）。二つを混同しないよう、以降は後者を「要求集合が空」と呼び、「純粋」とは呼ばない。

[#stdlib]
== 標準ライブラリ

[#reserved-namespace]
=== 予約名前空間

標準ライブラリはコンパイラが同梱し、短い修飾子で参照する——`+List+`（リスト）・`+String+`（文字列）・`+Map+`（マップ）・`+Bool+`（真偽）・`+Int+` / `+Decimal+`（数）。関数はすべてこの修飾子を通して呼ぶ：`+List.map(f, xs)+`・`+String.trim(s)+`・`+List.length(xs)+`。自動 import は無い——名前は暗黙には届かない。裸で書けるのは演算子（`+==+` `+<+` `+{plus}{plus}+` 等）・`+if+` / `+match+` ・リテラル・自モジュールのヘルパーだけである。引数の並びは <<pipe>> の規約に従う（主対象は最後）。

裸で呼びたい名前は明示的に取り込む：`+import List ( map, filter )+` と書けば、その先は `+map(f, xs)+` と修飾なしで書ける。Elm の `+import List exposing (map)+` と同じで、修飾アクセスは常に効き、`+import+` は修飾子を省ける砂糖にすぎない。同じ裸の名前を二つのモジュールから取り込むと曖昧なので、そのときは片方を修飾で呼ぶ（`+String.length+` と `+List.length+` を一つのモジュールで両方使うなど。<<imports>>）。

修飾子 `+List+` 等は式の位置でだけモジュールを指す。型の位置の `+List<T>+`（山括弧）とは別で、衝突しない（Elm と同じ）。`+length+` / `+get+` は `+String+` と `+List+`、`+List+` と `+Map+` にまたがるが、`+String.length+` / `+List.length+`、`+List.get+` / `+Map.get+` と別の名前になるので、多重定義ではなくそれぞれのモジュールの関数である。

短い修飾子が指す実体は予約名前空間 `+souther.string+` / `+souther.list+` / `+souther.map+` / `+souther.bool+` に置かれる。`+souther+` 名前空間は予約されていて、ユーザーモジュールはこの名前を名乗れない——`+module souther.foo+` はコンパイルエラーとする。予約名前空間だけが core の特権（総称・再帰・intrinsic）を持つので、ユーザーが自分にそれを与えられないようにするための線引きである（<<algebraic-types>> の有界性）。`+southernmost+` のように文字列として `+souther+` で始まるだけの名前は、この名前空間の下ではないので通常のユーザー名である。

[#type-variables]
==== 型変数

core の関数は型変数 `+'a+`（F#・OCaml と同じ先頭アポストロフィ）を型位置に書ける——パラメータ・戻り値・`+List<'a>+`・`+Map<String, 'a>+`・関数型 `+('a) -> 'b+` のいずれでも。宣言リスト（`+<A>+`）は書かず、シグネチャに現れた `+'a+` が暗黙に全称量化される（F# の auto-generalization）。識別子は日本語を含み大小を区別しない（<<unit-data>>）ので、Haskell・Elm の「小文字＝型変数」は採れない。先頭アポストロフィだけが大小非区別のまま成立する ML 系の解である。

型変数を書けるのは予約名前空間の core だけで、ユーザーモジュールが `+'a+` を書くとコンパイルエラーとする。ユーザーモデルは有界のまま保つ（<<algebraic-types>>）。非再帰の core ヘルパーが `+'a+` を持つときは、呼び出し位置への inline 展開で単相化される——`+'a+` は各呼び出しの具体型に解決され、多相メソッドは生成されない。

[#intrinsics]
==== intrinsic

プリミティブは core の関数として宣言し、本体に `+= intrinsic "key"+` と書く。`+key+` は Backend が吐く JVM 操作を指す名前で、`+"string.trim"+` のように*型ごとに前置*して一意にする（`+list.length+` と `+string.length+` を分ける）。

[,text]
----
let trim (s: String) : String = intrinsic "string.trim"
----

intrinsic は戻り値型を必ず宣言する（`+intrinsic+` からは推論できない）。呼び出しは宣言シグネチャに対して型検査され——つまり `+trim+` は「シグネチャが core に置かれた組み込み関数」で、コンパイラのハードコードした表ではなくこの宣言が引数と戻り値を決める——Backend が `+key+` に対応するバイトコードを吐く。inline も lower もしない。書けるのは予約名前空間だけである。ユーザーは JVM を直接呼べず、blessed な関数を呼ぶだけになる（<<asymmetric-interop>>）。

[#pipe]
=== 値パイプと引数の並び

値を関数へ順に通すとき、`+|>+` で左から右へ書ける。`+e |> f(a)+` は `+f(a, e)+`——左辺が右辺呼び出しの*最後の引数*に入る。括弧なしの関数名 `+e |> f+` は `+f(e)+` になる。`+|>+` は最も結合が緩く左結合なので、`+a |> f |> g+` は `+g(f(a))+` と読める。

[,text]
----
入力.生
    |> split(",")                          // split(",", 入力.生)
    |> map(片 -> 片 |> trim |> lowercase)
    |> filter(片 -> String.length(片) >= 1)
    |> sort                                // sort(...)
----

標準ライブラリの引数はこれに合わせて F#・Elm の並びにしてある——*主対象（コレクション・文字列・Map）を最後に、関数や区切りやキーを先に*置く。`+List.map(f, xs)+`・`+List.filter(p, xs)+`・`+List.fold(step, seed, xs)+`・`+String.split(sep, s)+`・`+Map.get(key, m)+`。だから左辺の値が最後の引数に落ちて、パイプが上から下へ流れる。`+fold+` の畳み込みブロックは `+(acc, x)+` の順のまま（F# の `+List.fold+`）。

右辺に書けるのは関数呼び出しか関数名だけで、`+e |> 3+` はコンパイルエラーとする。`+|>+` はパース時に呼び出しへ展開されるので、以降のどの段も `+|>+` を見ない——`+require+` が `+if+` に展開されるのと同じ純粋な構文糖である。behavior 段を合成する `+>->+`（<<composition>>）とは別で、あちらは behavior を、こちらは式の値をつなぐ。

[#stdlib-string]
=== String

[,text]
----
length  trim  lowercase  uppercase  contains  startsWith  endsWith  substring  concat
split    join  replace    words      fromInt
----

下段は Elm の String モジュールから、Souther の型（`+Char+` / `+Maybe+` / `+Float+` を持たない）で表せる中核を移した分である。`+split(sep, s)+` は区切りで割って `+List<String>+` にする——空片も残す（`+split(",", "a,,b")+` は `+["a", "", "b"]+`）。`+words(s)+` は空白の連なりで割り、空片は落とす。`+join(sep, xs)+` は区切りで連ねる。`+replace(target, replacement, s)+` は現れた `+target+` を残らず置き換える。`+fromInt(n)+` は `+Int+` を10進表記の文字列にする。引数は主対象の文字列を最後に置く（<<pipe>>）。Elm でも `+String.map+` / `+toInt+`（`+Maybe+` を返す）のように `+Char+` や `+Maybe+` に依るものは、対応する型が無いので移していない。

[#stdlib-int]
=== Int

[,text]
----
add  subtract  multiply  divide  remainder  compare
----

`+Int+` は符号付き64ビットである（<<primitives>>）。演算がその範囲を超えたら*中断する*——ラップしない。桁あふれはモデルのバグであって業務結果ではないので、invariant 違反と同じく `+ConstraintViolation+` を投げる（<<algebraic-types>>・<<violation-destination>>・<<jvm-abort>>）。

ゼロ除算は `+Int | DivisionByZero+` を返す。こちらは業務フロー上あり得る入力（割る数が0）なので、中断ではなくケースで返す。

四則は演算子 `{plus}` `+-+` `+*+` `+/+` でも書ける（<<primitives>> の記号表）。`+/+` は切り捨て除算で、ゼロ除算では*中断する*（桁あふれと同じ）。`+Int | DivisionByZero+` をケースで受けたいときは関数の `+divide+` / `+remainder+` を使う——演算子＝気軽・中断、関数＝ケースで安全、の使い分けである。

[#stdlib-decimal]
=== Decimal

[,text]
----
add  subtract  multiply  divide  compare
----

除算は丸めスケールと丸めモードを引数で明示する。`+Decimal.divide(a, b, 2, HALF_UP)+` は小数2桁で HALF_UP 丸めをする。丸めモードは組み込みの識別子で、`+None+` と同じく組み込みの値として解決する（<<unit-data>>）。

[,text]
----
HALF_UP  HALF_EVEN  HALF_DOWN  UP  DOWN  CEILING  FLOOR
----

`+java.math.RoundingMode+` に対応する。スケールとモードを省いた `+Decimal.divide(a, b)+` は書けない——丸めをドメインの決めごととして表に出すためで、これは <<primitives>> の「スケールは偶然」と同じ立場である。ゼロ除算は `+Decimal | DivisionByZero+` を返す。

四則は演算子 `{plus}` `+-+` `+*+` `+/+` でも書ける。演算子の `+/+` は既定の丸め——F#/.NET の `+System.Decimal+` に倣った有効桁（約29桁）・HALF_UP——で割るので、`+10m / 3m+` は `+3.3333…+` になり中断しない。ゼロ除算は*中断する*。丸めをドメインの決めごととして明示したいとき、あるいはゼロ除算をケースで受けたいときは、引き続き関数 `+Decimal.divide(a, b, scale, mode)+` を使う。

[#stdlib-list]
=== List

==== 構築

リストはリテラルかガード内包で作る。

[,text]
----
[e1, e2, ...]         // リテラル。1つ以上の要素
[要素 | 条件, ...]     // ガード内包。条件が全部成り立てば1要素、でなければ空
----

空のリテラル `+[]+` も書ける。それ自身は要素型を持たず、型は文脈から決まる——`{plus}{plus}` の相方、`+if+` / `+match+` のもう一方のケース、`+fold+` の種が育つ先、あるいはフィールドや戻り値が求める `+List<T>+`。空リストは実行時に要素へ依らないので、これは常に健全である。DSLの箇条書きを写すには、引き続きガード内包（条件が偽なら空）が素直である。

`{plus}{plus}` は連結である。両辺ともリストでなければならない。

要素型は、同じなら畳み、違う data どうしならケースを並べた直和に広げる（<<sum-data>> の向きと同じ）。

[,text]
----
[高額出張 { 基準金額 = 基準額 }] ++ [先方費用負担]      // List<高額出張 | 先方費用負担>
----

ガード内包と `{plus}{plus}` は、仕様DSLが箇条書きで並べた条件を写すためにある。

[,text]
----
// 仕様DSL（chapter10）が理由を箇条書きで並べている:
//   理由1: 予定費用が10万円以上 / 理由2: 申請者が一般社員 / 理由3: 費用負担が先方
   [高額出張 { 基準金額 = 基準 } | 予定費用.value >= 基準.value]
++ [権限不足 { value = 申請者.役職 } | 一般社員か(申請者)]
++ [先方費用負担                 | 先方負担か(費用負担区分)]
----

理由ごとに0要素か1要素のリストを作り、連結する。*DSLの箇条書き1行が、Souther の1行になる。* 結果は `+List<高額出張 | 権限不足 | 先方費用負担>+` で、`+事前承認理由+` の3ケースと一致する。単位dataは名前だけで構築するので（<<unit-data>>）、`+先方費用負担+` はそのまま書ける。

条件に `+申請者.役職 == 一般社員+` と書けないことに注意する。`+==+` は同じ型どうしにしか使えず（<<primitives>>）、`+役職+` と そのケースである `+一般社員+` は別の型である。ケースの判定は `+match+` で行い、述語にしたければヘルパー `+let+` に包む（上の `+一般社員か+`）。順序（`+>=+`）は `+Int+` / `+String+` / `+Decimal+` / `+Date+` / `+DateTime+` にしか無いので、`+金額+`（newtype）どうしではなく `+.value+` を比べる。

==== 操作

[,text]
----
map  filter  fold  all  any  length  get
reverse  member  sum  product  concat  isEmpty  sort
----

どれも修飾子 `+List+` を通して呼ぶ（<<stdlib>>）。`+List.get+` は `+Option<T>+` を返す。`+List.map+` / `+List.filter+` / `+List.fold+` / `+List.all+` / `+List.any+` はブロックを引数に取る（<<blocks>>）。`+import List ( map, all )+` と取り込めば、その先は修飾なしで書ける。

[,text]
----
import List ( map, all )

map(x -> 明細を直す(x), xs)
all(r -> 軽微か(r), 理由リスト)
----

ブロックの中から注入されたものを呼べば、その要求は囲む behavior の `+requires+` に現れる（<<requires>>）。

これらのうち原始的なのは `+List.fold+` 一つだけである。`+map+` / `+filter+` / `+all+` / `+any+` は `+souther.list+` が `+List.fold+` から導く自言語の `+let+` で（<<stdlib>>・ADR-0028）、呼び出し位置に展開される。`+map+` / `+filter+` は空リスト `+[]+` を種にして育て、`+all+` / `+any+` は `+Bool+` を畳む。`+List.length+` / `+String.length+`、`+List.get+` / `+Map.get+` は、かつて一つの多重定義だったが、修飾で別々のモジュールの関数に分かれた——`+String+` と `+List+` に跨る `+length+` を、それぞれの名前で呼ぶ。

下段も Elm の List モジュールからの移植で、`+sort+` を除いてどれも `+List.fold+` 由来の自言語の `+let+` である。`+reverse+` は種を先頭側に伸ばし、`+sum+` / `+product+` は数を畳み、`+concat+` は `{plus}{plus}` で畳んで入れ子を平らにし（Elm の `+List.concat+`）、`+member+` は `+==+` で畳むので比較できる要素なら効き、`+isEmpty+` は長さを見る。`+List.concat+` と `+String.concat+` は名前が同じだが、修飾で別々のモジュールの関数に分かれるので共存する。`+sort+` だけは（Elm と同じく）`+fold+` から導けない原始的なプリミティブである——自言語の並べ替えは比較に加えて途中結果を置く場所が要るが、`+fold+` はそれを与えないからで、要素の自然順で並べる。要素は順序を持つ型（`+Int+` / `+String+` / `+Decimal+` / `+Date+` / `+DateTime+`。<<primitives>>）でなければならず、data や newtype のリストを `+sort+` に渡すのはコンパイルエラーとする（それらは順序を持たないので、実行時に落とすのでなく先に弾く。並べたいなら `+List.map+` で順序を持つフィールドへ写してから渡す）。`+List.head+` / `+maximum+` のように `+Maybe+` を返すものや `+sortBy+`（比較関数をランタイムの値として渡す必要がある）は移していない。

=== Map

[,text]
----
get  containsKey  keys  values
----

`+get+` は `+Option<V>+` を返す。

[#stdlib-bool]
=== Bool

[,text]
----
not
----

`+not+` は `+souther.bool+` に置かれた**最初の self-host 関数**である。プリミティブでなく Souther で `+let not (b: Bool) = if b then false else true+` と書かれ、`+Bool.not+` として修飾で呼ぶ（`+import Bool ( not )+` で裸に）。非再帰なので呼び出し位置に inline 展開され、専用のクラスは生成されない。否定はこの `+not+` で書く——前置 `+!+` 演算子は持たない（`+not+` があるので冗長）。

Souther の構文には現れない。data の形状から導出された decoder / encoder は、Raoh の Decoder / Encoder として ClassFile backend が生成する（<<codec-generation>>、<<encoder>>）。プリミティブ解析・エラー集積・判別振り分け・エンコードは Raoh の combinator が担い、souther-runtime 側に facade は持たない。境界のカスタムコーデック（<<custom-codec>>）は Java 側から Raoh を直接使う。

[#jvm-output]
== JVM出力

[#target-jdk]
=== 対象JDK

生成物はJava 21以上で動作することを目標とする。コンパイラ実装はJava 25を使用してよい。Unicode識別子はJavaの識別子規則に沿って生成する（必要ならエスケープ／ローマ字化はbackendの責務）。

生成する `+.class+` のクラスファイルバージョンは Java 21（major 65）に固定する。バックエンドを動かしたJDKの既定に任せてはならない。souther-runtime も生成物側なので、同じく Java 21 を対象にコンパイルする（コンパイラ本体だけが Java 25 でよい）。

[#jvm-product]
=== 直積data

private constructorを持つfinal classとして生成する。`+...+` スプレッドで取り込んだフィールドは、生成先のクラスに平らに展開する。

[,text]
----
public final class 提出済み {
    private final 従業員 申請者;      // ...出張申請共通項目 由来
    private final 金額 予定費用;       // ...出張申請共通項目 由来
    // ...
    private final LocalDateTime 提出日時;

    private 提出済み(...) { ... }
}
----

`+exposing+` した data には、フィールドの公開読み取りアクセサを生成する（<<field-visibility>>）。別モジュールと Java からフィールドを読めるようにするためである。モジュール＝パッケージなので、package-private では越境で読めない。コンストラクタは非公開のままなので、読めても構築はできない。公開しない data のフィールドは同一モジュール内からだけ読め、アクセサは生成しない。

[#jvm-sum]
=== 直和data

sealed interface と final 実装クラスとして生成する。ケースが名前付きdataなら、その final class が sealed interface を implements する。単位dataはフィールドなしの final class（またはシングルトン）として生成する。

[#jvm-codec]
=== Decoder / Encoder

公開dataには入力源ごとの取得APIを生成する。返すのは Raoh の Decoder / Encoder である（`+Decoder<I, T>+` は `+I -> Result<T>+`）。

[,text]
----
public static Decoder<Map<String,Object>, 出張申請> decoder()  // Map（raoh-core。newtype は Decoder<Object,T>）
public static Decoder<JsonNode, 出張申請> jsonDecoder()    // JSON（raoh-json）
public static Decoder<Record,   出張申請> recordDecoder()  // DB 行（raoh-jooq）
public static Encoder<出張申請, Map<String,Object>> encoder()  // object/sum -> Map（JSON化は境界で valueToTree）
// newtype は裸のスカラへ encode する。例: 会員ID.encoder() : Encoder<会員ID, String>
----

具体的な実装は ClassFile backend が Raoh の combinator を直接バイトコードとして生成する（<<codec-generation>>）。souther-runtime は Raoh に依存せず、Raoh 依存は生成コードとアプリ側にある。`+jsonDecoder()+` / `+recordDecoder()+` はその data の形状が対応する場合にのみ生成する（<<codec-generation>>。JSON は temporal 無しの形状、`+Record+` は平坦な形状）。`+decoder()+`（素の値 / Map）は常に生成する。

[#jvm-behavior]
=== behavior

トップレベルbehaviorはJavaの関数オブジェクトまたはstatic APIとして生成する。要求集合が空のbehaviorは直接呼び出せる。要求集合を持つbehaviorには、依存実装を束縛するAPIを生成する。

[,text]
----
var handle = Handle.bind(new JdbcFindMember(dataSource));
----

生成されるクラス名は behavior 名から導く（先頭を大文字化）。data 名はもともと大文字始まりなので、`+quote+` という behavior は `+Quote+` というクラスになり、同じモジュールの `+data Quote+` と衝突する。これはコンパイルエラーで、どちらかを改名して区別する。

[#jvm-construction-privacy]
=== 生成権限の露出防止

dataのconstructorは非公開（package-private）、`+__construct+` などの内部生成メソッドも非公開にする。実装を書かない behavior の Java 実装は、生成された抽象基底クラス（<<java-base-class>>）を `+extends+` し、失敗ケースを基底の `+protected+` ファクトリ経由で構築する。data のコンストラクタを呼ぶ抜け道は無いので、実装は生成パッケージの外に置ける（同一パッケージ配置は不要）。

Java 側に対して保証するのは、検証を経ていない値が作れないことである。値が作れないこと、ではない。`+decoder()+` は公開API（<<jvm-codec>>）なので、Java からいつでも decode を試みられる。ただし invariant はそこで検査され（<<invariant-on-construct>>）、違反は `+Result+` の失敗になる。単位dataは入力を無視して decode する（<<codec-generation>>）ので、その `+decoder()+` は誰にでも構築を許す。ただし単位dataには検証すべきものが無い。

Reflection、Unsafe、バイトコード改変は保証対象外とする。

[#jvm-abort]
=== invariant 違反の中断

ドメインの中の invariant 違反（<<algebraic-types>>・<<violation-destination>>）は、生成コードでは `+net.unit8.souther.runtime.ConstraintViolation+` を投げる形で現れる。破った data 名と invariant の位置を持つ。

Souther のコードはこれを捕まえられない。捕まえる構文が無いので、業務フローの制御には使えない（<<out-of-scope>>）。境界の Java は捕まえられるが、業務の失敗とは別のものとして扱う。業務の失敗は出力のケースで届き（<<unmarked-sum>>）、それは 400 に写る類のものである。中断はモデルのバグなので、500 に写る。

decode の違反はこれではない。Raoh の `+Result+` の失敗として返る（<<decoder-error>>）。

[#jvm-anonymous-union]
=== behavior 出力の無名 union

behavior の出力が無名 union（`+-> 会員 | 会員なし+`）のとき、その union を表す sealed interface を生成する。名前は behavior のクラス名（先頭を大文字化したもの。<<jvm-behavior>>）に `+結果+` を付けた `+<Behavior名>結果+`、`+permits+` は union の葉のケース、各ケース data がこれを implements する。無名 union は共通スーパータイプを持たない（<<asymmetric-interop>>）ので、これで Java 側は単一の戻り型として受け取り、`+switch+` で網羅検査できる。

[,java]
----
// findMember が -> 会員 | 会員なし のとき（13.3・24章と同じ behavior）
sealed interface FindMember結果 permits 会員, 会員なし {}
// 会員・会員なし がこれを implements する
Behavior<会員ID, FindMember結果>
----

生成しない場合が二つある。出力が名前付き直和 data なら、それ自体が sealed interface（<<jvm-sum>>）なのでその型を戻り型にする。出力が単一ケースなら、そのケース型をそのまま戻り型にする。

一つの data 型は複数の behavior の出力に現れうるので、複数の `+<Behavior名>結果+` を同時に implements することがある。`+>->+` 合成 behavior の出力（本線と離脱を畳んだ直和。<<type-routing>>）も無名 union なので、合成名から同じ規則で生成する。生成した型の encode は直和と同じく判別子 `+"type"+` でケースを振り分ける（<<sum-discrimination>>）。

この型が表に出るのは注入 behavior だけである。Java 実装が戻り型として書く（<<java-example>>）。let や `+>->+` 合成の behavior では、Java 消費側は `+var+` と `+switch+` で受けるので生成コード内部に留まる。モデル作者は Souther のソースにこの型名を書かない。`+-> 会員 | 会員なし+` と書くだけである。

[#compiler-pipeline]
== コンパイラ構成

[,text]
----
Source -> Lexer -> Parser -> AST
  -> Derive (data の形状から decoder/encoder を AST に充填)
  -> Type Check
       名前解決 / 生成権限検査 / 要求集合の検査と合成での推論 / 網羅性検査 / invariant 型検査
  -> ClassFile Backend -> .class（バイトコード直接生成）
----

型検査は AST 上で行い、生成権限・要求集合・網羅性・invariant を一括で確かめる。独立した Domain IR / Decoder IR / Encoder IR は持たず、バックエンドが AST から直接バイトコードを出す。バックエンドは Class-File API（`+java.lang.classfile+`）で `+.class+` を生成し、javac は経由しない。将来 Java ソース生成など別の backend が要るときは、この段に IR を挟む余地を残している。

[#ast-tracking]
== AST とバックエンドが追跡する情報

独立した中間表現は持たず（<<compiler-pipeline>>）、次を AST と型検査の結果として保持する。

[#ast-nodes]
=== AST の定義ノード

[,text]
----
Module  Data(ProductData)  SumData  UnitData  Include  Field  Invariant
DecoderDef  EncoderDef  SpecBehavior  PipeBehavior  FnDef  Import
Construct  FieldInit  Expression  Case  RawExpr  DecRef  Block
----

behavior の定義ノードは、<<behavior>>の二つの右辺に対応して二つある。`+SpecBehavior+` は入力型・出力型・`+constructs+` だけを持ち、本体を持たない。同名の `+FnDef+` があればそれが実装で、無ければ Java 注入の対象になる（<<injected-behavior>>）。`+PipeBehavior+` は段の名前の並びを持つ。それ自体が実装なので `+FnDef+` を取らず、注入もされない。型は段から計算する（<<type-routing>>）。合成か単純かはこの二つのノードで区別され、後段はどちらかを見分けるために右辺を読み直さなくてよい。

`+FnDef+` は引数名と本体を持つ。同名の `+SpecBehavior+` があれば型はそちらから取り、無ければヘルパー `+let+` なので引数の型も自分で持つ（<<fn-declaration>>）。

`+SumData+` のケースは名前付きdataへの参照だけを持つ（<<sum-data>>）。`+...+` スプレッドは取り込むフィールドと invariant を平らに引き継ぐ。`+DecoderDef+` / `+EncoderDef+` は導出（<<decoder>>・<<encoder>>）が data の形状から充填する。`+require+` と `+let+` は脱糖済みなので AST には現れず、`+if+` とブロックの束縛として現れる（<<require>>・<<block-expression>>）。

[#ast-behavior-info]
=== behavior について追跡する情報

[,text]
----
入力型  出力型（あり得る業務結果の直和）  要求集合  生成権限（constructs）
----

出力型は単一型または直和。`+SpecBehavior+` はこの四つを宣言から取る——要求集合も `+requires+` から取り、`+let+` から推論した集合と突き合わせる（<<requires>>）。`+PipeBehavior+` は入力型・出力型を段の並びから計算し（<<type-routing>>）、要求集合も段の和として推論する（<<composition-with-requirements>>）。

本体はここに無い。behavior は本体を持たず、`+FnDef+` が持つ（<<behavior>>・<<ast-nodes>>）。バックエンドは behavior 側のこの情報と `+FnDef+` の本体を合わせて、AST から直接バイトコードを生成する。

`+事前承認する+` の型:

[,text]
----
事前承認する :
    Behavior<{現在時刻}, (事前承認待ち, 従業員ID), 事前承認済み | 承認権限なし>
----

`+Behavior+` の3つは全部宣言から来る。`+{現在時刻}+` は `+requires+`、入力は引数リスト、出力は `+->+` の右である。`+let+` が3引数なのは注入前だからで、`+Handle.bind(clock)+` が最後の引数を固定すると、宣言した2引数の型になる（<<requires>>）。

== Java側実装例

[,java]
----
// 13.3 の抽象基底 FindMember を extends する（implements ではない）。
public final class JdbcFindMember extends FindMember {
    private final DataSource dataSource;

    public JdbcFindMember(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    @Override
    public FindMember結果 apply(会員ID id) {   // 19.8 の sealed: 会員 | 会員なし | 保存データ不正
        try (var connection = dataSource.getConnection();
             var statement = connection.prepareStatement(
                 "select id, email, display_name from member where id = ?")) {

            // 値の取り出しは encoder 経由（8.5）。会員ID は newtype なので encode は裸の String。
            statement.setString(1, 会員ID.encoder().encode(id));

            try (var resultSet = statement.executeQuery()) {
                if (!resultSet.next()) {
                    return 会員なし();                            // 継承した protected ファクトリ（13.3）
                }
                var raw = Map.<String, Object>of(
                    "id",   resultSet.getString("id"),
                    "メール", resultSet.getString("email"),
                    "表示名", resultSet.getString("display_name"));

                // decode は Raoh の Result<会員>。invariant 違反は 保存データ不正 のケースへ写す
                return switch (会員.decoder().decode(raw, Path.ROOT)) {
                    case Ok<会員> ok   -> ok.value();              // 会員（本線）
                    case Err<会員> err -> 保存データ不正();          // 継承した protected ファクトリ
                };
            }
        } catch (SQLException e) {
            // プラットフォーム障害はケースに畳まず投げる。Souther が素通しし、境界が扱う（503 等。ADR-0029）。
            // SQLException は検査例外なので、境界が捕まえる DataAccessException 系の非検査例外に包む
            // （IllegalStateException や裸の RuntimeException にしない ── ハンドラを素通りし、
            //  インフラ障害をバグに見せる）。
            throw new DataAccessResourceFailureException("member lookup failed", e);
        }
    }
}
----

[#future-work]
== 将来の拡張と現状の制限

言語のコアは実装済みである。次に挙げるのは将来の拡張、あるいはツールと境界コーデックの現状の制限であって、恒久的に持たないもの（<<out-of-scope>>）とは別である。

* 完全な形式文法（BNF）。区切りと優先順位の要点は <<delimiters>> に先出ししている。
* 人間可読な Java ソース生成 backend、増分コンパイル、LSP / IDE プラグイン、ソースマップ。Core 層（<<compiler-pipeline>>）が backend 中立なので、Java ソース backend はそこから書ける。
* JSON Schema 生成、Wasm 出力、JavaScript 出力、非同期の外界依存。
* 手書き decoder / encoder 構文。別キー名・正規化・業務固有の判別子は、境界のカスタムコーデック（Java 側）で扱う。
* 高度な（全域的な）型推論。現状は宣言主義で、behavior は仕様から型を受け、ヘルパーは自分で型を書く（<<fn-declaration>>）。
* 関数を data のフィールドに格納すること。関数は外部表現に写せないので、置けば decoder / encoder の導出（<<decoder>>・<<encoder>>）に例外を作ることになる（<<blocks>>）。
* 入力源を使われ方で絞る生成の最適化（現状は data の形状が対応する源ぶんを生成する。<<codec-generation>>）。
* *境界コーデックの穴（現状の制限）*: `+Date+` / `+DateTime+` フィールドを持つ型の JSON decoder と、入れ子 data を持つ型の jOOQ decoder は、現状は生成しない。素の値 / Map 源の decoder は生成される。

[#non-functional]
== 非機能要件

* コンパイルエラーにソース位置を含める
* 生成する `+.class+` は標準の検証を通り、Java 21+ から素直に利用できる形にする
* コンパイラ各段階を独立してテスト可能にする
* souther-runtime を Raoh に依存させない。decoder/encoder は生成バイトコードが Raoh を直接使い、Raoh 依存は生成コードとアプリ側に置く
* 言語コアをJavaクラスモデルへ依存させない
* 要求集合を静的に追跡する。単純な behavior は宣言と実装を突き合わせ、合成は段から推論する
* 生成コードから不変条件の迂回APIを公開しない
* 仕様DSLの1定義が Souther の1定義へ、翻訳を挟まず対応づく

[#core-semantics]
== 最重要の意味論

Souther は、dataについて次の二つを追跡する。

[,text]
----
どの値がそのdataであるか
どの式がそのdataを生成する権限を持つか
----

behaviorについては次の三つを追跡する。

[,text]
----
入力型   出力型（あり得る業務結果の直和）   必要な依存の集合
----

behaviorは概念的に次の型を持つ。

[,text]
----
Behavior<Requirements, Input, Output>
----

失敗を分ける第4の枠は無い。失敗は Output の一部のケースで、本線から外れるかどうかは合成で決まる。`+>->+` は、出力ケースの振り分け（後段が消費するケースを流し、残りを伝播）と要求集合の合成を同時に行う。

[,text]
----
f : Behavior<R1, A, O_f>
g : Behavior<R2, B, O_g>
f >-> g : Behavior<R1 ∪ R2, A, (O_g) | (O_f から消費分を除いた和)>
----

この規則が、Souther の中心的な合成意味論である。出力は仕様DSLの `+-> 出力 OR ...+` に、要求集合は `+// 依存+`・`+// 副作用+` に対応する。書く側が明示するのは、入力・出力（印の無い直和）・依存（`+requires+`）・生成権限（`+constructs+`）である。仕様DSLが葉に `+// 依存:+` を書き、合成には書かないのと同じ線で、推論に任せるのは合成だけである（<<requires>>・<<composition-with-requirements>>）。

`+Requirements+` は注入で固定される引数であって、動的にハンドルされる効果ではない。だから `+let+` の側では普通の引数として現れ、`+bind+` がそれを部分適用する（<<requires>>）。`+Behavior<R, I, O>+` は、`+(I, R) -> O+` を `+R+` で閉じたものだと読める。要求集合に変数が入らないのはそのためである。それがこの体系全体を単純に保っている。

Souther は、仕様DSLがコメントに預けた値制約・生成経路・外界依存を、invariant・生成経路の閉じ込め・`+requires+` と注入に格上げすることで、仕様モデルを実装モデルへ素直に写す。
