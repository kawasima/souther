[#purpose]
== Purpose

Souther is a JVM-targeted language for transcribing the *specification DSL* of Specification Model-Driven Development (SMDD) into an implementation model with minimal translation.

The spec DSL expresses business rules using only `+data+` (AND / OR / List / `+?+`) and `+behavior+` (`+->+` / `+>->+`). Constraints on the values themselves ("amount is non-negative", "the approver is the applicant's manager"), the confinement of construction paths, and dependency on the outside world are not written there. The spec DSL deliberately leaves these three in comments.

Souther's role is to make those three executable.

[cols="1,1",options="header"]
|===
| What the spec DSL leaves in comments | How Souther makes it executable

| Value constraints (`+// 0以上+`, `+// 承認者は上長+`) | `+invariant+`

| The invariant "this value has been validated" | Confinement of construction paths (`+decoder+` / `+constructs+`)

| `+// 依存: 製品カタログ+` / `+// 副作用: メール送信+` | A `+behavior+` with no `+let+` at a leaf; `+requires+` (an injected Java implementation) at the call site

|===

The structure of the spec DSL (distinction, requiredness, multiplicity, state transition, totality) maps directly onto Souther's types. Value constraints and outside-world dependencies are promoted to types and injection. Souther implements, on the implementation-model side, the SMDD division of labor: structure is guarded by types, value constraints by tests and the types of implementations, and outside-world dependencies by injection.

[#mapping-principle]
=== Mapping principle

Each construct of the spec DSL maps to Souther as follows.

[cols="1,1",options="header"]
|===
| Spec DSL | Souther

| `+data 金額 = 整数 // 0以上+` | `+data 金額 = Int  invariant value >= 0+` (newtype; <<newtype>>)

| `+data X = A OR B+` (A, B are named data) | `+data X = A \| B+` (sum; cases are existing named data)

| `+data 立替 = 単位型+` | `+data 立替+` (data with no fields)

| `+data X = A AND b+` (A is a record, b is an added item) | `+data X = { ...A, b: B }+` (field composition)

| `+List<T>+` / `+T?+` | `+List<T>+` / `+T?+`

| `+behavior f = In -> Out+` (Out has multiple success states) | `+behavior f : (...) -> Out1 \| Out2+` (bare sum)

| `+behavior f = In -> Out OR 失敗+` | `+behavior f : (...) -> Out \| 失敗+` (a failure is just data; departure from the mainline is decided by composition)

| `+f >-> g+` | `+f >-> g+`

| `+// 依存+` / `+// 副作用+` | A `+behavior+` with no `+let+`

| `+// 事前条件+` | `+require ... else+`

|===

[#boundary]
=== The boundary with the outside world

Souther does not handle the outside world directly. The boundary is expressed with primitive values, collections, and Decoders / Encoders.

[,text]
----
outside world
  ↓
primitive values / collections
  ↓ Decoder
internal data
  ↓ behavior
internal data
  ↓ Encoder
primitive values / collections
  ↓
outside world
----

The JVM is used as the distribution platform that makes the generated types and behaviors available to Java programs.

[#design-principles]
== Design principles

[#closed-construction]
=== Construction paths of data are closed

A value of a given `+data T+` MAY be constructed only through one of the following.

* the Decoder of `+T+`
* the implementation of a behavior declaring `+constructs T+` (a Souther `+let+`, or an injected Java implementation)
* compiler-generated code

A behavior that merely writes `+T+` as its return type MUST NOT be able to construct `+T+`. The authority to construct belongs to the behavior, not to its implementation. Therefore `+constructs+` is written on the behavior (<<constructs>>); the `+let+` and the Java implementation receive the authority from the same declaration.

A helper `+let+` with no corresponding behavior (<<fn-declaration>>) MAY also construct data. In that case the constructed data MUST appear in the `+constructs+` of the behavior that calls that `+let+`.

[#invariant-on-construct]
=== Invariants are checked on every construction

Every invariant declared on a `+data+` MUST be checked on every construction path. A value that violates an invariant MUST NOT be returned.

The destination of a violation differs inside and outside the boundary: in a Decoder it becomes a failure of Raoh's `+Result+`; inside the domain it aborts (<<violation-destination>>).

[#separate-representation]
=== External representation and internal data are separated

A Decoder constructs internal data from an external representation; an Encoder converts internal data to an external representation. An external representation MUST NOT be cast directly to internal data.

[#behavior-composition-unit]
=== behavior is the unit of composition

Internal computation, outside-world dependency, validation, and conversion are all expressed as behaviors and composed sequentially with `+>->+`. A behavior is a top-level input/output relation with no owning class.

As a type, a behavior states *only the specification*. A simple behavior has no implementation; the implementation is written as a `+let+` (<<fn>>). If none is written, it is injected from Java (<<injected-behavior>>). The exception is `+>->+` composition, which is itself the implementation.

Only behaviors that appear in the spec DSL are behaviors; implementation helpers remain `+let+`. This keeps the list of behaviors aligned with the spec DSL.

[#no-impl-for-outside]
=== No implementation is written for outside-world dependencies

Outside-world dependencies — DB, HTTP, files, clock, ID generation — MUST NOT be implemented in the language. These correspond to the `+// 依存:+` and `+// 副作用:+` notes in the spec DSL. Only the type is declared in the language.

[,text]
----
behavior findMember : (id: MemberId) -> Member | FindMemberError    // no let
----

The implementation is injected from Java. Writing no implementation (`+let+`) is what "provided from outside" means (<<injected-behavior>>). A caller writes that name in its `+requires+` (<<requires>>). The distinction between dependency (only reads the outside) and side effect (changes the outside) is documentation of intent; it does not affect the rules of value composition.

[#unmarked-sum]
=== Business results are expressed as bare sums

Every business result is returned as a sum of domain data. A *business* "failure" is not a first-class concept in the language. A behavior's output is a bare sum. The case that a downstream stage does not consume in a `+>->+` composition is what departs from the mainline (<<composition>>). *Unexpected* failures such as out-of-memory or a network drop MUST NOT be listed in the type.

An invariant violation MUST NOT appear here. It is not a business result but a model bug, and its destination is an abort (<<algebraic-types>>).

Multiple independent validation errors are accumulated inside the Decoder (<<case-propagation>>).

[#asymmetric-interop]
=== Java interop is asymmetric

Java MAY use the generated artifacts. Souther MUST NOT call arbitrary Java APIs directly.

[,text]
----
Java   -> Souther artifacts     allowed
Souther -> arbitrary Java API    forbidden (touch the outside only via a behavior with no implementation)
----

[#out-of-scope]
== Out of scope

Souther does not have the following. These are design decisions, not features to be added later.

* arbitrary Java API calls
* mutable state
* exceptions (see below)
* null
* threads
* asynchronous processing
* inheritance
* structural intersection types (`+A & B+`; see <<union-intersection>>)
* reflection
* macros
* type classes
* higher kinds
* dependent types
* proof by SMT
* termination proofs
* a custom VM
* a package manager
* a REPL

Because the implementation of outside-world dependencies is delegated to Java, Souther itself has no DB, HTTP, or file APIs.

*On exceptions.* What is forbidden is *using exceptions to control business flow*. Business results are returned as cases of a sum (<<unmarked-sum>>); neither `+throw+` nor `+catch+` can be written.

The reporting path for model bugs, however, is not blocked. An invariant violation *aborts* (<<algebraic-types>>). An abort is not a Souther expression and has no catching syntax, so it cannot be used to control business flow.

A Java Binding at the boundary (the Java implementation of a behavior with no implementation; <<injected-behavior>>) is different. Against a *platform failure* (DB down, network drop, host not responding — anything that is not a domain outcome) it MAY throw an exception; Souther does not fold it into a case but passes it through to the boundary. The language (`+.sou+`) has no exceptions, but the Java side that touches the outside world may throw. The distinction is *domain outcome vs. platform failure*: a business-meaningful result such as "not found" or "insufficient stock" is a case; "infrastructure unavailable" or "not responding" is an exception (<<java-impl-rules>>, ADR-0029).

[#modules]
== Source files and modules

The extension is `+.sou+`. Each file defines one module.

[,text]
----
module example.businesstrip
----

A single file compiled on its own MAY omit the `+module+` header and start directly with a `+data+`
/ `+behavior+` / `+let+`; the module is then named after the file (the file-name stem), or `+Main+`
when compiled from a source string with no file name (ADR-0043). This follows F#/Elm, which make the
header optional only for a single file. A module compiled together with others and reached by
`+import+` MUST write the header — an unnamed module cannot be the target of an `+import+`, so
omitting it there is a compile error.

Keywords and the standard library are in English, but user-defined identifiers (types, fields, behaviors) MAY use Unicode. Because the spec DSL is written in Japanese, business vocabulary can be used as identifiers directly. Identifiers SHALL be *NFC-normalized* before binding and comparison; in Japanese a precomposed character and a combining sequence can look identical, so without normalization names that look the same would diverge in binding. Source is UTF-8.

[,text]
----
data 出張申請 = 申請準備中 | 提出済み | 事前承認待ち | 事前承認済み
----

Public elements are listed with `+exposing+`.

[,text]
----
module example.businesstrip exposing (
    出張申請,
    出張申請を提出する,
    事前承認する
)
----

`+exposing+` exposes at type granularity. A public data's `+decoder()+` / `+encoder()+` is always public API (<<jvm-codec>>), so member-level narrowing such as `+出張申請.decoder+` is not written — there is nothing to narrow.

When an exposed behavior is a `+>->+` composition, its output SHALL be written in `+exposing+` in the form `+name : cases+` (<<declared-composition-output>>). A composition's output is inferred from its stages (<<composition-with-requirements>>), but at the public boundary a signature is required rather than inference, so that the contract used by another module or by Java does not change silently when an upstream stage changes. Omitting it is a compile error (E1605).

[,text]
----
module example.order exposing (
    注文を確定する : 確定注文 | 在庫不足
)
----

`+exposing+` is the module's interface description: implementations are inferred, only the boundary is stated explicitly. This is the same position as OCaml's `+.mli+`, F#'s `+.fsi+`, and TypeScript's `+.d.ts+`.

[#imports]
Imports SHALL be listed explicitly. Wildcard imports are forbidden. A cyclic import is a compile error.

[,text]
----
import example.employee (
    従業員,
    従業員ID
)
----

The standard library is imported in the same form: the qualifier `+List+` / `+String+` / `+Map+` / `+Bool+` / `+Int+` / `+Decimal+` as the name, and the functions used listed. Once imported, they MAY be written unqualified (<<stdlib>>).

[,text]
----
import List ( map, filter, fold )
import String ( trim, split )
----

This corresponds to Elm's `+import List exposing (map)+`. Qualified access (`+List.map+`) always works without importing, so `+import+` only elides the qualifier. Importing the same bare name from two modules is ambiguous — writing both `+import String ( length )+` and `+import List ( length )+` in one module leaves `+length+` undetermined. In that case, import one and qualify the other call (`+String.length(...)+`). As with the wildcard prohibition, no name is delivered implicitly.

[#reserved-words]
== Reserved words

[,text]
----
module   import    exposing
data     invariant
behavior constructs requires
match    with      as
let      if        then     else
require
true     false
----

Operators and symbols: `+=+` `+|+` `+>->+` `+|>+` `+?+` `+->+` `+.+` `+...+` `+[+` `+]+`

Comparison, logic, arithmetic: `+==+` `+/=+` `+<+` `+<=+` `+>+` `+>=+` `+&&+` `+||+` `{plus}` `+-+` `+*+` `+/+` (`+/+` is division; `+/=+` is inequality; negation is not a prefix operator but `+not+` in `+souther.bool+`; <<stdlib-bool>>).

Lists: `+[+` `+]+` (literal and guard comprehension), `{plus}{plus}` (concatenation, of two lists or two strings; <<stdlib-list>>).

`+>->+` is associative (<<type-routing>>). `+|>+` is the value pipe, threading a value through functions — `+e |> f(a)+` is `+f(a, e)+` (<<pipe>>).

`+->+` is the arrow. It joins types — in a behavior declaration (<<behavior>>) and in the function type of a `+let+` argument (<<fn-declaration>>) — and it introduces an anonymous body — in a `+match+` case (<<match>>) and a lambda (<<blocks>>). As in F# and Elm, there is one kind of arrow. `+=+` binds a name to a right-hand side (whether the right-hand side is a type or a term is decided by the rules of <<behavior>>).

`+decoder+` / `+encoder+` / `+discriminate+` are not reserved words: the mapping to external representation is not syntax but is derived from a data's shape (<<decoder>>, <<encoder>>). `+error+` / `+success+` / `+failure+` are not reserved words either: a *business* failure is neither a special type nor syntax but ordinary domain data, and whether it departs from the mainline is decided by composition (<<unmarked-sum>>, <<behavior>>, <<composition>>).

There is no type name that carries an invariant violation. A violation inside the domain aborts, so no value needs to be carried (<<algebraic-types>>, <<violation-destination>>).

[#delimiters]
=== Delimiters and disambiguation

The full formal grammar (BNF) is deferred (<<future-work>>). The delimiters and precedence needed for reading and writing are fixed here.

* Both `+data+` declaration fields and record-literal `+型名 { ... }+` fields are `+,+`-separated (<<product-data>>, <<record-literal>>). A declaration gives a type, so `+名前: 型+`; a literal binds a value, so `+名前 = 値+` (`+:+` = has this type, `+=+` = is defined as this; ADR-0026). A `+...+` spread is an element that composes types but appears in the same comma sequence as fields (<<field-spread>>, ADR-0012, ADR-0030). An `+invariant+` is a clause placed after the type body (`+{ ... }+` or `+= Y+`) and is not part of the field list (<<invariant>>, ADR-0031).
* Argument lists and the enumerations in `+constructs+` / `+requires+` are `+,+`-separated (<<behavior-io>>, <<constructs>>, <<requires>>).
* Every `+,+`-separated list — declaration and record-literal fields, parameter and argument lists, tuple elements, list literals, and the `+constructs+` / `+requires+` name lists — MAY carry a single trailing `+,+` before its closing delimiter. It is optional and has no meaning; it accommodates writers coming from Go or TypeScript. The sample code in this spec keeps a leading-comma layout instead (the `+,+` opens each continuation line, as in Elm) and writes no trailing comma.
* `+|+` is the case separator of a sum and binds more loosely than `+->+`. `+(x: T) -> A | B+` reads as `+(x: T) -> (A | B)+` (the whole output is a sum). An anonymous `+|+` MUST NOT be written on input (<<union-intersection>>).
* `+>->+` is the stage separator of a pipeline: not a binary operator but a flat n-ary sequence. It is associative (<<type-routing>>). Only a behavior name MAY appear in a stage position; parentheses such as `+(f >-> g)+` MUST NOT be written.
* `+|>+` is the value pipe, binding most loosely and left-associatively. `+a |> f |> g+` reads as `+g(f(a))+`. The left side enters the last argument of the right-hand call and is expanded to a call at parse time (<<pipe>>). This is distinct from the `+>->+` that composes behaviors; it threads the value of an expression.
* `+?+` is a type postfix (`+従業員ID?+`) and desugars to `+Option<従業員ID>+` (<<optional>>).
* `+.+` is field reference; `+...+` is spread (<<record-literal>>).
* The type position admits: a named type, a stdlib generic type (such as `+List<T>+`), an anonymous output sum (`+A | B+`), and a postfix `+?+`. A function type `+(A) -> B+` is admitted only in a `+let+` argument position (<<fn-declaration>>).

[#external-representation]
== External representation

The external representation handled by Decoders and Encoders is not Souther's own types but the neutral value model of the boundary library Raoh (a tree of scalar / List / Map). Souther owns the domain's types, invariants, and behaviors; the external representation and its parsing and generation are delegated to the peripheral library Raoh (<<separate-representation>>, <<decoder>>).

From a data's shape, Souther derives the decode side per input source (bare values / Map = raoh-core, JSON `+JsonNode+` = raoh-json, jOOQ `+Record+` = raoh-jooq) and the encode side to a neutral Map (raoh-core). The decode skeleton (fields, keys, discriminator) is common across sources; only the terminal field access varies per source. JSON output serializes the Map at the boundary (`+objectMapper.valueToTree+`, etc.; <<codec-generation>>).

An external representation may contain null, but internal data forbids null. null exists only at the boundary (Raoh's input) and MUST NOT pass through a Decoder into the domain.

[#builtin-types]
== Built-in types

[#primitives]
=== Primitive types

[,text]
----
Bool
Int          // signed 64-bit integer
Decimal      // java.math.BigDecimal
String
Date         // date. ISO 8601. java.time.LocalDate
DateTime     // date-time. ISO 8601. java.time.LocalDateTime
----

`+Date+` / `+DateTime+` correspond to the spec DSL's `+年月日+` / `+年月日時分秒+`. Value construction goes through a Decoder (from an ISO 8601 string). The current time is not obtained in the language; it is declared as a behavior with no implementation (<<injected-behavior>>).

==== Numeric literals

An integer literal is written as `+500+` and is `+Int+`. A `+Decimal+` literal takes an `+m+` suffix — `+500m+`, `+1.5m+`, `+0.5m+` (following F#'s `+m+` suffix). A fractional literal without `+m+`, such as `+1.5+`, is an error. Souther has no binary floating-point type and defaults fractions to `+Decimal+` (exact decimal), so `+Int+` and `+Decimal+` MUST be made explicit at the literal (ADR-0033).

==== Equality

`+==+` and `+/=+` are *value equality*, not reference identity. Souther data are immutable values, so whether two are the same is decided by whether their fields are the same.

[,text]
----
require 承認者ID == 申請.申請者.上長ID      // two 従業員ID built separately are equal if their values are equal
----

* primitives compare values
* data compare *fields recursively*, including fields taken in by a `+...+` spread
* sums are equal when they are the same case and their contents are equal
* `+List<T>+` compares elements in order; `+Map<String,T>+` compares keys and values
* `+T?+` is equal when both are `+None+`, or both are `+Some+` with equal contents

Only *values of the same type* may be compared. Comparing different types is a compile error.

Ordering (`+<+` `+<=+` `+>+` `+>=+`) is defined for `+Int+` (numeric), `+String+` (lexicographic), `+Decimal+` (by value, ignoring scale), and `+Date+` / `+DateTime+` (chronological). A sum has no ordering, and neither does a product data. A single-value newtype (`+data X = Y+`; <<newtype>>) over one of the ordered types *is* ordered — by the value it wraps. Elm gives ordering only to `+Int+` / `+Float+` / `+Char+` / `+String+`, and dates are opaque types reduced to `+Int+` (milliseconds) before comparison, because it stands on JavaScript. Souther stands on the JVM, where `+BigDecimal+` / `+LocalDate+` / `+LocalDateTime+` are all `+Comparable+`, so it gives ordering to these as well.

[#newtype-comparison]
===== Comparing a newtype

A single-value newtype is compared by the value it wraps, for both equality and ordering, so `+.value+` is not written — `+value+` is the newtype's implicit inner name, not something a caller spells out. The nominal boundary is kept:

* Two of the *same* newtype compare their wrapped values: `+金額 <= 金額+`, `+金額 == 金額+`.
* A *bare literal* of the wrapped type takes that newtype from the other operand: `+金額 <= 100+`, `+金額 == 0+` (the `+100+` is a `+金額+`). This mirrors how `+[]+` takes its element type from context (<<algebraic-types>>).
* Two *different* newtypes over the same base do not compare: `+金額 <= 数量+` is a compile error, even though both wrap `+Int+`.
* A non-literal value of the wrapped type is not taken implicitly: `+金額 <= n+` (where `+n+` is an `+Int+`) is a compile error — write `+金額 <= 金額(n)+`.

The wrapped type must itself be ordered for `+<+` / `+<=+` / `+>+` / `+>=+`; equality works over any wrapped type. Arithmetic on a newtype is not defined.

`+Decimal+` ignores scale. `+1.0+` and `+1.00+` are equal.

[,text]
----
data 金額 = Decimal
// read as 1.00 from a DB column, 1.0 from JSON. Treated as the same amount.
----

Scale is not lost from the value. An Encoder emits the scale it read, so a round trip (<<round-trip>>) is preserved. Scale is dropped *only from identity comparison*. If the number of rounding digits is a domain concern, write it as an invariant or a separate field, not as `+Decimal+` scale.

The `+hashCode+` of a data containing `+Decimal+` is computed after dropping scale (`+stripTrailingZeros+`). If equality ignores scale, the hash MUST ignore it too, or using the value as a `+Map+` key breaks. Java sees the same value equality: a generated data class implements `+equals+` / `+hashCode+` from all fields and can be used as a `+Map+` key directly (<<jvm-product>>). This is distinct from Java's `+==+` (reference) and `+BigDecimal.equals+` (scale-sensitive).

[#collections]
=== Collection types

[,text]
----
List<T>
Map<K, V>
Set<T>
----

Collections are immutable. A `+List+` is ordered and may repeat; a `+Set+` is unordered with no duplicate elements, compared by value equality (<<primitives>>, ADR-0009); a `+Map+` is keyed by `+K+`, which is `+String+` or a String-backed newtype (`+data 商品ID = String+`), so a key carries its domain type (`+Map<商品ID, 在庫>+`; ADR-0040). A newtype-keyed map crosses the boundary as a JSON object with bare-string keys: each key is decoded into the key newtype (its invariant enforced, a bad key failing at that key's path) and encoded back bare. A `+Set+`'s external representation is a JSON array, deduplicated on decode (ADR-0039).

Collections are *covariant* in their element. A case `+A+` may be used as its sum `+S+` (<<sum-data>>), so `+List<A>+` may be passed where `+List<S>+` is expected, `+Set<A>+` where `+Set<S>+` is expected, `+Map<String, A>+` where `+Map<String, S>+` is expected, and `+A?+` where `+S?+` is expected. This is sound because collections are immutable and there is no room to write a sibling case later. Covariance of mutable arrays (Java) is unsound because writes are a hole; that is the difference. This is the same position as Scala's immutable `+List+` and Kotlin's read-only `+List+`. Covariance applies only in the case→sum direction; sibling cases (`+A+` and `+B+`) are unrelated.

[#algebraic-types]
=== Built-in algebraic and auxiliary types

[,text]
----
Option<T>    = Some(T) | None
Never                      // a type with no value: an impossible case (empty sum)
Unit                       // a type with exactly one value
----

Souther has no `+Result+` / `+Either+` type. A behavior's output is a bare domain sum with no two-slot type separating success and failure (<<behavior>>, <<composition>>).

`+Never+` and `+Unit+` are not type names writable at the surface; they are reading aids. `+Never+` denotes an impossible case: a single-case output (`+-> A+`) can be read as having an empty failure row. `+Unit+` corresponds to the spec DSL's `+単位型+`, but it is never provided as a way to write a field; it is always expressed as a "data with no fields" (<<sum-data>>).

The only polymorphic types are the stdlib types listed here (`+Option+` / `+List+` / `+Map+`). There are no user-defined generics. Neither `+data+` nor `+behavior+` takes a type parameter.

==== An invariant violation aborts

When the construction of a data with an invariant violates it inside the domain (<<invariant-mvp>>), the computation *aborts*. It returns no value and no case.

[,text]
----
behavior 値引きする : (d: 見積) -> 値引き済み
    constructs 値引き済み
----

Writing a computation that breaks the invariant of `+値引き済み+` does not change the output type; breaking it aborts. An abort is not a Souther expression and has no catching syntax, so it cannot control business flow (<<out-of-scope>>). At the boundary (<<java-example>>), Java catches it and maps it to 500, distinct from a business failure (400).

If the author wants to return the violation, that is a business judgment and MUST be written explicitly with `+require ... else+` (<<require>>). An invariant is a declaration that "a value passing here is always so", not a branch designed to be broken.

A decode-side violation does not abort; Raoh's `+Result+` failure carries it (<<decoder-error>>). The same invariant is handled differently inside and outside the boundary.

==== How Option appears

`+Option<T>+` *cannot be written as a type* either. It appears only through two paths:

* the desugaring of a field `+T?+` (<<optional>>)
* the return value of a standard library function (`+List.get+` / `+Map.get+`; <<stdlib-list>>, <<stdlib-map>>)

It MUST NOT be written in a behavior's output. To state "might not be found" as a business result, that is a domain sum (<<unmarked-sum>>).

[,text]
----
behavior findMember : (id: 会員ID) -> 会員 | 会員なし     // write this
----

`+Some(T)+` is an exception to the rules of <<sum-data>> (a case is a reference to a named data and cannot carry a payload). Because `+Option+` is built in, users cannot imitate this form. Its contents are taken out with `+match+`; the `+v+` in `+| Some v+` is positionally bound to the wrapped value, not to `+Some+` (<<match>>).

[#optional]
=== Optional type

Optionality is expressed with `+?+`.

[,text]
----
data 出張完了 =
    { 事前承認者: 従業員ID?
    }
----

This desugars to:

[,text]
----
事前承認者: Option<従業員ID>
----

To put a value in a `+?+` field, write a `+従業員ID+` value directly and it is wrapped in `+Some+`; write `+None+` for empty. Users never write `+Some(...)+` (as in <<algebraic-types>>, this form is built-in only).

[,text]
----
出張完了 { 事前承認者 = 承認者ID }    // wrapped in Some(承認者ID)
出張完了 { 事前承認者 = None }        // empty
----

Wrapping in `+Some+` and `+None+` are not data construction, so `+constructs+` is not required (<<constructs>>). Extraction is done with `+match+`'s `+| Some v+` / `+| None+` (<<match>>).

`+?+` MUST NOT be used for anything other than field optionality.

The Decoder's external type is (Raoh's Decoder):

[,text]
----
Decoder<T> = external representation -> Result<T>    // failure carries Raoh's Issues
----

[#data]
== data

`+data+` defines typed data of the internal world. All fields are immutable.

A declaration takes the form `+data 名前 = 右辺+`. If the right-hand side is `+{ ... }+` it is a product; `+A | B+` a sum; a single type name `+Y+` a newtype wrapping `+Y+` (<<newtype>>); if the right-hand side is omitted, a unit. This mirrors the spec DSL's `+data 名前 = ...+`.

[#product-data]
=== Product data (AND)

The spec DSL's `+AND+` maps to a product. A field name is the business role; the type may be specified separately.

[,text]
----
// spec DSL: data 従業員 = 従業員ID AND 役職 AND 上長ID
data 従業員 =
    { id: 従業員ID
    , 役職: 役職
    , 上長ID: 従業員ID // role name (上長ID) differs from type (従業員ID)
    }
----

Role and type may be written separately. In the spec DSL the same concept has a different name per layer: `+承認者ID+` (operation input) becomes `+事前承認者ID+` (stored state) becomes `+pre_approver_id+` (DB column). A field such as `+事前承認者: 従業員ID+` writes the role in the field name and the concept in the type.

[#self-referential-data]
==== Self-referential data

Data MAY reference itself (directly or via another data). Structures such as an org chart or a category tree appear routinely in business.

[,text]
----
data 社員 = { 上司: 社員?, 氏名: String }        // the manager is again a 社員 (the chain ends at None)
data 部門 = { 子部門: List<部門>, 名称: String }   // a tree (a leaf has an empty 子部門 list)
----

The derived codecs follow this self-reference. A nested `+{"氏名":"a","上司":{"氏名":"b"}}+` decodes to a `+社員+` whose `+上司+` is again a `+社員+`. Traversing a self-reference *by computation* (climbing the manager chain to the root, counting tree depth) is written with a recursive helper `+let+` (<<fn-declaration>>, ADR-0038). `+fold+` folds only a fixed-structure `+List+`, so a self-reference of unbounded depth can be traversed only with helper recursion.

A self-reference requires a *base case*. A cycle that returns to itself through required fields only (not `+?+`, not `+List+`/`+Map+`) needs its own value to construct itself, which continues indefinitely — it is not constructible. Therefore *a required self-referential cycle with no base case is a compile error*.

* `+上司: 社員+` (no `+?+`) is a cycle with no base case. It is a compile error, not a runtime stack overflow.
* `+上司: 社員?+` (optional) and `+子部門: List<社員>+` (list) are base cases (the cycle is cut by `+None+` / the empty list), so they are allowed.
* A sum is an OR composition, so if a cycle passes through a sum it may bottom out in another case. The check stops at a sum and produces no false positive.

[#field-spread]
=== Field composition (spread)

In the spec DSL, a state sharing common items is written as `+data 提出済み = 出張申請共通項目 AND 提出日時+`, meaning "has all fields of `+出張申請共通項目+`, plus `+提出日時+`". Souther expresses this with the `+...+` spread.

[,text]
----
data 出張申請共通項目 =
    { 申請者: 従業員
    , 予定費用: 金額
    , 出張先: String
    , 出張目的: String
    , 出張開始日: Date
    , 出張終了日: Date
    }

// spec DSL: data 提出済み = 出張申請共通項目 AND 提出日時
data 提出済み =
    { ...出張申請共通項目 // take in the common item's fields flatly
    , 提出日時: DateTime
    }
----

The `+...+` spread is defined as follows.

* The taken-in fields are laid out flatly. A `+提出済み+` value is referenced directly as `+x.申請者+` (not nested as `+x.共通.申請者+`).
* The taken-in data's invariants are inherited too. The invariants of `+出張申請共通項目+` are checked when `+提出済み+` is constructed. The spread carries not only fields but invariants.
* A field-name collision is a compile error.
* A spread is not inheritance. `+提出済み+` is not a subtype of `+出張申請共通項目+`; the two are not assignment-compatible. They only share fields.
* A spread is an element of the same comma sequence as fields. It appears inside `+{+` separated by `+,+`, and multiple may be written. By convention it is placed first. The notation follows ReScript's record-type spread (`+{ ...a, ...b, c: C }+`) and uses the same `+...+` as the value-level record-literal spread (<<record-literal>>).

To hold the common items as a single value rather than flattened, nest it as a normal field (`+共通: 出張申請共通項目+`). The spec DSL's `+AND+` is a spread; explicit nesting is a normal field.

[#sum-data]
=== Sum data (OR)

The spec DSL's `+OR+` maps to a sum. *A case MAY reference an already-defined named data.* As in the business-trip model, where each state has its own fields, invariants, and transition behaviors and is bound together by `+OR+`, this structure is represented directly.

[,text]
----
// spec DSL: data 出張申請 = 申請準備中 OR 提出済み OR 事前承認待ち OR 事前承認済み
data 出張申請 =
    申請準備中
  | 提出済み
  | 事前承認待ち
  | 事前承認済み
----

Nested sums MAY be written directly.

[,text]
----
data 費用負担区分 = 自社負担 | 先方負担
data 自社負担     = 立替 | 仮払い | 会社カード
----

A case is always a reference to a separately declared named data. An inline record MUST NOT be written in a case position. The spec DSL is the same: a case is always declared separately (for `+data 役職 = 管理職 OR 一般社員+`, `+data 管理職 = レベル+` / `+data 一般社員 = 単位型+`).

*A case value is a value of its sum.* As in functional languages, a value of a case type (e.g. `+提出済み+`) may be used transparently wherever its sum type (`+出張申請+`) is expected — field assignment, argument, or return value. Only the upward direction (case→sum) is allowed; the downward direction (sum→specific case) requires `+match+`. A nested sum (`+data 自社負担 = 立替 | 仮払い+` being a case of `+費用負担区分+`) is decided the same way, folding to the leaf case.

[#unit-data]
=== Unit data

A data with no fields is declared with no body. It corresponds to the spec DSL's `+= 単位型+`.

[,text]
----
// spec DSL: data 立替 = 単位型
data 立替
data 仮払い
data 会社カード
data 先方負担
data 一般社員
----

Unit data appear frequently as sum cases (such as `+一般社員+` in `+data 役職 = 管理職 | 一般社員+`).

A unit data value is constructed by *writing the name directly* (as a zero-argument constructor becomes a value in functional languages). An empty record such as `+T {}+` is not used.

[,text]
----
// construct a unit in an implementation (let). Requires the behavior's constructs (§2.1, §12.3)
[先方費用負担 | 費用負担が先方]     // 先方費用負担 is a unit, constructed by name alone
----

This is a guard comprehension (<<stdlib-list>>): it yields one `+先方費用負担+` if the condition holds, an empty list otherwise.

A bare identifier in an expression resolves as: a variable if a bound local exists; otherwise a built-in value (such as `+None+`; <<algebraic-types>>); otherwise the construction of a unit data of the same name; otherwise undefined. Souther does not distinguish identifiers syntactically by case (so that Japanese business vocabulary can be used as identifiers), so this resolution is done in the symbol table.

Because a local shadowing a built-in value is always an error, using a *built-in value name* (`+None+`, rounding mode `+HALF_UP+`, etc.; <<stdlib-decimal>>) as a binding name — `+let+`, a `+match+` binding, or a lambda argument — is a compile error. This prevents a shadowed built-in from silently becoming unreachable. A unit data name is domain vocabulary and may be used as a binding, in which case the local takes precedence and constructing a unit data of that name is not writable in the same scope.

[#field-visibility]
=== Field visibility

*Reading* a field is possible wherever that data's type is visible. Within the same module, any data of that module may be read. A data of another module may have its fields read if it is exposed with `+exposing+` — being importable means it is exposed. An exposed data has all fields readable (there is no per-field exposure). A behavior may read the fields of its input data even when they come from another module.

What is constrained is *not reading but construction*. Reading a field and constructing a new value of that data are separate authorities, so allowing cross-boundary reads leaves the guarantee that no unvalidated value can be built (<<asymmetric-interop>>) intact. There is no need to define a getter-only behavior just to pass a value out (<<behavior-composition-unit>>).

On the JVM, a public read accessor is generated for the fields of an exposed data (<<jvm-product>>). Because module = package, this is required to read across the boundary. Java can read too, but that is the same exposure as the encoder already emitting all fields as JSON; what is guarded is construction. The constructor stays non-public, and to build a value Java must go through the invariant-checking decoder (<<jvm-construction-privacy>>).

[#union-intersection]
=== Union types and intersection types

Souther has union types. They appear in three forms.

* named sum `+data X = A | B+` (sealed; subject to exhaustiveness checking)
* anonymous union in a behavior output `+-> 提出済み | 事前承認待ち+` (<<unmarked-output>>)
* anonymous union with failure cases in a behavior output `+-> Out | E1 | E2+` (<<type-routing>>)

An anonymous union appears *only in a behavior's output*. An anonymous union MUST NOT be written in an input (parameter) type. To pass a sum, make a named data (<<unmarked-output>>); naming it makes it sealed, with exhaustiveness checking and a JVM type (<<jvm-sum>>). Where the spec DSL writes `+(A OR B)+` in an input, the sum is named as a `+data+` and then referenced. A JVM type is generated for an output anonymous union (<<jvm-anonymous-union>>).

Structural intersection types (`+A & B+`, a type operator for "a value that is both A and B") are *not* provided. The spec DSL's `+AND+` means "has all of A's fields, plus more", not "is both A and B", and is fully expressed by the `+...+` spread of <<field-spread>> (take fields in flatly, inherit invariants, not a subtype).

[#newtype]
=== newtype (`+data X = Y+`)

When the right-hand side is a single type name `+Y+`, `+X+` is a newtype wrapping `+Y+`. `+Y+` may be a primitive (`+String+` / `+Int+` / `+Date+`, etc.), a named data, or a sum. `+X+` is a distinct type from `+Y+` (nominally distinguished) with the same (bare) external representation as `+Y+`. The spec DSL's `+data X = Y+` (right-hand side a single name) is written directly.

[,text]
----
// spec DSL: data 従業員ID = 文字列 // not empty
data 従業員ID = String
    invariant String.length(value) > 0

// spec DSL: data 権限不足 = 役職
data 権限不足 = 役職

// spec DSL: data 費用明細 = 費目 (費目 is a sum)
data 費用明細 = 費目
----

`+data X = Y+` is equivalent to having a single field `+value: Y+`. Construction applies the value to the type name, `+X(v)+` (`+金額(500)+`, `+会員ID("m-01")+`), equal to `+X { value = v }+`. Reference is `+x.value+`; an invariant refers to the inside as `+value+`. The difference from `+data X = { value: Y }+` (with braces) is *the external representation only*: `+data X = Y+` reads and writes `+Y+`'s representation bare (`+従業員ID+` is `+"e-01"+`), while `+data X = { value: Y }+` becomes an Object `+{"value": "e-01"}+`.

The constructor `+X(v)+` is written with parentheses like function application. Juxtaposition `+X v+` is not adopted, because Souther's application is uniformly `+f(x)+` and because, absent a capitalization rule on identifiers, `+金額 -5+` would be ambiguous between construction and subtraction (ADR-0032). An invariant violation aborts like any other data construction (<<violation-destination>>). A construction whose argument is a compile-time constant statically verified to satisfy the invariant cannot abort, so it may appear anywhere in the type body; a construction where a constant breaks the invariant, such as `+金額(-5)+`, is a compile error. A construction passing a runtime value is limited, like any other invariant-bearing construction, to a behavior's result position, and aborts there on violation.

Whether it is a newtype is *decided by this syntax*, not inferred from the single-field shape. Adding a field to `+data 金額 = { value: Int }+` does not silently change its representation (`+{...}+` is always an Object, even with a single field). For the bare representation, write `+data 金額 = Int+`.

Because there is no `+|+`, there is no room to read `+data 権限不足 = 役職+` as a one-case sum. A sum always has `+|+` (<<sum-data>>). When `+Y+` is a sum, `+X+`'s external representation is `+Y+`'s discriminated representation, and if it becomes a case of a nested sum it is folded to the leaf and dispatched (<<sum-discrimination>>).

[#invariant]
== invariant

[#invariant-declaration]
=== Declaration

A value constraint written as a comment in the spec DSL is made executable as an `+invariant+`.

[,text]
----
// spec DSL: data 金額 = 整数 // 0以上、円単位
data 金額 =
    { value: Int
    }
    invariant value >= 0

data DateRange =
    { start: Date
    , end: Date
    }
    invariant start <= end
----

[#invariant-mvp]
=== Checking on construction

Invariants are checked at runtime. The compiler inserts an invariant check on every path that constructs a data. Invariants taken in by a `+...+` spread are checked on the same path.

[#invariant-expressions]
=== Expressions allowed / forbidden in an invariant

Allowed: comparison, logical operations, arithmetic, field reference, built-in functions, blocks, and a `+let+` satisfying the three conditions below.

[,text]
----
let 正の数 (v: Int) = v >= 0

data 金額 = Int  invariant 正の数(value)
data カゴ = { items: List<明細> } invariant List.all(items, x -> 正の数(x.数量))
----

What may be called is a `+let+`, not a behavior (<<fn>>). An invariant runs on a data's construction path, where touching the outside world is unacceptable. A behavior may touch the outside world — if no implementation is written, a Java implementation is injected — so it MUST NOT be called from an invariant. To name a business rule, use a `+let+`.

A `+let+` may be called from an invariant when it satisfies all of the following, all statically decidable.

* the requirement set is empty (the corresponding behavior writes no `+requires+`; <<requires>>)
* it constructs no data (the construction set of any helper `+let+` it calls is also empty; <<blocks>>)
* it is not recursive

The first is decided by reading the declaration. `+let 事前承認する+` does not touch the outside world directly and so is pure in the sense of <<purity>>, but `+behavior 事前承認する+` writes `+requires 現在時刻+`, so its requirement set is non-empty and it MUST NOT be called from an invariant.

A helper `+let+` always satisfies the first and second. To touch the outside world it can only receive a function as an argument (<<fn-declaration>>), and an invariant has no way to pass one. The third (not recursive) can be broken only by a recursive helper (<<fn-declaration>>, ADR-0038), so an invariant calling a recursive helper is a compile error.

Forbidden: Decoder, Encoder, data construction, recursion, non-deterministic processing. A behavior call is not forbidden separately — an invariant calls a `+let+`, not a behavior (above), and if a `+let+` goes through a behavior that touches the outside world, its requirement set is that much non-empty and is rejected by condition 1. Only pure (empty-requirement-set) computation enters an invariant.

[#violation-destination]
=== A violation's destination differs inside and outside the boundary

The same invariant is handled differently depending on where it breaks.

* *Decoder (boundary)*: the violation enters a failure (`+Issue+`) of Raoh's `+Result+` (<<decoder-error>>). A value that came from outside was invalid — an *expected result*. Which field broke it can be stated via `+path+` and `+code+`.
* *behavior (inside the domain)*: it *aborts* (<<algebraic-types>>). A computation among validated values broke a rule — a *model bug*.

[,text]
----
data 値引き済み = { 額: Int } invariant 額 >= 0

behavior 値引きする : (d: 見積) -> 値引き済み          // no violation case needed
    constructs 値引き済み
----

*A behavior's output needs no slot for the violation.* Constructing an invariant-bearing data does not change the output type.

If the business decides "return a different result when the amount is negative", that is not an invariant violation but a business judgment and MUST be written explicitly with `+require ... else+` (<<require>>). These are different: an invariant declares "a value passing here is always so", while `+require+` is a business branch of "if not so, return this". The former breaking is a bug; the latter not holding is an expected result.

[#decoder]
== Decoder

[#decoder-role]
=== Role

A Decoder is the boundary that constructs internal data from external input. It implements the spec DSL's parse-don't-validate (validate raw data at the boundary and construct a domain type only on success). Souther derives a Raoh Decoder from a data's shape. The derived Decoder reads an external representation (<<external-representation>>) and, on success, constructs T; failure is represented by Raoh's `+Result+` (the failure side carrying `+Issues+`).

[,text]
----
会員ID.decoder() : Decoder<会員ID>     // external representation -> Result<会員ID> (failure = Raoh Issues)
----

A decode failure is not a Souther domain type. Malformed input is not a business result but a boundary concern (<<unmarked-sum>>), so it is not listed among domain sum cases but placed on the failure side of Raoh's `+Result+`. Decode is not a behavior but the edge of the boundary and is not a `+>->+` stage (<<sequential-composition>>). Composition in the domain world is done only with behaviors and `+>->+`.

A Decoder does not appear in Souther's syntax. A default Decoder is derived from a data's field names and types. A data definition writes only `+data+` / `+invariant+` / `+behavior+`. This keeps the mapping to external representation (key names, normalization, discriminators) out of domain definitions. When the default derivation is insufficient, a custom Decoder is written on the boundary side (Java / Raoh) (<<custom-codec>>).

[#decoder-derivation]
=== Derivation rules

The default Decoder is derived from a data's shape by the following conventions.

* A JSON key matches the field name. Each field of an Object is read from the key of the same name.
* A newtype (`+data X = Y+`; <<newtype>>) is read directly from `+Y+`'s representation, not an Object (a bare scalar if `+Y+` is primitive, the derived Decoder of `+Y+` if a named data or sum). `+{ ... }+` is read as an Object even with a single field.
* A nested named data is read recursively with that data's derived Decoder.
* `+List<T>+` is read element by element with `+T+`'s derived Decoder.
* `+T?+` (Option; <<optional>>) is `+None+` if the key is absent or the value is `+null+`, and `+Some+`-wrapped after reading `+T+` if a value is present.
* Invariants are checked on construction; a violation becomes a failure (`+Issue+`) of Raoh's `+Result+`. Because decode always succeeds or fails, the destination of a violation always exists (<<violation-destination>>).
* Errors of independent fields are accumulated (Applicative; <<case-propagation>>). Accumulation is handled by Raoh combinators.

[,text]
----
// a newtype reads from a bare integer (data X = Y form; §8.7)
data 金額 = Int
    invariant value >= 0
// derived Decoder: Int(n) -> 金額 { value: n } (with invariant check)

// a multi-field data reads from an Object, field name = key
data 従業員 =
    { id: 従業員ID
    , 役職: 役職
    , 上長ID: 従業員ID
    }
// derived Decoder: reads { "id": ..., "役職": ..., "上長ID": ... } with each field's derived Decoder
----

The invariant check on construction and spread (`+...+`) follow the same semantics as a record literal (<<field-spread>>, <<record-literal>>).

[#sum-discrimination]
=== Discrimination of sum data

The derived Decoder of a sum data reads the discriminator field `+"type"+` and dispatches to the case's derived Decoder. The tag is the case name.

[,text]
----
data 出張申請 =
    申請準備中 | 提出済み | 事前承認待ち | 事前承認済み
// derived Decoder: reads "type" and dispatches "申請準備中" -> 申請準備中, "提出済み" -> 提出済み, ...
----

A nested sum is folded to the leaf case and dispatched.

[,text]
----
data 自社負担     = 立替 | 仮払い | 会社カード
data 費用負担区分 = 自社負担 | 先方負担
// derived Decoder: "type" is one of "立替" / "仮払い" / "会社カード" / "先方負担".
// The tag "自社負担" does not appear in the external representation.
----

Folding is done because there is only one `+"type"+`. Tagging a direct case would break the round trip (<<round-trip>>). Once `+費用負担区分+`'s encode writes `+{ "type": "自社負担" }+`, which leaf it was is lost, and the subsequent decode of `+自社負担+` would try to read the same key and reject it as not its own tag. Folding at the leaf lets the same external representation be read by any layer's decoder. This is the same as case→sum assignment folding to the leaf (<<sum-data>>). Exhaustiveness in `+match+` is checked per layer (<<match>>), which is a separate matter from external representation.

A case's contents are decided by the case's shape. A product case lays fields flatly in the same Object as `+"type"+` (`+{"type":"提出済み", "申請者":…, …}+`). A unit case writes only `+"type"+` (`+{"type":"一般社員"}+`). A newtype case (`+data X = Y+`; <<newtype>>) cannot place `+"type"+` on a bare scalar, so it puts the contents under the `+"value"+` key.

[,text]
----
data 役職 = 管理職 | 一般社員
data 管理職 = レベル        // newtype
data 一般社員              // unit
// derived: 管理職 -> {"type":"管理職","value":3}, 一般社員 -> {"type":"一般社員"}
----

This maps directly onto Raoh's `+discriminate("type", …)+`. The discriminator selects `+"管理職"+` and passes the same Map to the case's decoder, which reads `+"value"+` and assembles `+管理職+`. This is the same framework as a product case reading fields flatly, only the read location becomes `+"value"+`. A standalone newtype (not a sum case) stays bare (`+3+`); the `+"value"+` wrapping appears only inside a sum.

Folding to the leaf concerns nested *sums*; a newtype case stops there. A newtype is not a sum, so making `+data 権限不足 = 役職+` (役職 is a sum) a case of `+事前承認理由+` yields `+{"type":"権限不足","value":{"type":"管理職","value":3}}+`, where the inner 役職 is discriminated independently under `+"value"+`.

When the discriminator key or tag is business-specific (reading `+"status"+` and dispatching to `+"draft"+` / `+"submitted"+`, or reading a persistence `+representation_version+` for versioning), that is a matter of the external representation and cannot be derived from a data's shape. It is done with a custom Decoder on the boundary side (<<custom-codec>>).

[#custom-codec]
=== Custom boundary codec

When the default derivation is insufficient (a different key name, normalization, default filling, a business-specific discriminator, persistence version branching), write a Decoder on the Java side that assembles the external representation and calls the data's (invariant-checking) construction. The `+JdbcFindMember+` calling `+会員.decoder()+` in <<java-example>> is this form. External-representation conversion stays confined to the boundary and is not brought into domain definitions.

A business-specific sum tag does not mean rebuilding the case decoders. Souther generates a public per-case decoder (`+提出済み.decoder()+`) and a sum decoder that is only a `+discriminate("type", …)+` over them (<<sum-discrimination>>). The boundary keeps the per-case decoders and rewrites just that one layer with its own key and tags — reading `+"status"+` and dispatching `+"draft"+` / `+"submitted"+` is `+discriminate("status", variant("draft", 申請準備中.decoder()), variant("submitted", 提出済み.decoder()))+`. Field assembly and invariant checks are reused; only the tag table is written. The encoder side is less symmetric: the derived sum encoder bundles the tag with each case's encoding, so a custom sum encoder re-tags each case's Map itself.

[#decoder-error]
=== Decoder errors

A decode failure is carried by the failure side of Raoh's `+Result+`. An error is a Raoh `+Issue+` (with `+path+` / `+code+` / `+message+`), and failures of independent fields are accumulated (<<case-propagation>>). Souther does not own the *boundary's* failure vocabulary. No domain type such as `+復号失敗+` or `+DecodeError+` is introduced; the representation of boundary failure is delegated to the peripheral library Raoh.

This differs from a violation inside the domain (<<algebraic-types>>). A break at the boundary means a value from outside was invalid — an expected result, and which field broke it should be statable (`+path+` and `+code+`), which Raoh holds. A break inside the domain means a computation among validated values broke a rule — a model bug — so it aborts (<<violation-destination>>). The boundary's vocabulary is not brought into the domain, and the domain's vocabulary is not taken out to the boundary.

At the boundary (HTTP, etc.), the `+Result+` failure is received and mapped to an appropriate response (e.g. 400). The `+MemberController+` branching on the decode `+Result+` in <<java-example>> is this form.

[#codec-generation]
=== Generation method and Raoh

The derived Decoder, without going through an independent Decoder IR, is generated *directly as bytecode* by the ClassFile backend as a Raoh Decoder (`+net.unit8.raoh.decode.Decoder+` and combinators). The generated decoder reads from the neutral tree of the external representation (<<external-representation>>; Raoh's `+Object+` / `+Map+` input) and returns `+Result<T>+`.

[,text]
----
Source -> AST -> (fill decoder/encoder into the AST by derivation) -> type check -> ClassFile backend -> .class (emits Raoh Decoders)
----

Souther's runtime (souther-runtime) does not depend on Raoh. Only the generated bytecode and the apps using it depend on Raoh. Primitive parsing, error accumulation (Applicative), and discriminator dispatch (<<sum-discrimination>>) are handled by Raoh combinators; Souther only maps a data's shape (keys, types, discriminator, case tags) onto Raoh decoder construction.

Decoders are generated per input source. Bare values / Map use raoh-core combinators, JSON `+JsonNode+` uses raoh-json, and jOOQ `+Record+` uses raoh-jooq. The skeleton (keys, types, discriminator, case tags) is common across sources; only terminal field access is swapped per source. For each exposed data, decoders are generated for the sources its shape supports. Bare values / Map and JSON are always generated. JSON temporal fields (`+Date+` / `+DateTime+`) are read as ISO 8601 strings (`+string().date()+` / `+string().dateTime()+`). jOOQ `+Record+` decoders are generated only for a flat shape with no nesting, List, or Map (scalar and newtype columns only) — a row is flat, so a nested object does not fit in one column. Unit data ignore input, so they support all sources.

[#encoder]
== Encoder

[#encoder-role]
=== Role

An Encoder is a total function converting internal data to an external representation (<<external-representation>>). Souther derives a Raoh Encoder from a data's shape.

[,text]
----
出張申請.encoder() : Encoder<出張申請>     // 出張申請 -> external representation (Raoh's neutral tree)
----

Like a Decoder, an Encoder does not appear in Souther's syntax. A default Encoder is derived from a data's field names and types (<<decoder-role>>). Encode is also an edge of the boundary and is not a `+>->+` stage.

[#encoder-derivation]
=== Derivation rules

The default Encoder is derived by rules symmetric to the Decoder.

* A newtype (`+data X = Y+`; <<newtype>>) writes directly to `+Y+`'s representation (a bare scalar if `+Y+` is primitive). `+{ ... }+` writes to an Object even with a single field.
* A multi-field data writes to an Object. The key is the field name; the value is converted by each field's derived Encoder. Fields taken in by a `+...+` spread are laid flatly too.
* A nested named data is converted by that data's derived Encoder.
* `+List<T>+` is converted element by element with `+T+`'s derived Encoder.
* `+T?+` (Option; <<optional>>): if `+Some(v)+`, write `+v+` at the key; if `+None+`, omit the key. `+None+` is not written and is read back as `+missing key → None+`, so the round-trip law (<<round-trip>>) holds.
* A sum data tags the discriminator field `+"type"+` with the case name and converts with the case's derived Encoder. A product case lays fields flatly, a unit case writes only the tag, a newtype case puts contents in `+"value"+` (<<sum-discrimination>>). A nested sum writes the leaf case's tag.

[,text]
----
data 金額 = Int  invariant value >= 0
// derived Encoder: 金額 -> Int(self.value) (bare integer)

data 提出済み =
    { ...出張申請共通項目
    , 提出日時: DateTime
    }
// derived Encoder: 提出済み -> { "申請者": ..., "予定費用": ..., ..., "提出日時": ... }
----

To match key names or representation to an external spec, write a custom Encoder on the boundary side (<<custom-codec>>).

[#round-trip]
=== Round-trip law

The following is recommended (not required).

[,text]
----
decode(encode(value)) == value
----

`+encode(decode(raw)) == raw+` is not required, because a Decoder may normalize, fill defaults, or drop surplus items.

[#behavior]
== behavior

A behavior defines the *specification* of a behavior over internal data. It is a top-level input/output relation with no owner.

A declaration takes the form `+behavior 名前 = 右辺+`. There are two right-hand sides, and what is written and what is left undetermined are inverted between them.

If *an argument list and output* (<<behavior-io>>) are written, what is written is the type. The implementation is undetermined: writing a `+let+` makes it the implementation (<<fn>>); writing none injects it from Java (<<injected-behavior>>). A behavior of this form has no body. What the spec DSL notes as `+// 依存:+` comes here.

If a `+>->+` composition (<<composition>>) is written, what is written is the assembly step of chaining stages in this order, and that is itself the implementation. So no `+let+` is needed and no injection occurs. The type is undetermined and computed by the rules of <<type-routing>>.

Both are the same form as the spec DSL's `+behavior+`. The DSL writes both with `+=+`.

[,text]
----
// spec DSL
behavior 事前承認する  = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認権限なし   // right side is a type
behavior 注文を確定する = 注文を検証する >-> 価格を記載する >-> 在庫を引き当てる      // right side is a composition
----

That a type or a term may come to the right of `+=+` is because the DSL writes it that way; it is not a distinction Souther introduced. The same two lines map to Souther as:

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    constructs 事前承認済み, 承認権限なし          // a type. Implementation is a let or Java injection

behavior 注文を確定する = 注文を検証する >-> 価格を記載する >-> 在庫を引き当てる
                                                 // a composition. It is itself the implementation

// spec DSL: // 依存: 時刻取得
behavior 現在時刻 : () -> DateTime                // write the type, no let → injected from Java
----

Specification and implementation are separated not because `+data+` already does so, but *so that one line of the spec DSL remains as is* (<<non-functional>>). Just as `+invariant+` rides on `+data+`, `+constructs+` rides on `+behavior+`. Both are the part Souther makes executable of what the spec DSL left in comments (<<mapping-principle>>).

[#behavior-io]
=== Input and output

The spec DSL's input `+AND+` maps to an argument list. A parameter is always written `+名前: 型+`.

[,text]
----
// spec DSL: behavior 事前承認要否を判定する = 予定費用 AND 申請者 AND 費用負担区分 -> 事前承認理由リスト
behavior 事前承認要否を判定する : (
    予定費用:    金額,
    申請者:      従業員,
    費用負担区分: 費用負担区分
) -> List<事前承認理由>
----

A parameter name is for the reader and does not affect type identity. The implementation (`+let+`) gives its own names and corresponds by position. A type MUST NOT be written without a name: if `+(String)+` were writable, whether it meant "the type String" or "the name String" would be unreadable.

Zero arguments is written `+()+` (as in `+現在時刻+`). "No argument" and "takes one `+Unit+`" are different; `+Unit+` is not exposed at the surface (<<algebraic-types>>).

[#unmarked-output]
=== Output is a bare sum

The spec DSL's output `+OR+` maps directly to a bare sum. No success/failure distinction and no marker such as `+error+` is attached. A behavior is a function from input to one of the possible business results.

[,text]
----
// spec DSL: behavior 出張申請を提出する = 申請準備中 -> 提出済み OR 事前承認待ち
behavior 出張申請を提出する : (申請: 申請準備中) -> 提出済み | 事前承認待ち
    constructs 提出済み, 事前承認待ち

// spec DSL: behavior 事前承認する = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認権限なし
// 承認権限なし is just data. Not an error, not a Result.
behavior 事前承認する : (
    申請:    事前承認待ち,
    承認者ID: 従業員ID
) -> 事前承認済み | 承認権限なし
    constructs 事前承認済み, 承認権限なし
----

Which case "departs as a failure" is not decided by this behavior alone. When composed with `+>->+`, the case the immediately following stage does not consume departs the mainline (<<composition>>). If a downstream stage takes `+承認権限なし+` as input, that case is processed; if not, it departs and is carried to the output. The same value departs in one composition and stays on the mainline in another.

An input may be a sum (spec DSL `+(提出済み OR 事前承認済み) AND ...+`). Here *an anonymous sum MUST NOT be written directly in a parameter type*. First declare the sum as a named data and write that name in the parameter. The implementation opens the cases with `+match+`.

[,text]
----
// name (提出済み OR 事前承認済み)
data 完了可能申請 = 提出済み | 事前承認済み

// spec DSL: behavior 出張を完了する : (提出済み OR 事前承認済み) AND 実費用 AND 出張報告 -> 出張完了 OR 不正な実費用 OR 出張報告なし
behavior 出張を完了する : (
    申請:   完了可能申請,
    実費用:  金額,
    出張報告: String
) -> 出張完了 | 不正な実費用 | 出張報告なし
    constructs 出張完了, 不正な実費用, 出張報告なし
----

An anonymous union appears only in a behavior's *output* (<<union-intersection>>). An input type is always a single named type (including a named sum). Naming it lets the reader fix the input candidates by looking once at `+data 完了可能申請 = 提出済み | 事前承認済み+`, and on the JVM the input has a name so a Java implementation receives it as a single type (an anonymous union has no common supertype; <<asymmetric-interop>>, <<jvm-anonymous-union>>).

[#constructs]
=== Data construction authority (constructs)

A behavior that constructs data MAY write the constructed type in `+constructs+`. This is an internal construction authority; unlike a dependency (`+requires+`), it is invisible to callers. So for a behavior with a `+let+` implementation it is *optional* — omit it and it is inferred from the body. If written, it MUST *exactly match* the body (too few = E1002, too many = E1006). An injected behavior has no body, and its declaration is the input to factory generation, so it MUST always write it (<<java-base-class>>).

[,text]
----
behavior applyChange : (input: ChangeEmailRequest) -> Member
    constructs Member
----

To construct multiple data, list them: `+constructs Member, AuditRecord+`.

The following count as data construction: a record literal (`+型名 { ... }+`), a sum data constructor, a unit data construction, and constructing a collection containing new data as elements. Returning an existing value unchanged does not count as construction.

The built-in `+Some+` / `+None+` (<<algebraic-types>>) do not count as data construction. Option is not domain data but an auxiliary type and has no invariant, so <<invariant-on-construct>> has nothing to guard. So `+constructs+` is not required to put a value in a `+?+` field (<<optional>>) — and since Option cannot be written as a type, there is no way to declare `+constructs Some+` anyway (<<algebraic-types>>).

*If you write `+constructs+`, you may not omit anything the implementation constructs.* The departure value of `+require ... else+` (<<require>>) is the same: `+else 承認権限なし+` constructs one unit data, so if declaring, include it in `+constructs 承認権限なし+` (omitting it is E1002). Being a failure case is no exception — what <<closed-construction>> closes is construction, regardless of success or failure.

==== What constructs is for

`+constructs+` is not for guarding invariants — that is handled by `+__construct+` and the Decoder, both of which always check on construction (<<invariant-mvp>>). What `+constructs+` handles is *reading the model*: from the declaration, whether a behavior *creates* a new value or merely *passes through* an existing one. This is not visible from the output type.

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    constructs 事前承認済み, 承認権限なし        // creates new states

behavior 上限を検査する : (a: 金額) -> 金額 | 上限超過
    constructs 上限超過                        // 金額 returns the input as is; not created
----

Both have `+金額+` / `+事前承認済み+` in the output, but one creates and the other passes through. This distinction does not appear in the output type. Writing it leaves it in the declaration and makes a check against reading and mis-taking. Omit it and it is inferred — write it (opt-in) if you want the reading and the check.

Creating a type that does not appear in the output also shows here. The `+constructs 高額出張, 権限不足, 先方費用負担+` of `+事前承認要否を判定する = (...) -> List<事前承認理由>+` is information nowhere in the output type.

A helper `+let+` (with no corresponding behavior) may also construct data, but a helper `+let+` is not a specification statement and does not declare its own `+constructs+`. Its *construction set is inferred transitively* (folding in any helper `+let+` it calls). If the calling behavior writes `+constructs+`, it MUST include that set (E1002 if it does not); if omitted, that set is folded into the inference. A block is the same, constructing under the enclosing behavior's authority (<<blocks>>).

A dependency (`+requires+`) is different and MUST *always* be written (<<requires>>). `+requires+` is an outward contract to the caller and injector; leaving it to inference would let the contract change silently. `+constructs+` is an internal authority invisible to callers, so they diverge here. Thus `+requires+` is a declaration and `+constructs+` (for a behavior with a `+let+` implementation) is optional. A `+>->+` composition infers both output and requirements (<<composition-with-requirements>>).

[#record-literal]
=== Record literals and spread

Construction is done with a record literal that writes the constructed type name.

[,text]
----
事前承認済み {
    申請者 =      申請.申請者,
    予定費用 =     申請.予定費用,
    提出日時 =     申請.提出日時,
    事前承認日時 =  現在時刻(),
    事前承認者 =   承認者ID
    // ... list the rest of the common items too
}
----

To fill same-named fields together, use a spread `+...src+`. It copies `+src+`'s same-named fields and states the rest explicitly. Surplus fields of `+src+` are ignored; a field missing from `+型名+` is a compile error.

[,text]
----
// copy the common items and 提出日時 of 申請 (事前承認待ち), and add two
事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
----

To replace only some fields while keeping the same type, also write the type name and use a spread. If `+x+` has type `+Member+`, write `+Member { ...x, 住所 = 新住所 }+`. No dedicated update syntax is provided.

When a field value is a binding of the same name as the field, the value may be omitted. `+Member { 住所 }+` is shorthand for `+Member { 住所 = 住所 }+`, filling from the same-named binding in scope.

A record literal and a spread both require construction authority (`+constructs+`). Because the type name appears at the construction point, the authority check and field check are done on the spot.

[#blocks]
=== Blocks

To apply processing to each element of a list, pass a block as an argument. A block is written `+引数 -> 式+`.

[,text]
----
// spec DSL: data 未検証注文  = ... AND List<未検証注文明細>
//           data 検証済み注文 = ... AND List<検証済み注文明細>
// validate and repack each detail. A block is needed here.
behavior 明細を検証する : (xs: List<未検証注文明細>) -> List<検証済み注文明細>

let 明細を検証する (xs) = List.map(x -> 1件を検証する(x), xs)
----

A block is an anonymous `+let+`. It is the same as a `+let+` (<<fn>>), only unnamed. A named one may be passed by name.

[,text]
----
let 妥当か (r: 事前承認理由) = ...

List.all(理由リスト, r -> 妥当か(r))   // wrap in an anonymous block
List.all(理由リスト, 妥当か)           // pass by name directly. Same thing
----

A block and a `+let+` may be passed as arguments, and a `+let+` may take one as an argument (<<requirement-propagation>>). *Only placing it in a data field is not allowed.* A function cannot be mapped to an external representation, so placing one would create an exception to decoder / encoder derivation (<<decoder>>, <<encoder>>). This is a representation constraint unrelated to side effects. The spec DSL itself holds a rule it wants to swap not as a function but as data (`+data フィールド定義 = ... AND バリデーションルール?+`).

A block may also be bound to a name with `+let+` and applied (`+let f = (x) -> ...+`, then `+f(a)+`). Like a named `+let+` (<<fn-declaration>>), as long as it does not escape — applied only, never leaving as a value — a block is expanded into its call/apply site and forms no closure. When it escapes — *returning* a function, or choosing at runtime (`+let f = if ... then (x) -> ... else ...+`) — a closure capturing free variables is generated. It is a synthesized class implementing the runtime `+Fn+`, holding the captured variables in fields. Where expandable it is expanded; only where necessary is it a closure. The *cannot place in a data field* constraint above is unchanged by closures: it is a representation constraint, separate from whether a closure exists.

A block and a helper `+let+` are the same in constructing data. A block constructs under the enclosing behavior's `+constructs+`, and a hoisted helper `+let+` constructs under the `+constructs+` of the behavior that calls it (the construction set is inferred transitively and reconciled; <<constructs>>). Either way the construction authority is on the behavior and does not disappear with naming. So hoisting a constructing `+[高額出張 { 基準金額 = 基準 } | …]+` (<<stdlib-list>>) into `+let 高額出張を作る (基準) = 高額出張 { 基準金額 = 基準 }+` passes if the calling behavior declares `+constructs 高額出張+`.

[#requires]
=== Dependencies (requires)

A behavior that touches the outside world writes what it requires in `+requires+`.

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    requires 現在時刻
    constructs 事前承認済み, 承認権限なし
----

What is written is the name of a behavior with no implementation (<<injected-behavior>>). Write it if the `+let+` uses that name. Even when used indirectly by passing it to a helper `+let+`, the one passing is you, so you write it.

Only a simple behavior with a `+let+` may have `+requires+`. A behavior with no implementation does not write it: it is the leaf of the dependency, and its name being listed is itself the statement of dependency. A `+>->+` composition does not write it either: it is decided by the union over stages, readable from the stages (<<composition-with-requirements>>).

==== Correspondence to the spec DSL

The DSL attaches `+// 依存:+` / `+// 副作用:+` to a *leaf*.

[,text]
----
// responsibility 2: save the business-trip application to the database
// 依存: 出張申請データベース
// 副作用: save to the database
behavior 出張申請を保存する = 未検証出張申請 AND 事前承認要否 -> 保存済み出張申請
----

In Souther this becomes a behavior with no implementation, so no `+requires+` is needed. The same as the DSL attaching no note to a composition: it is readable from the stages.

The DSL holds only these two because it places side effects at the ends of a workflow. The form "use the outside world once in the middle of a pure computation" does not appear in a specification model. Souther allows it, so a position with no source in the DSL — a behavior with a `+let+` — arises. `+requires+` lets the DSL's note be written on a non-leaf behavior too. The positions increase, but the content is the same "what it depends on" as the DSL; no lines are added.

==== requires becomes a let argument

What is written in `+requires+` appears as a `+let+` argument, after the business inputs, in declaration order.

[,text]
----
let 事前承認する (申請, 承認者ID, 現在時刻) = {
    require 承認者ID == 申請.申請者.上長ID
        else 承認権限なし

    事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
}
----

The behavior has 2 arguments and the `+let+` has 3 because *the behavior's type is the type after partial application*.

[,text]
----
let:      (申請, 承認者ID, 現在時刻) -> 事前承認済み | 承認権限なし
bind:     fix 現在時刻                                  f_a(x) = f(x, a)
behavior: (申請, 承認者ID)          -> 事前承認済み | 承認権限なし
----

`+Handle.bind(new JdbcClock())+` (<<jvm-behavior>>) is this partial application. What remains after binding is the behavior's declared type, and Java calls `+apply(申請, 承認者ID)+` with 2 arguments. `+requires+` does not appear in the caller's arguments.

This merely surfaces the "requirements are constructor arguments" of the end of <<blocks>>. In the `+let+`, `+現在時刻+` is just an argument, not an effect or a special type. So the `+let+` side needs no place to write `+requires+`.

A helper `+let+` (with no corresponding behavior) fits the same form. To touch the outside world, it receives a function as an argument — a helper writes its own type (<<fn-declaration>>), so it writes it there.

[,text]
----
let 記録する (x: Int, 現在時刻: () -> DateTime) = ...       // helper. Receives it as an argument
----

The `+現在時刻+` appearing inside a `+let+` is an argument in either case. There is no path to call a behavior with no implementation directly as a top-level name. So a helper `+let+` having no place to write `+requires+` is not a restriction. The passer writes it in its own `+requires+` and passes it as an argument.

==== Checking

The set of behaviors with no implementation that a `+let+` uses and the set written in `+requires+` MUST match. Too few is E1602, too many is E1603 (<<e1602>>, <<e1603>>). An over-declaration erring on the safe side is not admitted either.

[#tuple]
== Tuples

`+(a, b, ...)+` is a tuple bundling two or more values into one (ADR-0036). It is an expression-level value, used only in the middle of a computation — a helper `+let+` returning multiple values and destructuring them — and MUST NOT be written in a data field or a behavior input/output. Like a function (<<blocks>>), it does not cross the boundary, so no codec is derived. Domain vocabulary stays as named data.

A tuple *type* `+(A, B, ...)+` may be written in a helper or standard-library signature — a parameter, a return type, or a type argument such as `+List<(String, 'a)>+` — since a signature is not a codec boundary (ADR-0037). It is still rejected in the two boundary positions above: a data field and a behavior's input/output, where a codec would have to be derived. A one-element `+(T)+` is not a tuple type.

[,text]
----
let 統計 (xs: List<Int>) = (List.sum(xs), List.length(xs))   // return multiple values

let 平均 (xs) = {
    let (合計, 個数) = 統計(xs)      // positional destructuring bind
    ...
}
----

`+let (x, y) = t+` destructures a tuple positionally and binds. Names correspond to elements left to right. A one-element `+(e)+` is just a parenthesized expression and is not a tuple. A tuple may be held as is in a `+let+`, passed to a helper, or be an element of `+List<(a, b)>+` (a first-class value). Extraction is by destructuring only; there are no index functions such as `+fst+` / `+snd+`.

A lambda argument `+(a, b) -> ...+` is multiple arguments (<<blocks>>), distinct from a tuple. To destructure a tuple inside a lambda, receive one argument and open it in the body with `+let (a, b) = p+`.

[#fn]
== let — implementation

A `+let+` is the implementation of a behavior. A behavior is the specification (<<behavior>>), and a `+let+` is what satisfies it.

[#fn-declaration]
=== Declaration and binding

A `+let+` is written `+let 名前 (引数) = 本体+`. If a behavior of the same name exists, it implements it.

[,text]
----
behavior 事前承認する : (申請: 事前承認待ち, 承認者ID: 従業員ID) -> 事前承認済み | 承認権限なし
    requires 現在時刻
    constructs 事前承認済み, 承認権限なし

let 事前承認する (申請, 承認者ID, 現在時刻) = {
    require 承認者ID == 申請.申請者.上長ID
        else 承認権限なし

    事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
}
----

The type comes from the behavior. The `+let+` side only names its arguments, corresponding by position. The behavior-side names are for the reader; what binds is the `+let+`-side names. `+constructs+` is on the behavior, so it is not written on the `+let+` (<<constructs>>).

Arguments are the behavior's argument list followed by `+requires+` in declaration order. A `+let+` is the pre-injection function and a behavior's type is the post-injection type, so a `+let+` has as many more arguments as there are `+requires+` (<<requires>>). If `+requires+` is empty, both have the same arguments.

A `+let+` with no corresponding behavior is a *helper*. Not being a domain behavior, it does not pollute the behavior list, cannot be placed in a `+>->+` stage, and does not appear in `+exposing+`. It states its own type, though a non-recursive helper may leave a value parameter's type to be inferred from its call sites (below).

[,text]
----
let 正の数 (v: Int) = v >= 0
let 適用する (xs: List<Int>, p: (Int) -> Bool) = List.all(p, xs)
----

The return type is inferred. However, a *recursive helper* MUST declare its return type — `+let 深さ (s: 社員): Int = ...+` — because the result type cannot be inferred through the cycle, and only with a declaration can a self-call be type-checked. The declaration is required only for a recursive helper. A non-recursive helper is inferred as before, and a `+let+` with a same-named behavior cannot write a return type (the type comes from the behavior).

The body is an expression. `+{ ... }+` is a block expression (<<block-expression>>), written to place a `+let+` or `+require+`. `+let 正の数 (v: Int) = v >= 0+` and `+let 正の数 (v: Int) = { v >= 0 }+` are the same.

`+=+` introduces a named definition. The bodies of an anonymous block (<<blocks>>) and a `+match+` case are introduced by `+->+` — the same symbol as the type arrow (as in F# and Elm). Binding a lambda to a named definition (`+let f = (x) -> ...+`) is not adopted.

==== Rules

* A `+let+` with a same-named behavior does not write argument types; the type comes from the behavior. A count mismatch is a compile error.
* A helper (a `+let+` with no same-named behavior) states its own argument types, with one inference: a *non-recursive* helper is inline-expanded at each call site (spec 12.5), so a *value* parameter with no annotation takes its type from how it is called, and that type MUST agree at every call site — a helper is monomorphic (ADR-0050). A *function-typed* parameter MUST be written; a function cannot be inferred from a bare name at the call site. A parameter that no call site can type — an uncalled helper, or an argument bound outside the visible scope — MUST be annotated. A *recursive* helper is lowered to a method rather than inlined (below), so it writes all of its argument types.
* A helper *may recurse* (self- and mutual recursion; ADR-0038). A non-recursive helper is inline-expanded at its call site; a recursive helper is not inlined but lowered to a method — self- and mutual calls become plain method calls. Traversal of self-referential data that `+fold+` cannot write (climbing the manager chain to the root, counting tree depth) is written this way.
* A recursive helper MUST *declare its return type* (above). Without a declaration it is a compile error.
* A recursive helper MUST be *pure*: it MUST NOT call an injected behavior (`+requires+`). An effect touching the outside world is placed on the behavior that calls the helper.
* Termination is not checked. Non-terminating recursion overflows the JVM stack, and `+StackOverflowError+` propagates through as a platform failure (<<java-impl-rules>>, ADR-0029). There is no depth limit or fuel.
* A local lambda (`+let f = (x) -> ...+`) cannot recurse. Recursion is a property of a named module helper.
* Two `+let+`s MUST NOT be written for one behavior.
* A `+let+` may implement a behavior only when both are in the *same module*. A behavior of another module may be imported and called, but its implementation MUST NOT be given by a `+let+` in another module. This is the line that keeps the name-binding rule from conflicting with separate compilation.
* A helper `+let+` may construct data, but its construction set is inferred transitively and MUST be included in the calling behavior's `+constructs+` (<<constructs>>, <<blocks>>). It cannot be placed in a `+>->+` stage or written in `+exposing+`, because it does not appear in the spec DSL.
* A function type (`+(A) -> B+`) may be written only in a `+let+` argument position. It MUST NOT be written in a data field (<<blocks>>) or a behavior output. A behavior output is a sum of business results, and a function is not a business result.
* A `+let+` may return a function (`+let adder (n: Int) = (x) -> x + n+`). An escaping function — a return value or a runtime choice — becomes a closure capturing free variables (<<blocks>>). If only passed or applied on the spot, it is expandable and no closure is needed. The return type is inferred, so no type is written even when returning a function. A lambda argument type is not written either — it is inferred from how it is applied (a local rule, separate from the advanced type inference of <<future-work>>).

[#injected-behavior]
=== A behavior with no implementation — outside-world dependency

*If a behavior with only a type has no implementation*, its implementation is injected from Java. It corresponds to the spec DSL's `+// 依存:+` and `+// 副作用:+`.

An implementation is either a same-named `+let+` (<<fn-declaration>>) or a right-hand `+>->+` composition (<<composition>>). A `+>->+` composition behavior has no `+let+` but does have an implementation, so it is not subject to injection. What is injected is only one that wrote a type and no `+let+`.

[,text]
----
// spec DSL: // 依存: 時刻取得
behavior 現在時刻 : () -> DateTime

// spec DSL: // 依存: 会員リポジトリ
behavior findMember : (id: 会員ID) -> 会員 | 会員なし | 保存データ不正
    constructs 会員なし, 保存データ不正
----

Output cases are domain outcomes only. `+会員なし+` (not found) and `+保存データ不正+` (a stored value breaks a domain invariant) are business-meaningful results, so they are cases. A *platform failure* such as DB down is not placed here — the Java Binding throws an exception and Souther passes it through (<<java-impl-rules>>, ADR-0029).

No keyword is needed. Having no implementation is what "provided from outside" means. The spec DSL also has no word for `+required+`; whether something is a dependency is only noted in a comment. This form keeps the DSL line as is.

`+constructs+` is required on a behavior with no implementation too (<<constructs>>). The declaration reads the same as when implemented in Souther. `+findMember+` creates two failures and does not create `+会員+`; `+会員+` is obtained by passing a value read from outside through the decoder. The factory of <<java-base-class>> distributes from here.

Typical uses: DB query, HTTP call, file I/O, clock, ID generation, message sending. The concrete APIs are not brought into the language core. The distinction between dependency (only reads) and side effect (changes) is documentation of intent and does not affect value composition.

Forgetting to write an implementation is not an error; that behavior simply becomes subject to injection. But a caller MUST write that name in `+requires+` (<<requires>>), and doing so surfaces it in a `+bind+` argument (<<jvm-behavior>>). If there is no caller, it is not caught within the `+.sou+`.

Which behaviors get a `+let+` and which are injected is not decided mechanically from the DSL. The spec DSL's `+// 依存:+` is a note, not an obligation, so its absence does not mean an internal implementation. The 1:1 of <<non-functional>> is a declaration correspondence: one DSL behavior line maps to one Souther behavior declaration. It does not decide the implementation form (`+let+` / injection / `+>->+` composition). The model author chooses the form using the DSL's `+// 依存:+` and `+// 副作用:+` as cues: a note suggests injection, an internal computation a `+let+`, a chain of state transitions a `+>->+`.

[#java-base-class]
=== Java base-class generation

A behavior subject to injection (wrote a type, no `+let+`; <<injected-behavior>>) is generated as an *abstract base class*. It extends `+Behavior+` and, for each unit data listed in `+constructs+`, has a `+protected+` factory that constructs it.

[,text]
----
behavior findMember : (id: 会員ID) -> 会員 | 会員なし | 保存データ不正
    constructs 会員なし, 保存データ不正
----

generates, conceptually:

[,java]
----
public abstract class FindMember implements Behavior {
    protected FindMember() {}
    protected final 会員なし   会員なし()   { ... }   // only declared unit cases
    protected final 保存データ不正 保存データ不正() { ... }
    // apply(input) is inherited from Behavior (unimplemented). An implementation overrides it.
    // A platform failure such as DB down is not a case but an exception. The implementation throws; Souther passes it through.
}
----

Java `+extends+` this and implements `+apply+` (<<java-example>>). Success values (such as `+会員+`) are assembled by the decoder; failure cases are created by the inherited `+protected+` factory. Because a data's constructor is non-public, the implementation may be placed outside the generated package (in any package). An output case that is not a unit or that carries an invariant gets no factory and is obtained via the decoder.

This abstract base class is *always public*, regardless of whether it is in `+exposing+`, so that a Java implementation can `+extends+` it from any module; a behavior subject to injection is by definition implemented outside the module. `+exposing+` controls the exposure of domain types, but the injection base class is scaffolding for implementation and is exposed independently. Without it the injection example of <<example>> would not hold.

When a *non-unit* data is listed in `+constructs+` and that data is not exposed with `+exposing+`, it is a compile error (E1305). A factory is generated only for a unit data, and a non-exposed data has no decoder either, so a Java implementation would have no way to create that value. If exposed, it can be created via the decoder (<<jvm-codec>>). The requirement is that it be either a unit data or an exposed data.

What is confined is <<invariant-on-construct>>. An invariant-bearing data is always checked whether it goes through `+__construct+` or the decoder. So even if it can be created from outside, *no unvalidated value can be created*. A unit data has nothing to validate, so nothing is lost by allowing it to be created from outside.

[#java-impl-rules]
=== Rules for Java implementations

A Java implementation returns a *domain outcome* — a success value (such as `+会員+`) and a declared failure case (such as `+会員なし+` / `+保存データ不正+`).

A *platform failure* (DB down, network drop, HTTP failure — anything that is not a domain outcome) is not folded into a case but *MAY be thrown as an exception*. Souther does not make the exception a case but passes it through to the boundary (a generated `+>->+` pipeline or `+apply+` does not swallow the exception). The language (`+.sou+`) has no exceptions (E1302, <<out-of-scope>>), but a Java Binding that touches the outside world may throw. The boundary (the Java / framework side) receives and handles the exception — HTTP 503, a retry, a transaction rollback. A VM failure or an implementation bug also escapes as an exception (out of scope for guarantees).

Guidance for the distinction. A *domain outcome* (not found, insufficient stock, a stored value breaks a domain invariant — a business-meaningful result) is a case, appears in a domain type, and is subject to exhaustiveness checking. A *platform failure* (infrastructure unavailable or not responding) is an exception, does not pollute domain types, and is handled at the boundary as a cross-cutting concern. Folding DB-down into a case would make every DB-touching behavior's domain type carry an infrastructure case, and when the DB actually goes down there is no honest case value to return (one would have to fake "insufficient stock" or fabricate an infrastructure case). So the former is a case, the latter an exception (ADR-0029).

The exception *type* is agreed between the boundary and the implementation. With jOOQ + Spring, the boundary catches `+org.springframework.dao.DataAccessException+` and maps it to 503. jOOQ's own `+org.jooq.exception.DataAccessException+` is not a subclass of Spring's type, so this assumes Spring's exception translation for jOOQ is in effect (Spring Boot's jOOQ auto-config adds it by default). An implementation that throws a checked exception (such as raw JDBC's `+SQLException+`) wraps it in a non-checked exception of the same `+DataAccessException+` family (e.g. `+DataAccessResourceFailureException+`) — not `+IllegalStateException+` or a bare `+RuntimeException+` (which would pass through a `+DataAccessException+`-typed handler and make an infrastructure failure look like a program bug). The handler catches this failure type narrowly, and an implementation bug (an exception) falls through to a general 500.

`+保存データ不正+` (a stored value breaks a domain invariant on decode) stays a domain case. Infrastructure worked and returned data, and that data breaking a domain rule is a statement about the *domain*. Whether it deserves a name is the model author's judgment; this rule forces neither.

[#set-invariant]
=== Set invariants are passed as input data

An invariant that a single object cannot decide (schedule overlap, uniqueness) MUST be stated by passing the set needed for the decision as input data, not as a hidden dependency. The side effect of reading existing applications is placed at the workflow boundary (a behavior with no implementation).

[,text]
----
// spec DSL: behavior 日程重複を確認する = 同一申請者の出張申請リスト AND 申請準備中 -> 申請準備中 OR 日程重複
data 日程重複

behavior 日程重複を確認する : (
    同一申請者の出張申請リスト: List<出張申請>,
    申請:                    申請準備中
) -> 申請準備中 | 日程重複
    constructs 日程重複        // 申請準備中 returns the input as is; not created

// a behavior with no implementation (reads the outside)
behavior 同一申請者の申請を読む : (申請者: 従業員ID) -> List<出張申請>
----

`+日程重複を確認する+` stays a pure judgment; the side effect is on the outer behavior with no implementation. This corresponds to the conclusion of SMDD's DDD trilemma (purity is not layer separation but making explicit which functions have side effects).

[#requirement-propagation]
=== Requirement propagation

Anything referencing a behavior with no implementation inherits its requirement. Requirements accumulate outward, toward the side that wrote the name. If a `+let+` calls a behavior with no implementation, it becomes the requirement of the behavior holding that `+let+`. It accumulates further onto a composition placing that behavior in a stage.

At the destination, whether to write or infer diverges (<<requires>>).

* a simple behavior with a `+let+`: *write* in `+requires+`; receive it as a `+let+` argument
* a `+>->+` composition: *infer*; decided by the union of the stages' requirement sets

[,text]
----
behavior loadAndConvert = findMember >-> toView      // requires not written
----

This requires the implementation of `+findMember+`. Composing several gives a union. A composition is not made to write it, because it is readable from the stage declarations; the spec DSL also attaches no `+// 依存:+` to a composition.

The call graph is static at every stage, so the requirement set is decided as a concrete set. That no variable enters is what makes this design hold: inference and declaration can be reconciled, and the composition union can be computed, because the set is concrete (<<core-semantics>>).

This is unchanged by making a `+let+` higher-order.

[,text]
----
let 適用する (xs: List<会員ID>, p: (会員ID) -> 会員表示) = List.map(p, xs)
----

The requirement set of `+適用する+` is empty. Whatever `+p+` requires is irrelevant, because `+p+` arrives already closed. Requirements are injected as constructor arguments (<<jvm-behavior>>) and enter at the moment `+new 記録する(this.現在時刻)+` is created. The receiver need not know what it required. `+bind+` is the operation that partially applies that argument.

[#composition]
== behavior composition

[#sequential-composition]
=== Sequential composition `+>->+`

`+>->+` connects behaviors. Of the previous stage's output cases, those the next stage's input type can receive flow to the next stage; the cases it does not receive are carried to the output (departing the mainline).

[,text]
----
behavior handle = findMember >-> toResponse
----

The stage following `+>->+` is single-input (a 1-argument behavior). `+>->+` passes one value to the next stage, so the receiver cannot take two or more. To place a multi-argument behavior downstream, call it inline in the implementation or receive it with `+match+` instead of making it a stage.

*The first stage is not restricted.* The first receives the pipeline's own arguments, so any number is fine; the pipeline receives them as is.

[,text]
----
// spec DSL: behavior 却下する = 事前承認待ち AND 却下者ID -> 事前承認却下済み OR 承認権限なし
//           behavior 差し戻す = 事前承認却下済み -> 申請準備中
//           behavior 却下して差し戻す = 却下する >-> 差し戻す
behavior 却下して差し戻す = 却下する >-> 差し戻す
// 却下して差し戻す : (事前承認待ち, 従業員ID) -> 申請準備中 | 承認権限なし
----

The rule of <<type-routing>> requires single input only for the right side; however many the left side takes is irrelevant to routing.

A multi-input pipeline is itself a multi-input behavior, so it cannot be placed downstream of another pipeline. It can be the first stage.

A stage is a behavior, not a `+let+`. `+>->+` is composition on the specification side, and the spec DSL writes it so. A helper `+let+` does not appear in the spec DSL and cannot be a stage.

Decode / encode are not behaviors but edges of the boundary and are not `+>->+` stages. At the boundary, decode first, pass only the successful domain value to the behavior pipeline, and emit the output with encode (<<decoder>>, <<encoder>>). `+>->+` composes only the domain world (behaviors and their sum outputs).

[#type-routing]
=== Type-routing composition rule

A pipeline carries the *mainline*. Only the mainline is passed to a stage; a case a stage did not receive departs the mainline and is *never passed to the remaining stages* of that pipeline. A departed case is piled onto the output.

Let the previous stage's output sum be `+O_f+` and the next stage's input type `+I_g+`. Of `+O_f+`, cases `+I_g+` can receive flow to the next stage; the remaining cases `+R+` depart the mainline.

[,text]
----
f : A -> B | R           g : B -> C           f >-> g : A -> C | R
f : A -> B | R           g : B -> C | S       f >-> g : A -> C | S | R
f : A -> B1 | B2         g : B1 | B2 -> C     f >-> g : A -> C
----

The same holds for three or more stages: *only the mainline passes to the next stage*.

[,text]
----
f : A -> B | C     g : B -> D     h : C | D -> E
f >-> g >-> h : A -> E | C
----

`+h+` can receive `+C+`, but `+C+` already departed the mainline at `+g+`, so it does not reach `+h+`. What reaches `+h+` is only the `+D+` that `+g+` produced. This is the Railway. "Failure" is not special-cased; an unconsumed case simply departs the mainline, and which case departs is decided by the composition partner (<<unmarked-sum>>).

The output sum is treated as an open error row: `+(C | R) | S+` is flattened and same-named cases folded. If no case of `+O_f+` matches `+I_g+`, composition is impossible (<<e1701>>).

That a case departed the mainline is remembered by the composition, and not forgotten by naming a midpoint.

[,text]
----
behavior fg = f >-> g          // mainline D, departed C
behavior x  = fg >-> h         // only mainline D passes to h; C stays departed. x : A -> E | C
----

`+fg+` remembers mainline `+D+` and departed `+C+`. In `+fg >-> h+`, only mainline `+D+` passes to `+h+`, and `+C+` stays departed and is carried to the output. So a named-midpoint `+fg >-> h+` is the same `+A -> E | C+` as the one-line `+f >-> g >-> h+`.

`+>->+` is associative. Hoisting a sub-pipeline by name or regrouping does not change the result.

The mainline/departed split is composition plumbing, not a marker on values (<<unmarked-sum>>). The plumbing unwinds at both ends of the pipeline. Viewing `+fg+` alone, its output is the folded sum `+D | C+`, and the external representation is as before. Even if the same type of case appears in both mainline and departed, the plumbing remembers which rail, so they are not mistaken.

When a downstream stage receives only some cases, or you want to branch per case, open branches with `+match+` instead of `+>->+`.

[,text]
----
match 出張申請を提出する(申請) with
    | 提出済み as s   -> ...
    | 事前承認待ち as p -> ...
----

[#composition-with-requirements]
=== Composition with outside-world dependencies

A behavior's internal representation is:

[,text]
----
Behavior<Requirements, Input, Output>
----

`+Output+` is the sum of possible business results, with no fourth slot separating failure. Composition rule:

[,text]
----
f : Behavior<R1, A, O_f>
g : Behavior<R2, B, O_g>        // I_g = B receives cases of O_f
f >-> g : Behavior<R1 ∪ R2, A, (O_g) | (sum of O_f minus the consumed part)>
----

`+R1+` and `+R2+` are what the stages declared (<<requires>>). The composition's `+R1 ∪ R2+` is inferred by the composition; a composition does not write `+requires+`. Whether an implementation is internal or Java-side does not affect value composition. `+Requirements+` (dependencies needed) and `+Output+` (the sum of results produced) are composed independently: the former by set union, the latter by consumption / propagation.

`+Output+` consumption / propagation splits into the *mainline* passing downstream and the cases that departed unconsumed. The composition carries this split, so, like the `+R1 ∪ R2+` set union, `+>->+` is associative (<<type-routing>>). The mainline / departed split stays in the composition plumbing and folds back into a sum when it exits as `+Output+`. The internal representation needs no fourth slot for failure.

[#single-case-composition]
=== Composition of single-case output

A behavior with a single-case output (`+-> A+`) can be seen as having no passed-through case (an empty failure row, `+Never+`). If a downstream stage receives `+A+`, it flows through. The user need not wrap it in anything. If `+require ... else+` returned another case early, that case propagates as `+R+` of <<type-routing>>.

[#declared-composition-output]
=== Declaring and reconciling a composition's output

A `+>->+` composition's output is inferred from its stages (<<composition-with-requirements>>). A spec DSL composition line writes no output (`+behavior 注文を確定する = 注文を検証する >-> ...+`), so the default is this inference.

An inferred output *changes silently on a distant change* when a stage is a behavior of another module. If an upstream stage adds a case, the composition output grows silently; if a downstream stage narrows its input, a previously processed case moves to departed (<<type-routing>>) and likewise increases the output. The composition's definition site has nothing to signal this.

So a composition *MAY optionally declare its output*. If written, it MUST *exactly match* the inferred output or it is a compile error (E1604). Neither too few (a case appears that is not declared) nor too many (a declared case does not appear) is admitted.

[,text]
----
behavior 注文を確定する = 注文を検証する >-> 価格を記載する >-> 在庫を引き当てる
    -> 確定注文 | 在庫不足
----

`+-> 確定注文 | 在庫不足+` is the composition's declaration, not the output of the last stage `+在庫を引き当てる+`. Because a stage is a behavior reference (name only), the trailing `+->+` reads unambiguously as the composition's. Writing this line makes an error appear here when an upstream stage adds a case, so it does not grow silently downstream.

Input is not declared. A pipeline is point-free, and the input is decided by the first stage (<<sequential-composition>>). Only the output may be declared. A dependency (`+requires+`) has the same distant-change risk, but a composition leaves it inferred (<<requires>>).

For an internal composition this declaration is optional. But for a composition exposed with `+exposing+` it is *required*, written not on the definition line but in the exposing list as `+name : cases+` (<<modules>>). Leaving a public boundary to inference means that when an upstream stage adds a case, it reaches another compilation unit (another module, or a non-exhaustive Java switch or JSON serialization) silently, and the definition site has nothing to signal it. Placing a signature in `+exposing+` makes the composition's definition error (E1604) the moment a case is added, and the author chooses to handle the case or update the signature. If no signature exists, it is E1605. A non-composition behavior writes its type at the definition, so it MUST NOT write a signature in `+exposing+` (doing so is E1605). This division corresponds to ADR-0024.

[#case-propagation]
== behavior case propagation and Decoder error accumulation

A behavior's output sum is treated as an open error row; a case the downstream does not consume in a `+>->+` propagates through to the tail (<<type-routing>>). This is the composition style of the domain world.

Decode-side error accumulation is handled by the peripheral library Raoh. A derived Object Decoder reads each field from the key of the same name and collects all failures of independent fields (Applicative).

[,text]
----
{ "name": ..., "email": ..., "age": ... }
// read name / email / age with each field's derived Decoder and accumulate all independent failures (Raoh)
----

A derived Decoder's field accesses are mutually independent and collect all failures. Accumulated failures aggregate into Raoh's `+Result+` failure (`+Issues+`). In a custom boundary codec (<<custom-codec>>), one access may depend on a preceding result; from that point it becomes sequential (monadic), and a subsequent access is not run if the preceding one fails.

Accumulation (applicative) is inside the decode boundary; case propagation is behavior composition. They belong to different layers. Because decode is not a `+>->+` stage, the two never mix on the same sum (<<sequential-composition>>). Accumulation holds because a Decoder's field accesses are mutually independent (do not depend on a prior value).

[#examples]
== Examples

An `+example+` attaches compile-time-checked test rows to a behavior. It is evaluated as part of compilation: a row that does not hold is a compile error (<<e1905>>), so a stale example cannot survive a build. An `+example+` is not exposed and produces no run-time class.

[,text]
----
example 提出する
    | "上限内なら提出できる" :
        (申請準備中 { 申請者 = 従業員ID("e-001"), 予定費用 = 金額(50000) }, "2026-07-20T09:00")
            -> 提出済み
    | "上限超過は却下" :
        (申請準備中 { 申請者 = 従業員ID("e-001"), 予定費用 = 金額(100001) }, "2026-07-20T09:00")
            -> 却下 { 理由 = "high_cost" }
----

An `+example+` names a target behavior, then lists one or more `+|+`-led rows. A row is an optional description string (a business-case name, followed by `+:+`), an input argument tuple, `+->+`, and an expected result. `+example+`, `+examples+`, and `+for+` are contextual keywords, not reserved words, so they do not collide with an identifier or the `+example.*+` namespace.

[#example-expected]
=== The expected result

The expected side asserts at one of two granularities.

* A bare type name (`+-> 提出済み+`) asserts only the *arm* — which case of the output sum the result is — and ignores its fields.
* A construction (`+-> 却下 { 理由 = "high_cost" }+`), a newtype application (`+-> 金額(0)+`), or a literal asserts the *whole value* by structural equality.

Types make totality and the case set exhaustive (<<match>>, <<unmarked-sum>>); an `+example+` fixes the value rule for a representative input — which arm, or which value, a given input yields.

[#example-evaluable]
=== What can carry an example

A behavior with a `+let+` implementation is evaluated (<<fn>>). An injected behavior (no `+let+`, <<injected-behavior>>) or a `+>->+` composition cannot be — you fake an injected behavior, you do not example it (<<e1902>>). A behavior's `+requires+` dependencies are supplied by fakes at the example (<<example-fakes>>).

An input and an expected value are fixtures: literals, newtype applications, and record constructions. A row's argument is decoded into the behavior's parameter type through that type's derived Decoder (<<decoder>>), so a fixture that breaks an invariant is rejected (<<e1903>>).

[#example-fakes]
=== Faking a requires dependency

To evaluate a behavior that `+requires+` an injected dependency (<<requires>>), the example supplies a fake for it. Best is to need none: an outside-world value written as an *input* rather than an injected behavior keeps the behavior example-able with no fake (<<purity>>). When a dependency stays injected, a fake stands in, scoped to the example and never shipped (no run-time class is produced).

A *value dependency* — one whose faked result is a constant — is given on the row with `+with+`:

[,text]
----
example 事前承認する
    | (事前承認待ち { ... }, 従業員ID("boss-1")) with 現在時刻 = "2026-07-20T09:00" -> 事前承認済み
----

A *function dependency* — one whose result varies with its input — is given by a `+fake+` declaration: an input→output table matched by value equality, defaulting to a `+_+` row or, absent one, a miss error (<<e1909>>).

[,text]
----
fake 上長を探す
    | (従業員ID("e-1")) -> 従業員ID("boss-1")
    | _               -> 従業員ID("unknown")
----

Each requires dependency of the target must have a fake — a `+with+` on the row or a `+fake+` table — or it is a compile error (<<e1908>>). A fake, like an example, is written inline or in an attached `+examples for+` file, and is local to its module. `+fake+` is a contextual keyword.

[#example-placement]
=== Inline or an attached file

The base form is written inline, in the same source as the behavior. To keep fixtures out of the model source, examples may instead be written in an attached file that begins with `+examples for <module>+` and contains only `+example+` declarations. The compiler merges an attached file into its target module; it is not a module of its own and needs no `+import+` (it sees the target module's names directly). An attached file that names a module not being compiled is <<e1907>>; one that contains anything other than an `+example+` is <<e1906>>.

[#control-flow]
== Control structures

[#let]
=== let

[,text]
----
let normalized = String.trim(input)
----

Reassignment is forbidden. A `+let+` is placed inside a block expression (<<block-expression>>). What follows `+let x = v+` is the scope of that binding, nested at desugaring time.

[#if]
=== if

[,text]
----
if condition then expression1 else expression2
----

`+if+` is an expression. If both branch types agree, its type is that type; if they are different data, it is a sum of both (`+if c then 提出済み{...} else 事前承認待ち{...}+` is `+提出済み | 事前承認待ち+`). A mismatch between primitives is a compile error.

[#match]
=== match

[,text]
----
match 出張申請 with
    | 申請準備中 as d   -> ...
    | 提出済み as s     -> ...
    | 事前承認待ち as p  -> ...
    | 事前承認済み as a  -> ...
----

Place `+with+` after the scrutinee and list cases one per `+|+` (the same form as F#). The leading `+|+` of the first case is optional. Braces are not used. Souther is layout-independent, so Elm's `+case ... of+` (indentation-dependent) is not adopted; only `+|+`-separated cases hold without layout.

A case value is bound with `+as+` (the `+s+` in `+| 提出済み as s+` is 提出済み, read as `+s.提出日時+`). `+as+` binds the whole matched value.

Fields may be bound directly with a destructuring pattern, the inverse of construction. It uses the same `+{ }+`, `+,+`, `+=+` as construction `+会員 { id = x }+`, and `+{ id }+` is shorthand for binding to a same-named variable (paired with <<record-literal>>). Write only the fields used. Destructuring is possible only for a single named case; an or-pattern (below) binds to a sum type and has no fields. It may be combined with `+as+`: `+| 会員 { id } as 全体+` binds both fields and the whole.

[,text]
----
match c with
    | EmailContact { email = e } -> ...   // e is the email field
    | 会員 { id, メール }         -> ...   // { f } binds f to a same-named variable
----

To route multiple cases to the same processing, list `+|+` within a case, `+| A | B -> ...+` (an or-pattern). It writes "these states are treated the same" for a state-machine sum in one line. The case-separator `+|+` and the or-pattern `+|+` are the same symbol, but `+->+` ends a case's pattern part, so they do not conflict.

[,text]
----
match 出張申請 with
    | 提出済み | 事前承認待ち as x -> ...   // x is 出張申請 (the sum type)
    | 申請準備中 as d -> ...
    | 事前承認済み as a -> ...
----

An or-pattern's binding differs from a single case. A single case's `+as+` binds to that case's type (the `+s+` in `+| 提出済み as s+` is `+提出済み+`), but an or-pattern's `+as+` binds to *the scrutinee's sum type*, because no single case type fits all alternatives. So an or-pattern binding cannot read case-specific fields; carry it as the sum or open it again with match. Exhaustiveness counts cases one by one; a case received by two cases is a duplicate error. This binding rule is the same position as OCaml's and Rust's or-patterns (bind to the same type across all alternatives).

The `+Some+` of `+Option+` (<<algebraic-types>>) extracts the wrapped value by *positional binding*. The `+v+` in `+| Some v+` is bound to the contents, not to `+Some+` (as in F#'s `+Some x+` and Elm's `+Just x+`). `+Some+` is the only built-in case with a payload, so it reaches the contents by positional binding rather than `+as+` (which binds the whole value elsewhere). `+| Some as v+` cannot be written. A `+?+` field and the return of `+List.get+` / `+Map.get+` are extracted this way.

[,text]
----
data 出張完了 = { 事前承認者: 従業員ID? }

let 承認者の表示 (c: 出張完了) =
    match c.事前承認者 with
        | Some 承認者 -> 承認者.value      // 承認者 is 従業員ID (the contents)
        | None        -> "なし"
----

A match on sum data MUST be exhaustive. Exhaustiveness is decided over the direct cases the sum declared. A nested sum (`+自社負担+` of `+data 費用負担区分 = 自社負担 | 先方負担+` being further `+立替 | 仮払い | 会社カード+`) does not flatten leaf cases; receive `+自社負担+` and split with a nested `+match+`. Exhaustiveness is checked independently per layer, so cases never overlap. Case→sum assignment (<<sum-data>>) folds to the leaf; match is the reverse, descending the layers.

[#require]
=== require

[,text]
----
require amount <= balance
    else 残高不足
----

`+require+` is used inside the implementation (`+let+`) of a behavior whose output has a departure case. If the condition is false, it returns the `+else+` value and ends on the spot. The `+else+` value is one of the output sum's cases (just data). The implementation only places the value of the case it wants to return as the last expression. No special wrapping.

`+require+` is sugar for `+if+`. The following two mean the same.

[,text]
----
require <condition> else <value>
<rest of the body>
----

[,text]
----
if <condition> then <rest of the body> else <value>
----

Desugaring is done at parse time; later stages (type check, construction-authority check, backend) see only `+if+`. That both branches may have different types (<<if>>) is what makes this desugaring hold.

[#block-expression]
=== Block expressions

`+{ ... }+` is an expression. Place `+let+`s and `+require+`s, then put the result expression last.

[,text]
----
let 事前承認する (申請, 承認者ID) = {
    require 承認者ID == 申請.申請者.上長ID
        else 承認権限なし

    事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
}
----

If only the result expression is placed, no block is needed. `+let 正の数 (v: Int) = v >= 0+` and `+let 正の数 (v: Int) = { v >= 0 }+` are the same.

A block is an expression, so it may be written anywhere an expression may (`+if c then { let x = f(a)  g(x) } else h+`). Its contents desugar per the rules of <<let>> and <<require>> and appear in the AST as bindings and `+if+`. A bare `+{+` resolves to a block expression; a record literal is preceded by a type name (<<record-literal>>), so there is no confusion.

[#purity]
== Purity

Souther code *does not touch the outside world directly*. The following are forbidden.

Mutable variables, field assignment, static mutable fields, random, current time, environment variables, files, network, threads, exceptions, reflection.

An outside-world dependency is declared as a behavior with no implementation, and its implementation is injected from the Java side (<<injected-behavior>>). A `+let+` may receive one as an argument and call it, so a `+let+`'s result may depend on the outside world. But the way it touches must go through a call to the injected thing, and that fact is written in `+requires+` (<<requires>>). This corresponds to the conclusion of SMDD's DDD trilemma — purity is not layer separation but making explicit which functions have side effects (<<set-invariant>>). *Souther made this "explicit" literal.* Left to inference, which functions have side effects cannot be read from the declarations.

*"The requirement set is empty" is stronger than this sense of purity.* This chapter's sense is "does not touch directly", which every `+let+` satisfies. An empty requirement set means receiving nothing injected, which only some `+let+`s satisfy. What may be called from an invariant is the latter (<<invariant-expressions>>). To avoid conflating the two, the latter is henceforth called "empty requirement set" and not "pure".

[#stdlib]
== Standard library

[#reserved-namespace]
=== Reserved namespace

The standard library is bundled with the compiler and referenced by short qualifiers — `+List+` (list), `+Map+` (map), `+Set+` (set), `+String+` (string), `+Bool+` (boolean), `+Int+` / `+Decimal+` (numbers), `+Date+` / `+DateTime+` (temporal). Every function is called through this qualifier: `+List.map(f, xs)+`, `+String.trim(s)+`, `+List.length(xs)+`. There is no automatic import — names are not delivered implicitly. What may be written bare is operators (`+==+` `+<+` `+{plus}{plus}+`, etc.), `+if+` / `+match+`, literals, and the current module's helpers. Argument order follows the convention of <<pipe>> (primary subject last).

To call a name bare, import it explicitly: writing `+import List ( map, filter )+` lets you write `+map(f, xs)+` unqualified thereafter. As in Elm's `+import List exposing (map)+`, qualified access always works, and `+import+` is only sugar to elide the qualifier. Importing the same bare name from two modules is ambiguous, so qualify one of them (e.g. `+String.length+` and `+List.length+` used together in one module; <<imports>>).

Qualifiers such as `+List+` denote a module only in expression position, distinct from the type-position `+List<T>+` (angle brackets), which does not conflict (as in Elm). `+length+` / `+get+` span `+String+` and `+List+`, and `+List+` and `+Map+`, but they become distinct names `+String.length+` / `+List.length+`, `+List.get+` / `+Map.get+`, so they are not overloads but each module's own function.

The entity the short qualifier denotes is placed in the reserved namespaces `+souther.string+` / `+souther.list+` / `+souther.map+` / `+souther.set+` / `+souther.bool+` / `+souther.date+` / `+souther.datetime+`. The `+souther+` namespace is reserved, and a user module MUST NOT name itself under it — `+module souther.foo+` is a compile error. Only the reserved namespace holds core's privileges (generics, recursion, intrinsics), and this is the line preventing a user from granting them to itself (the boundedness of <<algebraic-types>>). A name that merely starts with the string `+souther+`, such as `+southernmost+`, is an ordinary user name, not under this namespace.

[#type-variables]
==== Type variables

A core function may write a type variable `+'a+` (a leading apostrophe, as in F# and OCaml) in a type position — a parameter, a return value, `+List<'a>+`, `+Map<String, 'a>+`, or a function type `+('a) -> 'b+`. No declaration list (`+<A>+`) is written; a `+'a+` appearing in a signature is implicitly universally quantified (F#'s auto-generalization). Because identifiers include Japanese and do not distinguish case (<<unit-data>>), Haskell's and Elm's "lowercase = type variable" is not adopted; a leading apostrophe is the ML-family solution that holds while case is not distinguished.

Only core in the reserved namespace may write a type variable; a user module writing `+'a+` is a compile error. The user model stays bounded (<<algebraic-types>>). When a non-recursive core helper has `+'a+`, it is monomorphized by inline expansion at the call site — `+'a+` is resolved to each call's concrete type, and no polymorphic method is generated.

[#intrinsics]
==== intrinsics

A primitive is declared as a core function with a body of `+= intrinsic "key"+`. `+key+` names the JVM operation the backend emits and is *prefixed per type* to make it unique (`+"string.trim"+`, separating `+list.length+` and `+string.length+`).

[,text]
----
let trim (s: String) : String = intrinsic "string.trim"
----

An intrinsic MUST declare its return type (it cannot be inferred from `+intrinsic+`). A call is type-checked against the declared signature — that is, `+trim+` is "a built-in function whose signature is placed in core", and this declaration, not a hardcoded compiler table, decides its arguments and return value — and the backend emits the bytecode for `+key+`. It is neither inlined nor lowered. Only the reserved namespace may write it. A user cannot call the JVM directly and only calls blessed functions (<<asymmetric-interop>>).

[#pipe]
=== The value pipe and argument order

To thread a value through functions, write `+|>+` left to right. `+e |> f(a)+` is `+f(a, e)+` — the left side enters the *last argument* of the right-hand call. A parenthesis-less function name `+e |> f+` is `+f(e)+`. `+|>+` binds most loosely and left-associatively, so `+a |> f |> g+` reads as `+g(f(a))+`.

[,text]
----
入力.生
    |> split(",")                          // split(",", 入力.生)
    |> map(片 -> 片 |> trim |> lowercase)
    |> filter(片 -> String.length(片) >= 1)
    |> sort                                // sort(...)
----

Standard library arguments are set to F#'s and Elm's order to match — *the primary subject (collection, string, Map) last, functions, separators, and keys first*: `+List.map(f, xs)+`, `+List.filter(p, xs)+`, `+List.fold(step, seed, xs)+`, `+String.split(sep, s)+`, `+Map.get(key, m)+`. So the left value drops into the last argument and the pipe flows top to bottom. A `+fold+`'s folding block stays in `+(acc, x)+` order (F#'s `+List.fold+`).

The right side admits only a function call or a function name; `+e |> 3+` is a compile error. `+|>+` is expanded to a call at parse time, so no later stage sees `+|>+` — the same pure syntactic sugar as `+require+` expanding to `+if+`. It is distinct from the `+>->+` that composes behavior stages (<<composition>>); that connects behaviors, this threads expression values.

[#stdlib-string]
=== String

[,text]
----
length  trim  lowercase  uppercase  contains  startsWith  endsWith  substring  append
split    join  replace    words      fromInt   concat
----

`+append(a, b)+` joins two strings in their written order (Elm `+String.append+`); the `{plus}{plus}` operator is the common form, `+append(a, b)+` equals `+a {plus}{plus} b+` (<<stdlib-list>>).

The lower row is the core of Elm's String module expressible in Souther's types (which lack `+Char+` / `+Maybe+` / `+Float+`). `+split(sep, s)+` splits by a separator into `+List<String>+`, keeping empty pieces (`+split(",", "a,,b")+` is `+["a", "", "b"]+`). `+words(s)+` splits by runs of whitespace, dropping empty pieces. `+join(sep, xs)+` joins with a separator. `+replace(target, replacement, s)+` replaces every occurrence of `+target+`. `+fromInt(n)+` makes an `+Int+` a decimal string. `+concat(xs)+` joins a `+List<String>+` with no separator (Elm `+String.concat+`, equal to `+join("", xs)+`). The primary subject string is last (<<pipe>>). Elm functions depending on `+Char+` or `+Maybe+` (such as `+String.map+` and `+toInt+`, which returns `+Maybe+`) are not ported, as there is no corresponding type.

[#stdlib-int]
=== Int

[,text]
----
add  subtract  multiply  divide  remainder  compare
----

`+Int+` is signed 64-bit (<<primitives>>). If an operation exceeds that range it *aborts* — it does not wrap. Overflow is a model bug, not a business result, so like an invariant violation it throws `+ConstraintViolation+` (<<algebraic-types>>, <<violation-destination>>, <<jvm-abort>>).

Division by zero returns `+Int | DivisionByZero+`. This is a possible input on the business flow (a divisor of 0), so it is returned as a case rather than aborting.

The four operations may also be written with the operators `{plus}` `+-+` `+*+` `+/+` (the symbol table of <<primitives>>). `+/+` is truncating division and *aborts* on division by zero (like overflow). To receive `+Int | DivisionByZero+` as a case, use the functions `+divide+` / `+remainder+` — operator = casual and aborting, function = safe with a case.

[#stdlib-decimal]
=== Decimal

[,text]
----
add  subtract  multiply  divide  compare
----

Division makes the rounding scale and mode explicit as arguments. `+Decimal.divide(a, b, 2, HALF_UP)+` rounds HALF_UP at 2 decimal places. The rounding mode is a built-in identifier, resolved as a built-in value like `+None+` (<<unit-data>>).

[,text]
----
HALF_UP  HALF_EVEN  HALF_DOWN  UP  DOWN  CEILING  FLOOR
----

These correspond to `+java.math.RoundingMode+`. `+Decimal.divide(a, b)+` without scale and mode cannot be written — to surface rounding as a domain decision, the same position as "scale is incidental" in <<primitives>>. Division by zero returns `+Decimal | DivisionByZero+`.

The four operations may also be written with the operators `{plus}` `+-+` `+*+` `+/+`. The operator `+/+` divides with a default rounding — significant digits (about 29) and HALF_UP, following F#/.NET's `+System.Decimal+` — so `+10m / 3m+` is `+3.3333…+` and does not abort. Division by zero *aborts*. To make rounding an explicit domain decision, or to receive division by zero as a case, use the function `+Decimal.divide(a, b, scale, mode)+`.

[#stdlib-list]
=== List

==== Construction

A list is made by a literal or a guard comprehension.

[,text]
----
[e1, e2, ...]         // literal. One or more elements
[要素 | 条件, ...]     // guard comprehension. One element if all conditions hold, empty otherwise
----

An empty literal `+[]+` may also be written. It has no element type of its own; its type is decided by context — the counterpart of `{plus}{plus}`, the other branch of an `+if+` / `+match+`, the seed a `+fold+` grows, or a `+List<T>+` a field or return value requires. An empty list does not depend on elements at runtime, so this is always sound. To transcribe a DSL bullet list, a guard comprehension (empty if the condition is false) is natural.

`{plus}{plus}` is concatenation (Elm's appendable). Both sides MUST be two lists or two strings: two lists give a list, two strings give a string (`+"a" {plus}{plus} "b"+` is `+"ab"+`, the operator form of `+String.append+`; <<stdlib-string>>). A mixed pair (a list and a string, or either with a non-list non-string) is a type error.

The element type is folded if the same, and widened to a sum of cases if different data (the same direction as <<sum-data>>).

[,text]
----
[高額出張 { 基準金額 = 基準額 }] ++ [先方費用負担]      // List<高額出張 | 先方費用負担>
----

Guard comprehension and `{plus}{plus}` exist to transcribe conditions the spec DSL listed as bullets.

[,text]
----
// the spec DSL (chapter 10) lists reasons as bullets:
//   reason 1: 予定費用 is at least 100,000 yen / reason 2: applicant is 一般社員 / reason 3: cost borne by the other party
   [高額出張 { 基準金額 = 基準 } | 予定費用.value >= 基準.value]
++ [権限不足 { value = 申請者.役職 } | 一般社員か(申請者)]
++ [先方費用負担                 | 先方負担か(費用負担区分)]
----

Each reason makes a 0- or 1-element list and concatenates. *One DSL bullet line becomes one Souther line.* The result is `+List<高額出張 | 権限不足 | 先方費用負担>+`, matching the three cases of `+事前承認理由+`. A unit data is constructed by name alone (<<unit-data>>), so `+先方費用負担+` is written directly.

Note that `+申請者.役職 == 一般社員+` cannot be written as a condition. `+==+` works only between the same type (<<primitives>>), and `+役職+` and its case `+一般社員+` are different types. Case judgment is done with `+match+`, and to make it a predicate wrap it in a helper `+let+` (the `+一般社員か+` above). Ordering (`+>=+`) exists only for `+Int+` / `+String+` / `+Decimal+` / `+Date+` / `+DateTime+`, so compare `+.value+`, not `+金額+` (a newtype) directly.

==== Operations

[,text]
----
map  filter  fold  all  any  length  get
reverse  member  sum  product  concat  isEmpty  sort
distinct  partition  groupBy  max  min  find  sortBy
----

All are called through the qualifier `+List+` (<<stdlib>>). `+List.get+` returns `+Option<T>+`. `+List.map+` / `+List.filter+` / `+List.fold+` / `+List.all+` / `+List.any+` take a block (<<blocks>>). Import them as `+import List ( map, all )+` to write unqualified thereafter.

[,text]
----
import List ( map, all )

map(x -> 明細を直す(x), xs)
all(r -> 軽微か(r), 理由リスト)
----

If an injected thing is called from within a block, its requirement appears in the enclosing behavior's `+requires+` (<<requires>>).

Of these, only `+List.fold+` is primitive. `+map+` / `+filter+` / `+all+` / `+any+` are self-language `+let+`s that `+souther.list+` derives from `+List.fold+` (<<stdlib>>, ADR-0028), expanded at the call site. `+map+` / `+filter+` seed with an empty list `+[]+` and grow it; `+all+` / `+any+` fold a `+Bool+`. `+List.length+` / `+String.length+`, `+List.get+` / `+Map.get+` were once a single overload but split into separate module functions by qualification — a `+length+` spanning `+String+` and `+List+` is called by each name.

The lower row is also ported from Elm's List module; except `+sort+`, all are self-language `+let+`s derived from `+List.fold+`. `+reverse+` extends the seed to the front, `+sum+` / `+product+` fold numbers, `+concat+` folds with `{plus}{plus}` and flattens nesting (Elm's `+List.concat+`), and `+member+` folds with `+==+` so it works for comparable elements. `+isEmpty+` looks at length. Only `+sort+` is a primitive that cannot be derived from `+fold+` (as in Elm) — self-language sorting needs, besides comparison, a place to hold intermediate results, which `+fold+` does not give; it sorts by the element's natural order. Elements MUST be an ordered type (`+Int+` / `+String+` / `+Decimal+` / `+Date+` / `+DateTime+`; <<primitives>>), and passing a list of data or newtype to `+sort+` is a compile error (they have no ordering, so it is rejected up front rather than at runtime; to sort them, map to an ordered field first with `+List.map+`).

The third row adds more of Elm's List module. `+distinct+` keeps the first occurrence of each element and drops later duplicates (Elm's `+List.Extra.unique+`), folding a `+Set+` of what it has seen; `+partition+` splits into `+(kept, rest)+` by a predicate; `+groupBy+` buckets elements into a `+Map<'k, List<'a>>+` by a key. These three are self-language `+let+`s that grow their result from within `+List.fold+`. `+max+` / `+min+` return the largest / smallest as an `+Option+` — `+None+` on an empty list — and compare by natural order, so like `+sort+` the element MUST be ordered (Elm's `+maximum+` / `+minimum+`); `+find+` returns the first element satisfying a predicate as an `+Option+`. `+sortBy(key, xs)+` orders by a projected ordered key rather than the element itself (F#/Elm `+List.sortBy+`): unlike `+sort+`, whose comparison is built in, its key is a runtime function value, so the sort carries an actual `+Fn+`. `+max+` / `+min+` / `+find+` / `+sortBy+` are primitives (they step outside `+fold+`); only Elm's `+List.head+` is still not ported — a domain reads the first element with `+List.get(0, xs)+`.

[#stdlib-map]
=== Map

A `+Map+` is immutable and string-keyed (<<collections>>). The accessors read; the building operations return a fresh map.

[,text]
----
get  containsKey  keys  values                   // read
empty  singleton  insert  remove  isEmpty  size  // build
toList  fromList                                 // convert to / from entry tuples
fold  map  update                                // higher-order
----

`+get+` returns `+Option<V>+`. `+empty+` is the empty map: like the empty list `+[]+`, its value type is fixed by context (a field or return position, or the first `+insert+`), so it is a built-in taking no argument rather than a call learning a type variable. `+singleton(k, v)+` is a one-entry map; `+insert(k, v, m)+` adds an entry or overwrites the existing one; `+remove(k, m)+` drops one, leaving `+m+` unchanged when the key is absent. Because a `+Map+` is immutable, `+insert+` / `+remove+` return a new map rather than mutating. `+size+` is the entry count and `+isEmpty+` whether there are none. The map operated on is last (<<pipe>>), so `+m |> Map.insert(k, v)+` reads left to right.

`+toList(m)+` is the entries as a `+List<(String, V)>+` — a list of key/value tuples in the map's iteration order — and `+fromList(entries)+` builds a map back (a later entry wins on a duplicate key). Iteration order is a deterministic, implementation-defined hash order (not insertion order): the same set of keys always yields the same order, so encoding is reproducible, but a program must not depend on which order that is. The tuple type in these signatures is allowed because a signature is not a codec boundary (<<tuple>>, ADR-0037).

`+fold+` / `+map+` / `+update+` are the operations that take a step function. `+Map+` adds no privileged loop of its own: `+fold(step, seed, m)+` runs `+List.fold+` over the entries (`+toList+`), so the one list-fold loop stays the sole primitive, and the entries combine in the map's deterministic hash order. Its step is `+(acc, key, value)+` (F#'s `+Map.fold+` folder is `+state key value+`, Elm's `+Dict.foldl+` is `+k v acc+`). `+map(f, m)+` keeps the keys and rewrites the values (`+(key, value) -> value+`, Elm's `+Dict.map+` / F#'s `+Map.map+`), growing a fresh map through `+fold+`. `+update(key, f, m)+` rewrites the value under one key with `+f+` and is a no-op when the key is absent. Its step is `+('a) -> 'a+`, not Elm's `+Maybe v -> Maybe v+`: `+Option+` is not surface-writable (ADR-0011), so a step cannot return one — insert a new key with `+insert+`, drop one with `+remove+`, and let `+update+` cover changing what is already there.

[#stdlib-set]
=== Set

A `+Set<'a>+` is unordered with no duplicates, compared by value equality (<<collections>>, ADR-0039). It is immutable, so the building operations return a fresh set.

[,text]
----
empty  singleton  insert  remove  contains       // build / query
union  intersect  difference                      // algebra
isEmpty  size  toList  fromList                   // convert
----

`+empty+` is the empty set — like `+[]+` and `+Map.empty+`, its element type is fixed by context, so it is a built-in taking no argument. `+singleton(v)+` is a one-element set; `+insert(v, s)+` adds `+v+`; `+remove(v, s)+` drops it (leaving `+s+` unchanged when absent); `+contains(v, s)+` tests membership. `+union+` / `+intersect+` / `+difference(a, b)+` are the set algebra (`+difference+` is `+a+` minus `+b+`). `+toList(s)+` lists the members in a deterministic, implementation-defined hash order (not first-seen order; the same members always list in the same order, so it is reproducible, but a program must not depend on which order that is) and `+fromList(xs)+` builds a set, dropping duplicates. The set operated on is last (<<pipe>>), so `+s |> Set.insert(x)+` reads left to right.

A `+Set+` field decodes from a JSON array and deduplicates it (a repeated element is dropped, not an error — as in Elm's `+Set.fromList+`); it encodes back to an array in that deterministic hash order (reproducible for the same members, but not the input order).

[#stdlib-bool]
=== Bool

[,text]
----
not
----

`+not+` is the *first self-host function* placed in `+souther.bool+`. It is not a primitive but written in Souther as `+let not (b: Bool) = if b then false else true+`, called qualified as `+Bool.not+` (bare via `+import Bool ( not )+`). Being non-recursive, it is inline-expanded at the call site and no dedicated class is generated. Negation is written with this `+not+` — there is no prefix `+!+` operator (redundant, since `+not+` exists).

[#stdlib-date]
=== Date

[,text]
----
addDays  addMonths  addYears  daysBetween
----

`+Date+` is a local calendar date (<<primitives>>). Calendar arithmetic is pure and belongs in the domain, not behind an injected behavior (<<injected-behavior>>) — only reading the *current* date needs the outside world. `+addDays(n, d)+` / `+addMonths(n, d)+` / `+addYears(n, d)+` shift `+d+` by `+n+` units (`+n+` may be negative), following `+java.time.LocalDate+`, which clamps the end of a month (`+addMonths(1, 2026-01-31)+` is `+2026-02-28+`). `+daysBetween(from, to)+` is the whole days from `+from+` to `+to+`, negative when `+to+` precedes `+from+`. The date operated on is last (<<pipe>>), so `+d |> Date.addDays(3)+` reads left to right.

[#stdlib-datetime]
=== DateTime

[,text]
----
addMinutes  addHours  addDays  minutesBetween  toDate
----

`+DateTime+` is a local date-time with no zone (<<primitives>>); the zone stays a boundary concern. `+addMinutes+` / `+addHours+` / `+addDays+` shift by their unit; `+minutesBetween(from, to)+` is the whole minutes between (negative when `+to+` precedes `+from+`); `+toDate(dt)+` drops the time of day, yielding a `+Date+`. As with `+Date+`, the date-time operated on is last (<<pipe>>).

Decoders and Encoders do not appear in Souther's syntax. Derived from a data's shape, they are generated by the ClassFile backend as Raoh Decoders / Encoders (<<codec-generation>>, <<encoder>>). Primitive parsing, error accumulation, discriminator dispatch, and encoding are handled by Raoh combinators; there is no facade on the souther-runtime side. A custom boundary codec (<<custom-codec>>) uses Raoh directly from the Java side.

[#jvm-output]
== JVM output

[#target-jdk]
=== Target JDK

The artifacts target Java 21 or later. The compiler implementation MAY use Java 25. Unicode identifiers are generated to conform to Java's identifier rules (escaping / romanization, if needed, is the backend's responsibility).

The class-file version of the generated `+.class+` is fixed at Java 21 (major 65). It MUST NOT be left to the default of the JDK running the backend. souther-runtime is also on the artifact side and is likewise compiled targeting Java 21 (only the compiler itself may be Java 25).

[#jvm-product]
=== Product data

Generated as a final class with a private constructor. Fields taken in by a `+...+` spread are laid flatly in the generated class.

[,text]
----
public final class 提出済み {
    private final 従業員 申請者;      // from ...出張申請共通項目
    private final 金額 予定費用;       // from ...出張申請共通項目
    // ...
    private final LocalDateTime 提出日時;

    private 提出済み(...) { ... }
}
----

A data in `+exposing+` gets public read accessors for its fields (<<field-visibility>>), so that another module and Java can read them; because module = package, package-private cannot read across the boundary. The constructor stays non-public, so it can be read but not constructed. A non-exposed data's fields are readable only within the same module, and no accessor is generated.

[#jvm-sum]
=== Sum data

Generated as a sealed interface and final implementation classes. If a case is a named data, its final class implements the sealed interface. A unit data is generated as a field-less final class (or singleton).

[#jvm-codec]
=== Decoder / Encoder

An exposed data gets a per-source acquisition API. What is returned is a Raoh Decoder / Encoder (`+Decoder<I, T>+` is `+I -> Result<T>+`).

[,text]
----
public static Decoder<Map<String,Object>, 出張申請> decoder()  // Map (raoh-core. newtype is Decoder<Object,T>)
public static Decoder<JsonNode, 出張申請> jsonDecoder()    // JSON (raoh-json)
public static Decoder<Record,   出張申請> recordDecoder()  // DB row (raoh-jooq)
public static Encoder<出張申請, Map<String,Object>> encoder()  // object/sum -> Map (JSON via valueToTree at the boundary)
// newtype encodes to a bare scalar. e.g. 会員ID.encoder() : Encoder<会員ID, String>
----

The concrete implementation is generated directly as bytecode by the ClassFile backend from Raoh combinators (<<codec-generation>>). souther-runtime does not depend on Raoh; the Raoh dependency is on the generated code and the app side. `+jsonDecoder()+` / `+recordDecoder()+` are generated only when the data's shape supports it (<<codec-generation>>; JSON for a shape without temporal, `+Record+` for a flat shape). `+decoder()+` (bare values / Map) is always generated.

[#jvm-behavior]
=== behavior

A fn- or `+>->+`-implemented behavior is generated as a public interface named after the behavior (first letter capitalized), with the implementation in a `+<Behavior名>$Impl+` class. Java declares the behavior by the interface name and never names the implementation. A single-input behavior's interface extends `+Behavior<In, Out>+` (so it composes with `+>->+` and its `+apply+` is typed by inheritance); a multi-input behavior's interface is a standalone functional interface that declares the typed, multi-argument `+apply+` directly. The interface carries a static factory: `+of()+` for a behavior with an empty requirement set, or `+bind(...)+` taking one implementation per requirement.

[,text]
----
var handle = Handle.bind(new JdbcFindMember(dataSource));   // Handle is the interface; bind builds Handle$Impl
----

An injected behavior (a `+behavior+` with no `+let+`) is instead an abstract class the Java implementation extends (<<java-base-class>>), not an interface.

The generated names are derived from the behavior name (first letter capitalized). Because a data name already starts with a capital, a behavior `+quote+` becomes a type `+Quote+`, colliding with a `+data Quote+` in the same module. This is a compile error; rename one to distinguish them.

[#jvm-construction-privacy]
=== Preventing exposure of construction authority

A data's constructor is non-public (package-private), and internal construction methods such as `+__construct+` are also non-public. The Java implementation of a behavior with no implementation extends the generated abstract base class (<<java-base-class>>) and constructs failure cases via the base's `+protected+` factories. There is no back door to call a data's constructor, so the implementation may be placed outside the generated package (same-package placement is not required).

What is guaranteed to Java is that no unvalidated value can be created — not that no value can be created. `+decoder()+` is public API (<<jvm-codec>>), so Java may attempt decode at any time. But the invariant is checked there (<<invariant-on-construct>>), and a violation becomes a `+Result+` failure. A unit data ignores input and decodes (<<codec-generation>>), so its `+decoder()+` lets anyone construct it — but a unit data has nothing to validate.

Reflection, Unsafe, and bytecode manipulation are out of scope for guarantees.

[#jvm-abort]
=== Abort on invariant violation

An invariant violation inside the domain (<<algebraic-types>>, <<violation-destination>>) appears in generated code as throwing `+net.unit8.souther.runtime.ConstraintViolation+`. It carries the broken data name and the invariant's location.

Souther code cannot catch this. There is no catching syntax, so it cannot control business flow (<<out-of-scope>>). The boundary Java can catch it, but handles it as distinct from a business failure. A business failure arrives as an output case (<<unmarked-sum>>) and maps to 400; an abort is a model bug and maps to 500.

A decode violation is not this; it returns as a Raoh `+Result+` failure (<<decoder-error>>).

[#jvm-anonymous-union]
=== A behavior output's anonymous union

When a behavior's output is an anonymous union (`+-> 会員 | 会員なし+`), a sealed interface representing that union is generated. Its name is the behavior's class name (first letter capitalized; <<jvm-behavior>>) plus `+Result+`, i.e. `+<Behavior名>Result+`; `+permits+` lists the union's leaf cases, and each case data implements it. An anonymous union has no common supertype (<<asymmetric-interop>>), so this lets Java receive it as a single return type and exhaustively check with `+switch+`.

[,java]
----
// when findMember is -> 会員 | 会員なし (the behavior of §13.3 and chapter 24)
sealed interface FindMemberResult permits 会員, 会員なし {}
// 会員 and 会員なし implement it; the injected behavior's abstract base FindMember implements Behavior<会員ID, FindMemberResult>
----

There are two cases where it is not generated. If the output is a named sum data, that is itself a sealed interface (<<jvm-sum>>) and is the return type. If the output is a single case, that case type is the return type.

One data type may appear in the output of multiple behaviors, so it may implement several `+<Behavior名>Result+` at once. The output of a `+>->+` composition behavior (the folded sum of mainline and departed; <<type-routing>>) is also an anonymous union, generated by the same rule from the composition name. The generated type's encode dispatches cases by the discriminator `+"type"+` like a sum (<<sum-discrimination>>).

The model author does not write this type name in Souther source; they only write `+-> 会員 | 会員なし+`. A Java consumer declares the behavior by its interface name (<<jvm-behavior>>) and calls `+apply+`, whose return type is `+<Behavior名>Result+` by inheritance, then matches the cases with `+switch+` — so the consumer never writes the `+Result+` name either. It surfaces explicitly only in an injected behavior's Java implementation, which writes it as the `+apply+` return type (<<java-example>>).

[#compiler-pipeline]
== Compiler structure

[,text]
----
Source -> Lexer -> Parser -> AST
  -> Derive (fill decoder/encoder into the AST from a data's shape)
  -> Type Check
       name resolution / construction-authority check / requirement-set check and composition inference / exhaustiveness check / invariant type check
  -> ClassFile Backend -> .class (direct bytecode generation)
----

Type checking is done on the AST, verifying construction authority, requirement sets, exhaustiveness, and invariants together. There is no independent Domain IR / Decoder IR / Encoder IR; the backend emits bytecode directly from the AST. The backend generates `+.class+` with the Class-File API (`+java.lang.classfile+`) and does not go through javac. If another backend such as Java source generation is needed in the future, there is room to insert an IR at this stage.

[#ast-tracking]
== Information tracked by the AST and backend

There is no independent intermediate representation (<<compiler-pipeline>>); the following is held as the AST and the type-check results.

[#ast-nodes]
=== AST definition nodes

[,text]
----
Module  Data(ProductData)  SumData  UnitData  Include  Field  Invariant
DecoderDef  EncoderDef  SpecBehavior  PipeBehavior  FnDef  Import
Construct  FieldInit  Expression  Case  RawExpr  DecRef  Block
----

Behavior definition nodes are two, corresponding to the two right-hand sides of <<behavior>>. `+SpecBehavior+` holds only input type, output type, and `+constructs+`, with no body. If a same-named `+FnDef+` exists it is the implementation; otherwise it is subject to Java injection (<<injected-behavior>>). `+PipeBehavior+` holds the sequence of stage names. It is itself the implementation, so it takes no `+FnDef+` and is not injected; its type is computed from the stages (<<type-routing>>). Composition vs. simple is distinguished by these two nodes, so later stages need not re-read the right-hand side to tell them apart.

`+FnDef+` holds argument names and a body. If a same-named `+SpecBehavior+` exists the type comes from it; otherwise it is a helper `+let+` and holds its own argument types, except that a non-recursive helper may leave a value parameter's type to call-site inference (<<fn-declaration>>).

A `+SumData+` case holds only a reference to a named data (<<sum-data>>). A `+...+` spread inherits taken-in fields and invariants flatly. `+DecoderDef+` / `+EncoderDef+` are filled by derivation from a data's shape (<<decoder>>, <<encoder>>). `+require+` and `+let+` are desugared, so they do not appear in the AST; they appear as an `+if+` and as block bindings (<<require>>, <<block-expression>>).

[#ast-behavior-info]
=== Information tracked about a behavior

[,text]
----
input type  output type (sum of possible business results)  requirement set  construction authority (constructs)
----

The output type is a single type or a sum. `+SpecBehavior+` takes these four from the declaration — the requirement set is also taken from `+requires+` and reconciled with the set inferred from the `+let+` (<<requires>>). `+PipeBehavior+` computes input and output types from the stage sequence (<<type-routing>>) and infers the requirement set as the union over stages (<<composition-with-requirements>>).

The body is not here. A behavior has no body; the `+FnDef+` holds it (<<behavior>>, <<ast-nodes>>). The backend combines this behavior-side information with the `+FnDef+`'s body to generate bytecode directly from the AST.

[#compile-errors]
== Compile errors

A code (e.g. `+E1301+`) is the stable, locale-independent identity of an error; the title, message, and hints are prose shown in the selected locale. The catalog below lists each code and the message it states.

An error is rendered from structured data, not a frozen string. The default (human) form is a titled report: a header naming the error and the source location, the offending line quoted with a caret under the token, found-vs-expected type blocks for a type mismatch, and any hints. An error that concerns more than one place — the two branches of a disagreeing `+if+`, the two sides of a failed composition — points at each with its own caret and label. `+--format json+` emits the same diagnostic as a JSON object (code, source region, message, hints) for tools and agents; `+--lang+` selects the locale (Japanese by default, English available). The quoted line and caret are shown for a single-file compile.

----
error E1906:
An `examples for` file may contain only examples, not data, behavior, let, or import.
----

Emitted when a file that begins with `+examples for+` contains anything other than `+example+` declarations (<<example-placement>>).

[#e1907]
=== Examples file names an absent module

[,text]
----
error E1907:
`examples for example.nope` names a module that is not being compiled.
----

Emitted when an attached `+examples for <module>+` file names a target module that is not among the compiled modules (<<example-placement>>).

[#e1908]
=== A requires dependency has no fake

[,text]
----
error E1908:
`事前承認する` requires `現在時刻`, but no fake was supplied for it.
hint: add `with 現在時刻 = ...` on the row, or a `fake 現在時刻` table
----

Emitted when an example targets a behavior whose `+requires+` dependency has neither a `+with+` on the row nor a `+fake+` declaration (<<example-fakes>>).

[#e1909]
=== A fake has no output for an input

[,text]
----
error E1909:
A fake had no output for an input: `上長を探す` has no output for 従業員ID("e-9")
----

Emitted when a `+fake+` table is asked for an input it does not list and has no `+_+` default row (<<example-fakes>>).

[#example]
== Example: the business-trip application model

An example transcribing the spec DSL (`+chapter10/types-and-states.dsl+` and others) to Souther. Each definition is annotated with its corresponding spec DSL in a comment.

[,text]
----
module example.businesstrip exposing (
    出張申請,
    出張申請を提出する,
    事前承認する
)

// --- value types (a simple value + a constraint comment in the spec DSL → invariant) ---
// data 金額 = 整数 // 0以上、円単位
data 金額 = Int
    invariant value >= 0

// data 従業員ID = 文字列 // 空文字列ではない
// newtype (data X = Y; §8.7). The derived Decoder reads from bare text and checks the invariant.
// Normalization (trim, etc.), if needed, is done in a boundary-side custom Decoder (§10.4).
data 従業員ID = String
    invariant String.length(value) > 0

// --- a nested sum with unit cases and a newtype ---
// data 役職 = 管理職 OR 一般社員 / data 管理職 = レベル / data レベル = 整数 // 1..5
data 役職 = 管理職 | 一般社員
data レベル = Int
    invariant value >= 1 && value <= 5
data 管理職 = レベル       // newtype over レベル (§8.7)
data 一般社員

// data 費用負担区分 = 自社負担 OR 先方負担 / data 自社負担 = 立替 OR 仮払い OR 会社カード
data 費用負担区分 = 自社負担 | 先方負担
data 自社負担     = 立替 | 仮払い | 会社カード
data 立替
data 仮払い
data 会社カード
data 先方負担

// --- product (role name ≠ type name) ---
// data 従業員 = 従業員ID AND 役職 AND 上長ID
data 従業員 =
    { id: 従業員ID
    , 役職: 役職
    , 上長ID: 従業員ID
    }

data 出張申請共通項目 =
    { 申請者: 従業員
    , 予定費用: 金額
    , 出張先: String
    , 出張目的: String
    , 出張開始日: Date
    , 出張終了日: Date
    }

// --- sum of states (case = named data, common items taken in flatly by a ... spread) ---
// data 出張申請 = 申請準備中 OR 提出済み OR 事前承認待ち OR 事前承認済み
// The derived Decoder reads the discriminator "type", dispatching by tag = case name ("申請準備中", etc.).
// To read by a DB "status" or an English tag ("draft", etc.), use a boundary-side custom Decoder (§10.4).
data 出張申請 =
    申請準備中 | 提出済み | 事前承認待ち | 事前承認済み

data 申請準備中 = { ...出張申請共通項目 }
data 提出済み = { ...出張申請共通項目, 提出日時: DateTime }

data 事前承認理由 = 高額出張 | 権限不足 | 先方費用負担
data 高額出張 = { 基準金額: 金額 }
// data 権限不足 = 役職 (newtype; §8.7)
data 権限不足 = 役職
data 先方費用負担

data 事前承認待ち =
    { ...出張申請共通項目
    , 提出日時: DateTime
    , 事前承認理由リスト: List<事前承認理由>
    }

data 事前承認済み =
    { ...出張申請共通項目
    , 提出日時: DateTime
    , 事前承認日時: DateTime
    , 事前承認者: 従業員ID
    }

// 承認権限なし is just data (a failure is ordinary data; no error declaration)
data 承認権限なし

// --- behavior specifications (the spec DSL's behaviors line up as is) ---

// 依存: 時刻取得 → no implementation. Injected from Java
behavior 現在時刻 : () -> DateTime

// behavior 事前承認要否を判定する = 予定費用 AND 申請者 AND 費用負担区分 -> 事前承認理由リスト
behavior 事前承認要否を判定する : (
    予定費用:    金額,
    申請者:      従業員,
    費用負担区分: 費用負担区分
) -> List<事前承認理由>
    constructs 高額出張, 権限不足, 先方費用負担

// behavior 出張申請を提出する = 申請準備中 -> 提出済み OR 事前承認待ち (both success → bare sum)
behavior 出張申請を提出する : (申請: 申請準備中) -> 提出済み | 事前承認待ち
    constructs 提出済み, 事前承認待ち

// behavior 事前承認する = 事前承認待ち AND 承認者ID -> 事前承認済み OR 承認権限なし (a failure is a case too)
// 依存: 時刻取得 → write it in requires
behavior 事前承認する : (
    申請:    事前承認待ち,
    承認者ID: 従業員ID
) -> 事前承認済み | 承認権限なし
    requires 現在時刻
    constructs 事前承認済み, 承認権限なし

// --- implementations ---

let 事前承認要否を判定する (予定費用, 申請者, 費用負担区分) = { ... }

let 出張申請を提出する (申請) = { ... }

// precondition: 承認者ID equals the applicant's 上長ID
// requires 現在時刻 appears as the last argument (§12.6)
let 事前承認する (申請, 承認者ID, 現在時刻) = {
    require 承認者ID == 申請.申請者.上長ID
        else 承認権限なし

    事前承認済み { ...申請, 事前承認日時 = 現在時刻(), 事前承認者 = 承認者ID }
}
----

The type of `+事前承認する+`:

[,text]
----
事前承認する :
    Behavior<{現在時刻}, (事前承認待ち, 従業員ID), 事前承認済み | 承認権限なし>
----

All three of `+Behavior+` come from the declaration: `+{現在時刻}+` is `+requires+`, the input is the argument list, the output is the right of `+->+`. The `+let+` has 3 arguments because it is pre-injection; when `+Handle.bind(clock)+` fixes the last argument, it becomes the declared 2-argument type (<<requires>>).

`+...申請+` copies the common items and `+提出日時+` of `+事前承認待ち+` by name, ignores `+事前承認理由リスト+` (not in `+事前承認済み+`), and adds `+事前承認日時+` and `+事前承認者+` to construct.

[#java-example]
== Java-side implementation example

[,java]
----
// extends the abstract base FindMember of §13.3 (not implements).
public final class JdbcFindMember extends FindMember {
    private final DataSource dataSource;

    public JdbcFindMember(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    @Override
    public FindMemberResult apply(会員ID id) {   // the §19.8 sealed: 会員 | 会員なし | 保存データ不正
        try (var connection = dataSource.getConnection();
             var statement = connection.prepareStatement(
                 "select id, email, display_name from member where id = ?")) {

            // value extraction goes through the encoder (§8.5). 会員ID is a newtype, so encode is a bare String.
            statement.setString(1, 会員ID.encoder().encode(id));

            try (var resultSet = statement.executeQuery()) {
                if (!resultSet.next()) {
                    return 会員なし();                            // inherited protected factory (§13.3)
                }
                var raw = Map.<String, Object>of(
                    "id",   resultSet.getString("id"),
                    "メール", resultSet.getString("email"),
                    "表示名", resultSet.getString("display_name"));

                // decode is a Raoh Result<会員>. An invariant violation maps to the 保存データ不正 case
                return switch (会員.decoder().decode(raw, Path.ROOT)) {
                    case Ok<会員> ok   -> ok.value();              // 会員 (mainline)
                    case Err<会員> err -> 保存データ不正();          // inherited protected factory
                };
            }
        } catch (SQLException e) {
            // A platform failure is not folded into a case but thrown. Souther passes it through; the boundary handles it (503, etc.; ADR-0029).
            // SQLException is checked, so wrap it in a non-checked exception of the DataAccessException family the boundary catches
            // (not IllegalStateException or a bare RuntimeException — those pass through the handler and make infrastructure failure look like a bug).
            throw new DataAccessResourceFailureException("member lookup failed", e);
        }
    }
}
----

Binding to a composition behavior:

[,java]
----
var handle = Handle.bind(new JdbcFindMember(dataSource));
var result = handle.apply(rawInput);
----

[#future-work]
== Future extensions and current limitations

The language core is implemented. The following are future extensions, or current limitations of tooling and boundary codecs — distinct from what is permanently absent (<<out-of-scope>>).

* A full formal grammar (BNF). The key delimiters and precedence are stated ahead in <<delimiters>>.
* A human-readable Java-source backend, incremental compilation, source maps. The Core layer (<<compiler-pipeline>>) is backend-neutral, so a Java-source backend can be written from it.
* Workspace-wide (cross-module) name resolution in the LSP. The `+souther-lsp+` server ships and resolves names within a single module; diagnostics and go-to-definition across imported modules are future work.
* JSON Schema generation, Wasm output, JavaScript output, asynchronous outside-world dependencies.
* Hand-written decoder / encoder syntax. A different key name, normalization, or business-specific discriminator is handled by a boundary-side custom codec (Java side).
* Advanced (global) type inference. The current design is declarative: a behavior takes its type from the specification, and a helper states its own type — apart from the local call-site inference of a non-recursive helper's value parameters (<<fn-declaration>>).
* Storing a function in a data field. A function cannot be mapped to an external representation, so placing one would create an exception to decoder / encoder derivation (<<decoder>>, <<encoder>>) (<<blocks>>).
* Optimization that narrows generation by usage of input sources (currently decoders are generated for the sources a data's shape supports; <<codec-generation>>).
* jOOQ flatness limit: a jOOQ `+Record+` row is flat, so a jOOQ decoder is not generated for a type with nested data / List / Map. Decoders for bare-value / Map / JSON sources are generated. This is a limit inherent to jOOQ rows and is largely permanent.

[#non-functional]
== Non-functional requirements

* compile errors are rendered from structured data — a code, a source region, a message, and hints; the human form quotes the source with a caret, a JSON form (`+--format json+`) is available for tools, and messages are localized (Japanese by default, English available)
* the generated `+.class+` passes standard verification and is usable straightforwardly from Java 21+
* each compiler stage is independently testable
* souther-runtime does not depend on Raoh; the generated bytecode uses Raoh directly for decoder/encoder, and the Raoh dependency is on the generated code and the app side
* the language core does not depend on the Java class model
* the requirement set is tracked statically; a simple behavior reconciles declaration and implementation, and a composition infers from its stages
* generated code exposes no API bypassing invariants
* one spec DSL definition maps to one Souther definition with no intervening translation

[#core-semantics]
== Core semantics

Souther tracks two things about a data.

[,text]
----
which values are that data
which expressions have the authority to construct that data
----

It tracks three things about a behavior.

[,text]
----
input type   output type (sum of possible business results)   the set of required dependencies
----

A behavior conceptually has the type:

[,text]
----
Behavior<Requirements, Input, Output>
----

There is no fourth slot separating failure. A failure is one of the cases of Output, and whether it departs the mainline is decided by composition. `+>->+` simultaneously performs output-case routing (flowing cases the downstream consumes, propagating the rest) and requirement-set composition.

[,text]
----
f : Behavior<R1, A, O_f>
g : Behavior<R2, B, O_g>
f >-> g : Behavior<R1 ∪ R2, A, (O_g) | (sum of O_f minus the consumed part)>
----

This rule is Souther's central composition semantics. Output corresponds to the spec DSL's `+-> 出力 OR ...+`, and the requirement set to `+// 依存+` / `+// 副作用+`. What the author states explicitly is input, output (a bare sum), dependencies (`+requires+`), and construction authority (`+constructs+`). On the same line as the spec DSL writing `+// 依存:+` at a leaf but not on a composition, only a composition is left to inference (<<requires>>, <<composition-with-requirements>>).

`+Requirements+` is an argument fixed by injection, not a dynamically handled effect. So on the `+let+` side it appears as an ordinary argument, and `+bind+` partially applies it (<<requires>>). `+Behavior<R, I, O>+` can be read as `+(I, R) -> O+` closed over `+R+`. That is why no variable enters the requirement set, and that keeps the whole system simple.

Souther maps a specification model to an implementation model with minimal translation by promoting the value constraints, construction paths, and outside-world dependencies the spec DSL left in comments to invariants, confinement of construction paths, and `+requires+` with injection.
